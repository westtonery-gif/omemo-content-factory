# WORKFLOW_ACCEPTANCE.md — Критерии приёмки реализации **Workflow** / **Workflow Step**

> **Место в иерархии документации (PROJECT.md, раздел 17).**
> `PROJECT.md` → `ARCHITECTURE.md` → `ROADMAP.md` → `DOMAIN_MODEL.md` → `WORKFLOW_SPEC.md` →
> **`WORKFLOW_ACCEPTANCE.md`** → `ADR` → `Implementation (Code)`.
>
> Документ отвечает на вопрос: **«Как мы поймём, что реализация Workflow / Workflow Step (и их
> разворачивание Content Director'ом) соответствует `WORKFLOW_SPEC.md` и `ADR-0009` (Rev 2)?»** Это
> набор **проверяемых сценариев приёмки** на языке предметной области — прямая основа для
> `tests/test_workflow.py`: один сценарий → один тест (или группа).
>
> Документ описывает **только наблюдаемое поведение**: целостность данных плана, модель порядка,
> разворачивание плана в Task и обнаружение невалидного плана. Поведение execution core (Run / Task
> / Output / Schema) **не предмет** этой приёмки — проверяется лишь его **неизменность**.
>
> **Источник истины:** `WORKFLOW_SPEC.md` → `ADR-0009` (Rev 2) → `DOMAIN_MODEL.md` → `PROJECT.md`.

**Версия документа:** 1.0
**Статус:** Принят (контракт приёмки для модели Workflow / Workflow Step)
**Дата:** 2026-06-27
**Владелец:** Архитектор предметной области

## 0. Соглашения

- Формат сценария — **Дано / Когда / Тогда** на языке домена.
- Каждый сценарий имеет **уникальный идентификатор** (например, `WORD-01`).
- Критерии проверяют **наблюдаемое поведение** через публичный контракт, **не** внутреннюю
  реализацию.
- **«Порядок исполнения»** в этом документе означает порядок, в котором Content Director
  разворачивает план в последовательность Task. Он **строго равен** порядку `Workflow.steps`
  (`WORKFLOW_SPEC.md` §4).
- **«Отвергается как ошибка предметной области»** означает: Workflow **не создаётся**, фиксируется
  **доменная** ошибка (не технический сбой).
- Префиксы: `WAC` — критерии приёмки; `WHP` — корректное построение; `WORD` — правила порядка;
  `WDEP` — edge-кейсы `depends_on`; `WCD` — разворачивание Content Director; `WFL` — обнаружение
  невалидного плана; `WINV` — инварианты; `WISO` — изоляция от execution core.

---

## 1. Acceptance Criteria

- **WAC-01. Корректное построение.** Непустой план с уникальными `step_id` создаётся; шаги
  читаются в объявленном порядке. ⟶ `WHP-01`
- **WAC-02. Порядок = список.** Порядок разворачивания строго равен порядку `steps`. ⟶ `WORD-01`,
  `WCD-01`
- **WAC-03. `depends_on` инертен.** `depends_on` не влияет на порядок, планирование и поток
  управления. ⟶ `WORD-02`, `WDEP-02..04`, `WCD-02`
- **WAC-04. Ссылочная корректность `depends_on`.** Каждый id в `depends_on` ссылается на
  существующий `step_id`. ⟶ `WDEP-05`, `WFL-03`
- **WAC-05. Обнаружение невалидного плана.** Пустой план, дубликаты `step_id`, висячая зависимость
  отвергаются при создании. ⟶ `WFL-01..03`
- **WAC-06. Разворачивание Content Director.** Workflow → последовательность Task в строгом порядке
  списка, без интерпретации `depends_on`, без графа, без оптимизации. ⟶ `WCD-01..03`
- **WAC-07. Неизменяемость.** Workflow / Workflow Step неизменяемы; новый план = новое значение. ⟶
  `WINV-01`
- **WAC-08. Изоляция от execution core.** Удаление Workflow не меняет Run/Task/Output/Schema; план
  создаётся без Run. ⟶ `WISO-01`, `WISO-02`

*Критериев приёмки: **8** (WAC-01 … WAC-08).*

## 2. Construction Scenarios (WHP)

**WHP-01. Корректное построение плана.**
- *Дано:* набор шагов `[research, write, edit]`, у каждого `step_id`, `task_type`, `agent_ref`,
  `schema_ref`; `step_id` уникальны.
- *Когда:* создаётся Workflow с именем и этими шагами.
- *Тогда:* Workflow создан; `steps` читаются **в том же порядке** `[research, write, edit]`; поля
  каждого шага доступны для чтения.

*Сценариев построения: **1**.*

---

## 3. Ordering Rules (WORD)

**WORD-01. Порядок разворачивания = порядок `steps`.**
- *Дано:* Workflow со `steps = [A, B, C]`.
- *Когда:* план разворачивается в последовательность Task (раздел 5).
- *Тогда:* порядок Task строго `A → B → C` (порядок списка).

**WORD-02. `depends_on` не меняет порядок.**
- *Дано:* Workflow со `steps = [A, B, C]`, где у `B` указан `depends_on = [C]` (зависимость от
  более позднего шага).
- *Когда:* план разворачивается.
- *Тогда:* порядок остаётся `A → B → C`; `depends_on` **проигнорирован** для целей порядка.

*Сценариев порядка: **2**.*

---

## 4. `depends_on` Edge Cases (WDEP)

**WDEP-01. Пустой/умолчательный `depends_on`.**
- *Дано:* шаги без `depends_on` (по умолчанию пуст).
- *Тогда:* план валиден; `depends_on` пуст; порядок = список.

**WDEP-02. Ссылка на существующий более ранний шаг.**
- *Дано:* `steps = [A, B]`, `B.depends_on = [A]`.
- *Тогда:* план валиден; порядок `A → B`; `depends_on` инертен.

**WDEP-03. Ссылка на более поздний шаг (forward reference).**
- *Дано:* `steps = [A, B]`, `A.depends_on = [B]`.
- *Тогда:* план валиден (ссылка существует); порядок **остаётся** `A → B`; переупорядочивания нет.

**WDEP-04. Само- / циклическая зависимость.**
- *Дано:* `A.depends_on = [A]` (или `A↔B` взаимно).
- *Тогда:* план **валиден** (все ссылки существуют) и `depends_on` инертен; **обнаружение циклов не
  выполняется** (отложено — см. раздел 10, флаг **G-1**).

**WDEP-05. Висячая ссылка.**
- *Дано:* `B.depends_on = [X]`, где шага `X` в плане нет.
- *Тогда:* отвергается как доменная ошибка (`UnknownDependencyError`); Workflow не создаётся.

*Сценариев edge-кейсов `depends_on`: **5**.*

## 5. Content Director Resolution Rules (WCD)

> Разворачивание Workflow → Task — единственное место, где план «встречает» исполнение
> (`WORKFLOW_SPEC.md` §8; `ADR-0009` §8). Здесь проверяется **результат разворачивания** (порядок и
> привязки), а не внутренности Run/Task.

**WCD-01. Разворачивание в Task строго по списку.**
- *Дано:* Workflow со `steps = [A, B, C]`.
- *Когда:* Content Director разворачивает план.
- *Тогда:* получается **ровно** N=3 Task в порядке `A → B → C`; для каждого Task ссылка на шаг
  соответствует `step_id`, а роль — `agent_ref` шага.

**WCD-02. Content Director игнорирует `depends_on`.**
- *Дано:* Workflow, где `depends_on` подразумевал бы иной порядок (как в `WORD-02`/`WDEP-03`).
- *Когда:* Content Director разворачивает план.
- *Тогда:* порядок Task = порядок списка; граф **не строится**, оптимизация порядка **не
  выполняется**.

**WCD-03. Разворачивание использует существующий pipeline без изменений.**
- *Дано:* Workflow, эквивалентный последовательности `TaskRequest`, которую сегодня принимает
  существующий путь исполнения.
- *Когда:* план разворачивается и исполняется.
- *Тогда:* наблюдаемое поведение Run/Task/Output/Schema **идентично** прямой подаче эквивалентной
  последовательности; Content Director остаётся **единственной точкой исполнения**.

*Сценариев разворачивания: **3**.*

---

## 6. Invalid Workflow Detection (WFL)

**WFL-01. Пустой план.**
- *Когда:* создаётся Workflow без шагов.
- *Тогда:* отвергается (`EmptyWorkflowError`); Workflow не создаётся.

**WFL-02. Дубликаты `step_id`.**
- *Когда:* создаётся Workflow с двумя шагами одинакового `step_id`.
- *Тогда:* отвергается (`DuplicateStepIdError`).

**WFL-03. Висячая зависимость.**
- *Когда:* создаётся Workflow, где `depends_on` ссылается на несуществующий `step_id`.
- *Тогда:* отвергается (`UnknownDependencyError`). *(Совпадает с `WDEP-05`.)*

*Сценариев обнаружения невалидного плана: **3**.*

---

## 7. Invariants (WINV)

Единственная уникальная проверка, не покрытая прочими блоками, — неизменяемость; остальные
инварианты `WORKFLOW_SPEC.md` §5 сведены в таблицу трассируемости.

**WINV-01. Неизменяемость плана.** Попытка изменить созданный Workflow или его Workflow Step не
поддерживается публичным контрактом; иной план оформляется как **новое значение** Workflow.
*(Структурно: см. раздел 10.)*

**Таблица трассируемости инвариантов (`WORKFLOW_SPEC.md` §5):**

| Инвариант | Покрытие |
|---|---|
| 1. Неизменяемость | `WINV-01` (структурно) |
| 2. Непустой план | `WFL-01` |
| 3. Уникальность `step_id` | `WFL-02` |
| 4. Ссылочная корректность `depends_on` | `WDEP-05` / `WFL-03` |
| 5. Ровно одна роль на шаг | `WHP-01`, `WCD-01` (роль = `agent_ref`) |
| 6. Целостность агрегата (Step только в составе Workflow) | `WHP-01` (Step создаётся только в Workflow) |
| 7. Строгий порядок (`depends_on` не влияет) | `WORD-01/02`, `WCD-02` |
| 8. Отсутствие runtime-состояния | `WISO-01` + структурно (нет состояния в контракте) |

*Уникальных сценариев инвариантов: **1** (`WINV-01`); остальные — в таблице.*

## 8. Isolation Scenarios (WISO)

**WISO-01. Execution core независим от Workflow.**
- *Дано:* исполнение последовательности Task без участия Workflow (существующий путь).
- *Тогда:* Run/Task/Output/Schema ведут себя как прежде; **удаление Workflow** не меняет execution
  core (он от него не зависит).

**WISO-02. План создаётся без Run.**
- *Дано:* каталог определений.
- *Когда:* Workflow создаётся и читается.
- *Тогда:* для построения плана **не требуется** ни один Run; Workflow существует вне оси
  исполнения.

*Сценариев изоляции: **2**.*

## 10. Флаги и ограничения проверяемости

**Флаги (требуют решения / отложены сознательно):**

- **G-1. Обнаружение циклов/само-зависимостей НЕ выполняется.** По `ADR-0009` §5 при создании
  проверяется только **ссылочная корректность** `depends_on`; цикл (`A↔B`) и само-ссылка (`A→A`)
  ссылочно валидны и **инертны** (`WDEP-04`). Это **сознательный non-goal** (никакого DAG/анализа
  графа). Если в будущем потребуется отвергать циклы — это **новый ADR**, не данный срез.
- **G-2. Запрет передачи семантики порядка через `depends_on`.** Проверяется **позитивно** (порядок
  = список даже при «логичной» зависимости — `WORD-02`, `WDEP-03`), а не как отдельный «негативный»
  сценарий: интерпретации `depends_on` в контракте просто **нет**.

**Ограничения сценарной приёмки (структурные гарантии):**

- **Неизменяемость на уровне поля (`WINV-01`).** Снаружи нет мутабельного объекта плана; «попытку
  переписать поле» нельзя выразить через публичный контракт — это **структурная гарантия** (frozen
  данные, ср. Output, ADR-0005). Проверяется позитивно: иной план = новое значение.
- **Незаданность пустых значений полей шага.** `WORKFLOW_SPEC.md` §3 помечает `task_type` /
  `agent_ref` / `schema_ref` как обязательные (присутствие), но **не определяет** отклонение
  пустых строк; поэтому такой сценарий **не выдумывается** здесь. Если требуется отклонять пустые
  значения — уточняется в `WORKFLOW_SPEC.md` (не в acceptance).

**Контекст реализации (не флаг):** разворачивание Content Director (`WCD-*`) — часть реализации
этого среза (`ADR-0009` §8); acceptance задаёт его контракт **до** кода. Реализация не должна
менять Run/Task/Output/Schema и существующий путь исполнения.

### Подтверждение готовности к pytest

К написанию `tests/test_workflow.py` можно переходить по всем сценариям разделов 2–8. Открытых
блокирующих вопросов нет; G-1/G-2 — сознательные non-goals, зафиксированные ADR-0009.

> При расхождении с `WORKFLOW_SPEC.md` исправляется **этот** документ: источник истины —
> `WORKFLOW_SPEC.md` → `ADR-0009` → `DOMAIN_MODEL.md` → `PROJECT.md` (раздел 17).
