> контракт уровня SPEC**: он фиксирует *форму*, которой восстановленный Run приводится в

> транзакции — это уровень `RunStore` (инфраструктура, ARCH §9/§10) и отдельный контракт.

## 1. Назначение и границы

### 1.1. Что фиксирует документ

1. **`RunSnapshot`** — форму *наблюдаемой истины* одного Run: минимальный полный состав, из
   которого восстановленный Run неотличим от сохранённого (RUN_SPEC §4a) и **продолжаемость не
   теряет и не дублирует** уже сделанную работу (ADR-0015 I3).
2. **`Run.restore(snapshot)`** — второй вход конструирования Run: приведение агрегата в
   существование прямо в сохранённом состоянии, с применением verify/reject (RUN_SPEC §4a).

### 1.2. Что НЕ входит

- **Механизм сохранения/восстановления, хранилище, сериализация, формат, атомарность,
  транзакции.** Это `RunStore` (инфраструктура; ARCH §9/§10) — отдельный реализационный
  контракт. `RunSnapshot` — чистое доменное значение (как Views), *persistence-ignorant*
  (RUN_SPEC §9).
- **Новые доменные состояния, события, переходы, инварианты.** Их нет: восстановление
  переустанавливает уже легально достигнутое состояние (RUN_SPEC §4a; ADR-0015 §3).
- **Реплей журнала событий** — запрещён (ADR-0015 §Rationale; ARCHITECTURE_FREEZE §3). Истина
  авторитетна на сущностях, а не восстанавливается из событий.

### 1.3. Принцип аддитивности

`Run.restore` — **чисто аддитивное** расширение: `Run.create`, таблица переходов, `transition`
и все существующие сигнатуры/поведение **не меняются** (ADR-0015 I7; PROJECT §4.11; прецедент
ADR-0004…0007). Восстановленный Run — тот же агрегат, достигнутый *иным способом появления*, не
новая сущность.

## 2. Почему View недостаточен (доказанная необходимость снимка)

Наблюдение через существующие read-only View **теряет** часть наблюдаемой истины. Это доказывает
необходимость `RunSnapshot` и одного нового детского типа (`TaskSnapshot`) — а не «сериализуй
Views»:

| Что теряется | Где сейчас | Последствие без сохранения |
|---|---|---|
| `retry_policy` (Task) | `TaskView` **не несёт** | после восстановления граница повторов Task сбрасывается в дефолт — нарушение точности продолжения |
| `rework_policy` (Run) | Run вообще не имеет View | то же для границы доработок Run |
| `task_seq` / `artifact_seq` / `review_seq` | ни в одном View | **id-коллизия при продолжении**: восстановили Run с `…-task-3`, вызов `open_task` начнёт счётчик с 0 → сгенерит `…-task-1` и **перезапишет** существующего ребёнка. Прямое нарушение **ADR-0015 I3** (без дублирования) |
| журнал `events` | ни в одном View | авторитетная трасса (ADR-0015 I2) потеряна |

**Полны и переиспользуются как есть:** `Output` (frozen, лежит внутри Task), `ArtifactView`,
`HumanReviewView` — round-trip-полны. Их поля покрывают всё приватное состояние своих сущностей.

**Вывод.** Минимально необходимые новые типы: **`RunSnapshot`** (контейнер агрегата) и
**`TaskSnapshot`** (поля Task + `retry_policy`, которых нет в `TaskView`). Artifact / Human
Review / Output переиспользуются напрямую. Это удовлетворяет правилу «новых типов — только при
доказанной необходимости».

---

## 3. Состав `RunSnapshot`

`RunSnapshot` — неизменяемое (frozen) доменное значение. Состав — ровно наблюдаемая истина Run,
без чужих данных и без поведения.

**Уровень Run (скаляры и политика):**

- `run_id: str` — идентичность агрегата (RUN_SPEC §3; неизменяемый вход).
- `content_brief_ref: str` — зафиксированный вход (неизменяемый).
- `workflow_version_ref: str` — зафиксированный вход (неизменяемый).
- `status: RunStatus` — ровно один текущий статус (RUN_SPEC §5 инв. 1).
- `rework_count: int` — счётчик итераций доработки (RUN_SPEC §3).
- `failure_reason: str | None` — причина `FAILED`, если достигнут (RUN_SPEC §3).
- `rework_policy: ReworkPolicy` — граница доработок (**не** в View; нужна для точного
  продолжения).

**Продолжение генерации идентификаторов (не в View; обязательны — иначе I3):**

- `task_seq: int`
- `artifact_seq: int`
- `review_seq: int`

**Журнал (авторитетная трасса; берётся вербатим, не переигрывается):**

- `events: Sequence[<любое событие журнала Run>]` — упорядоченная последовательность событий
  Run/Task/Output/Artifact/Human Review в том же порядке, что и в `run.events`.

**Состав агрегата (принадлежность, не копии):**

- `tasks: Sequence[TaskSnapshot]`
- `artifacts: Sequence[ArtifactView]` *(переиспользование — View полон)*
- `human_reviews: Sequence[HumanReviewView]` *(переиспользование — View полон)*

### 3.1. `TaskSnapshot` (новый тип — единственный среди детей)

Поля `TaskView` **плюс** `retry_policy`:

- `task_id: str`, `run_id: str`, `workflow_step_ref: str`, `agent_ref: str`, `task_input: str`
- `status: TaskStatus`, `attempt_count: int`, `failure_reason: str | None`
- `output: Output | None` *(Output frozen — переиспользование)*
- `retry_policy: TaskRetryPolicy` *(отсутствует в `TaskView`)*

> Размещение типов — доменный уровень (как Views: `RunSnapshot`/`TaskSnapshot` — проекции
> агрегата, persistence-ignorant). Конкретный модуль/предотвращение циклов импорта — деталь шага
> реализации, не предмет контракта.

## 4. Контракт `Run.restore(snapshot)`

### 4.1. Сигнатура и роль

- `@classmethod Run.restore(snapshot: RunSnapshot) -> Run` — **второй** вход конструирования,
  наряду с `create`. Не трогает `create`/`transition`/таблицу переходов.
- **Не совершает переход.** Переустанавливает состояние, которого оригинал уже достиг легально
  (RUN_SPEC §4a; ADR-0013). Сторожа переходов не обходятся — они просто не применяются, потому
  что перехода нет.
- **Не эмитит новых событий.** Журнал берётся из `snapshot.events` вербатим; `RunCreated` и
  прочие повторно не порождаются (иначе дублирование трассы — нарушение I2/I3).

### 4.2. Переустановка состояния

`restore` строит экземпляр Run прямо в сохранённом состоянии:

1. Неизменяемый вход (`run_id`, `content_brief_ref`, `workflow_version_ref`) — записывается
   write-once (тем же путём, что `__init__`).
2. Мутабельное состояние (`status`, `rework_count`, `failure_reason`, `rework_policy`,
   `task_seq`/`artifact_seq`/`review_seq`, журнал `events`) — устанавливается напрямую.
3. Каждый ребёнок (`Task` со своим `Output`, `Artifact`, `Human Review`) восстанавливается в
   своё сохранённое состояние **внутренним путём восстановления, доступным только Run**.
   Публичной детской фабрики не появляется (инкапсуляция как сейчас: дети конструируются только
   через корень).

### 4.3. verify/reject (применение RUN_SPEC §4a)

Восстановленный Run допускается **только** если снимок удовлетворяет инвариантам RUN_SPEC §5.
Нарушение → **доменная ошибка `RunRestorationError`** (подкласс `RunDomainError`), Run **не**
возвращается и **не** устанавливается «как есть». Проверяются структурные инварианты (легальность
проверяется, а не предполагается — §4a), **без** переигрывания истории:

1. **Ровно один статус.** `status` — валидное значение `RunStatus`. *(инв. 1)*
2. **Неизменяемый вход на месте.** `run_id`, `content_brief_ref`, `workflow_version_ref`
   непусты. *(инв. 2)*
3. **Целостность агрегата** *(инв. 7):*
   - у каждого `TaskSnapshot`, `ArtifactView`, `HumanReviewView`: `run_id == snapshot.run_id`;
   - у каждого `Output` (если есть): `task_id` совпадает с `task_id` владеющего Task;
   - каждый `artifact.output_ref` ссылается на `Output`, записанный на каком-то владеемом Task;
   - каждый `review.artifact_ref` ссылается на владеемый `Artifact`.
4. **Согласованность счётчиков (защита I3).** Каждый `*_seq` не меньше числа детей своего вида
   (`task_seq ≥ |tasks|`, `artifact_seq ≥ |artifacts|`, `review_seq ≥ |human_reviews|`) — иначе
   следующий сгенерированный id мог бы совпасть с существующим. *(вытекает из инв. 7 + ADR-0015
   I3)*

### 4.4. Постусловие

Возвращённый Run **неотличим** от сохранённого:

- те же скаляры, те же `tasks`/`artifacts`/`human_reviews` (по View), тот же порядок `events`;
- то же поведение: те же допустимые/запрещённые переходы (RUN_SPEC §4), те же доменные ошибки;
- **терминал остаётся терминалом** — обеспечивается таблицей переходов (у `COMPLETED`/`FAILED`
  нет исходящих рёбер), отдельной проверки не требует;
- продолжение (`open_task`/`create_artifact`/`open_human_review`) использует восстановленные
  счётчики и **не создаёт коллизий id**.

---

## 5. Чего `Run.restore` НЕ делает

- **Не переигрывает** события/переходы (нет реплея; ADR-0015).
- **Не совершает** переход и не эмитит событий.
- **Не знает** о хранилище/сериализации/атомарности — это `RunStore`.
- **Не гарантирует** отсутствие двух «живых» Run с одним `run_id` — это ответственность
  вызывающего/хранилища (RUN_SPEC §4a, второй класс ошибки), а не самой фабрики. `restore`
  обеспечивает лишь допущение по инвариантам §5.

---

## 6. Связь с `RunStore` (для полноты, вне предмета контракта)

- `RunStore.save(run)` наблюдает Run и получает `RunSnapshot` (наблюдение полно — раздел 2/3).
- `RunStore.load(run_id)` читает снимок и вызывает `Run.restore(snapshot)` → восстановленный Run
  либо доменная ошибка (`RunRestorationError`), либо `None` для неизвестного `run_id`.

Проводка (когда сохранять) и хранилище — контракт `RunStore` и Composition Root, не этого
документа.
