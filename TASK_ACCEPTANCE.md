# TASK_ACCEPTANCE.md — Критерии приёмки реализации **Task**

> **Место в иерархии документации (PROJECT.md, раздел 17).**
> `PROJECT.md` → `ARCHITECTURE.md` → `ROADMAP.md` → `DOMAIN_MODEL.md` → `TASK_SPEC.md` →
> **`TASK_ACCEPTANCE.md`** → `ADR` → `Implementation (Code)`.
>
> Документ отвечает на вопрос: **«Как мы поймём, что реализация Task полностью соответствует
> `TASK_SPEC.md` и `ADR-0004`?»** Это набор **проверяемых сценариев приёмки** на языке
> предметной области. Он не содержит кода, pytest и технических деталей и становится **прямой
> основой для `tests/test_task.py`**: один сценарий → один тест (или группа тестов).
>
> **Источник истины:** `TASK_SPEC.md` (по сущности Task) → `ADR-0004` (технический контракт) →
> `DOMAIN_MODEL.md` → `PROJECT.md`.

## 0. Соглашения

- Формат сценария — **Дано / Когда / Тогда** (Given / When / Then) на языке домена.
- Каждый сценарий имеет **уникальный идентификатор** (например, `THP-01`) для прямого
  отображения в будущий тест.
- Актёр, инициирующий открытие Task или смену его статуса, — **Content Director** (если не
  указано иное). Любое изменение Task идёт **только через Run** (корень агрегата).
- «Отвергается как ошибка предметной области» означает: операция **не выполняется**, состояние
  Task **не изменяется**, фиксируется доменная ошибка (не технический сбой).
- Префиксы блоков: `TAC` — критерии приёмки; `THP` — happy path; `TRW` — повторы/retry;
  `TFL` — ошибки/запреты; `TINV` — инварианты; `TAGG` — границы агрегата; `TID` — идентичность;
  `TEV` — события; `TXC` — межсущностная согласованность Run↔Task.
- Доступ к Task **только для чтения**: через Run отдаётся **снимок** (`TaskView`), а не
  изменяемый объект Task (`ADR-0004` §4).

## 1. Acceptance Criteria

Проверяемые критерии, выведенные из `TASK_SPEC.md` и `ADR-0004`. Каждый критерий проверяется
одним или несколькими сценариями разделов 2–9 (ссылки указаны).

- **TAC-01. Ровно один статус.** В любой момент у Task строго одно текущее состояние. ⟶
  `TINV-01`, `THP-*`
- **TAC-02. Неизменяемый вход.** Идентификатор, владеющий Run, Workflow Step, Agent и входные
  данные не меняются после создания. ⟶ `TID-02`, `TINV-02`
- **TAC-03. Единственный распорядитель статуса.** Открыть Task и менять его статус может только
  Content Director (через Run). ⟶ `TFL-03`, `TFL-04`, `TINV-03`
- **TAC-04. Только разрешённые переходы.** Допустимы лишь рёбра из `TASK_SPEC.md` §4; остальные
  отвергаются. ⟶ `TFL-01`, `TINV-04`
- **TAC-05. Терминальность.** `Succeeded`/`Failed`/`Skipped` окончательны; из них нет переходов.
  ⟶ `TFL-01`, `THP-03`, `THP-04`, `TINV-05`
- **TAC-06. Ограниченность попыток.** Число попыток ограничено политикой; повтор сверх предела
  отвергается. ⟶ `TRW-02`, `TINV-06`
- **TAC-07. Целостность агрегата.** Task принадлежит ровно одному Run, создаётся/меняется только
  через него; нет публичной фабрики и нет изменяемого Task снаружи. ⟶ `TAGG-01..03`, `TINV-07`
- **TAC-08. Прослеживаемость (события).** Значимые факты Task фиксируются как события в журнале
  владеющего Run. ⟶ `TEV-01..06`, `TINV-08`
- **TAC-09. Стабильная идентичность.** Идентификатор Task неизменен; смена статуса/попыток не
  создаёт новую сущность; разные Task имеют разные идентификаторы. ⟶ `TID-01`, `TID-03`
- **TAC-10. Согласованность с Run.** Run не достигает `Completed` при наличии нетерминальных
  Task. ⟶ `TXC-01`, `TXC-02`

*Критериев приёмки: **10** (TAC-01 … TAC-10).*

## 2. Happy Path Scenarios

**THP-01. Открытие Task.**
- *Дано:* существует Run.
- *Когда:* Content Director открывает Task по Workflow Step с указанием Agent и входных данных.
- *Тогда:* возникает Task в `Pending`; возвращается его идентификатор; число попыток = 0;
  порождается событие `TaskCreated`; Task виден в составе Run (`TaskView`).

**THP-02. Первый старт исполнения.**
- *Дано:* Task в `Pending`.
- *Когда:* Content Director переводит Task `Pending → Running`.
- *Тогда:* Task в `Running`; число попыток = 1; порождается событие `TaskStarted`.

**THP-03. Успешное завершение.**
- *Дано:* Task в `Running`.
- *Когда:* Content Director переводит Task `Running → Succeeded`.
- *Тогда:* Task в `Succeeded` (терминал); порождается событие `TaskCompleted`.

**THP-04. Пропуск шага.**
- *Дано:* Task в `Pending`.
- *Когда:* Content Director переводит Task `Pending → Skipped`.
- *Тогда:* Task в `Skipped` (терминал); выделенного события не порождается.

*Сценариев happy path: **4**.*

## 3. Retry Scenarios

**TRW-01. Повтор увеличивает число попыток и не порождает `TaskStarted`.**
- *Дано:* Task в `Running` с числом попыток 1.
- *Когда:* Content Director переводит Task `Running → Running` (повтор).
- *Тогда:* Task в `Running`; число попыток = 2; событие `TaskStarted` **не** порождается (новых
  событий нет).

**TRW-02. Повтор сверх предела политики отвергается.**
- *Дано:* Task с политикой повторов, допускающей максимум попыток `N`, доведён до `N` попыток в
  `Running`.
- *Когда:* запрашивается ещё один повтор (`Running → Running`), который превысил бы `N`.
- *Тогда:* отвергается как доменная ошибка; Task остаётся в `Running`; число попыток не
  меняется. *Легальный следующий шаг:* перевод в `Failed`.

**TRW-03. После повтора Task может успешно завершиться.**
- *Дано:* Task, прошедший хотя бы один повтор (попыток ≥ 2), в `Running`.
- *Когда:* Content Director переводит Task `Running → Succeeded`.
- *Тогда:* Task в `Succeeded`; число попыток сохраняет накопленное значение.

*Сценариев повторов: **3**.*

## 4. Failure & Forbidden-Transition Scenarios (ошибки предметной области)

**TFL-01. Запрещённые переходы отвергаются.**
- *Когда:* запрашивается любой переход, отсутствующий в `TASK_SPEC.md` §4 (например,
  `Pending → Succeeded`, `Pending → Failed`, `Running → Skipped`, `Running → Pending`, любой
  переход из `Succeeded`/`Failed`/`Skipped`).
- *Тогда:* отвергается как доменная ошибка; статус Task не меняется. *(Полный перебор рёбер.)*

**TFL-02. `Running → Failed` фиксирует причину.**
- *Дано:* Task в `Running`.
- *Когда:* Content Director переводит Task в `Failed` с указанием причины.
- *Тогда:* Task в `Failed` (терминал); причина зафиксирована; порождается событие `TaskFailed`,
  несущее причину.

**TFL-03. Смена статуса Task не Content Director.**
- *Дано:* Task в `Pending`.
- *Когда:* статус пытается сменить иной актёр (например, Agent).
- *Тогда:* отвергается как доменная ошибка; статус Task не меняется.

**TFL-04. Открытие Task не Content Director.**
- *Дано:* существует Run.
- *Когда:* Task пытается открыть иной актёр (например, Agent).
- *Тогда:* отвергается как доменная ошибка; Task не создаётся.

*Сценариев ошибок/запретов: **4**.*

## 5. Invariant Tests

По одному сценарию проверки на каждый инвариант `TASK_SPEC.md` §5.

**TINV-01. Ровно один статус.** В любой достижимой точке у Task строго одно состояние,
заменяемое атомарно при переходе.
**TINV-02. Неизменяемый вход.** Владеющий Run, Workflow Step, Agent и входные данные не меняются
на протяжении жизненного цикла (проверяется через снимок `TaskView`).
**TINV-03. Единственный распорядитель статуса.** Любая смена статуса возможна только Content
Director через Run.
**TINV-04. Только разрешённые переходы.** Полный перебор показывает: допустимы лишь рёбра из
`TASK_SPEC.md` §4, прочие отвергаются.
**TINV-05. Терминальность.** Из `Succeeded`/`Failed`/`Skipped` нет исходящих переходов; каждый
достижимый путь ведёт к терминалу.
**TINV-06. Ограниченность попыток.** Существует предел числа попыток; его превышение повтором
ведёт к доменной ошибке (а легальный путь — `Failed`).
**TINV-07. Целостность агрегата.** Каждый Task принадлежит ровно одному Run; его ссылка на Run
равна идентификатору этого Run; изменения — только через Run.
**TINV-08. Прослеживаемость.** Каждый значимый переход Task фиксируется событием в журнале
владеющего Run.

*Сценариев инвариантов: **8** (по числу инвариантов §5, кроме межсущностного п. 9 — он покрыт
блоком `TXC`).*

## 6. Aggregate Boundary Tests

**TAGG-01. Task вне Run не существует.** Task создаётся **только** через Run; публичной фабрики
Task нет. Открытый через Run Task принадлежит именно этому Run (его ссылка на Run равна
идентификатору этого Run).
**TAGG-02. Task разных Run изолированы.** Task, открытые в разных Run, принадлежат своим Run;
один Run не видит и не изменяет Task другого Run.
**TAGG-03. Снаружи доступен только снимок.** Чтение Task через Run отдаёт **снимок** (`TaskView`),
а не изменяемый объект; изменить статус можно только через Run. Полученный ранее снимок не
«живой»: после перехода прежний снимок сохраняет старое состояние, а новый — отражает новое.

*Сценариев границ агрегата: **3**.*

---

## 7. Identity Tests

**TID-01. Неизменяемость идентификатора.** Идентификатор Task остаётся прежним на всём
жизненном цикле (через все переходы статусов и повторы).
**TID-02. Неизменяемость входа.** Зафиксированные владеющий Run, Workflow Step, Agent и входные
данные не меняются при смене статуса/росте числа попыток (проверяется через `TaskView`).
**TID-03. Разные Task независимы.** Открытие двух Task даёт **разные** идентификаторы и
независимые состояния; смена статуса одного не затрагивает другой.

*Сценариев идентичности: **3**.*

---

## 8. Event Tests

Для каждого события Task: **когда возникает** и **когда возникать не должно**. Все события
Task фиксируются в журнале событий владеющего Run.

**TEV-01. TaskCreated.**
- *Возникает:* в момент открытия Task (`Pending`).
- *Не возникает:* при последующих переходах того же Task.

**TEV-02. TaskStarted.**
- *Возникает:* при первом входе в исполнение `Pending → Running`.
- *Не возникает:* при повторе (`Running → Running`) и в терминальных переходах.

**TEV-03. TaskCompleted.**
- *Возникает:* при переходе в `Succeeded`.
- *Не возникает:* при `Failed`/`Skipped`.

**TEV-04. TaskFailed.**
- *Возникает:* при переходе в `Failed`; несёт причину.
- *Не возникает:* при `Succeeded`/`Skipped`.

**TEV-05. Skipped без события.**
- *Дано:* Task в `Pending`.
- *Когда:* Task переводится в `Skipped`.
- *Тогда:* выделенного события не порождается (журнал событий Run не пополняется по этому
  переходу).

**TEV-06. События Task в журнале Run.**
- *Дано:* Run с открытым Task.
- *Когда:* Task проходит `open → Running → Succeeded`.
- *Тогда:* события `TaskCreated`, `TaskStarted`, `TaskCompleted` присутствуют в журнале событий
  **владеющего Run**, в порядке возникновения, наряду с событиями самого Run.

*Сценариев событий: **6**.*

## 9. Cross-Entity (Run ↔ Task) Tests

**TXC-01. Нетерминальный Task блокирует `Completed` Run.**
- *Дано:* Run доведён до `Waiting Human`; у него есть Task в нетерминальном состоянии
  (`Pending`/`Running`).
- *Когда:* запрашивается переход Run `Waiting Human → Completed` (Approve).
- *Тогда:* отвергается как доменная ошибка; Run остаётся в `Waiting Human`. *Почему:*
  производство не доведено до исхода по шагам (`TASK_SPEC.md` §5 п. 9; `ADR-0004` §9).

**TXC-02. Терминальные Task не препятствуют `Completed`.**
- *Дано:* Run в `Waiting Human`; все его Task в терминальном состоянии (`Succeeded`/`Skipped`).
- *Когда:* запрашивается переход Run `Waiting Human → Completed` (Approve).
- *Тогда:* переход выполняется; Run в `Completed`.

*Сценариев межсущностной согласованности: **2**.*

## 10. Regression Tests

Минимальный набор, который обязан проходить **после любого изменения** Task. Новых сценариев не
вводит — ссылается на существующие:

- **Поток:** `THP-01` (открытие), `THP-02`/`THP-03` (старт+успех), `THP-04` (пропуск).
- **Повторы:** `TRW-01` (повтор +1), `TRW-02` (предел).
- **Запреты:** `TFL-01` (запрещённый переход), `TFL-03` (смена статуса чужим актёром).
- **Границы и идентичность:** `TAGG-01` (Task принадлежит Run), `TID-01` (неизменность
  идентификатора).
- **События и согласованность:** `TEV-06` (события Task в журнале Run), `TXC-01` (блокировка
  `Completed`).

*Регрессионный набор: **11** сценариев (подмножество, без новых ID).*

### 3. Требования, которые невозможно проверить напрямую

Честно отмечаем ограничения **сценарной (примерной)** приёмки:

- **§5 п. 2 / §9 «неизменяемость входа на уровне внутренней сущности».** Снаружи корня нет
  изменяемого объекта Task и нет публичной фабрики, поэтому «попытку изменить неизменяемый
  атрибут» нельзя выразить через публичный контракт. Неизменяемость проверяется **позитивно**:
  поля входа стабильны на всём жизненном цикле (`TID-02`), а наружу отдаётся только снимок
  `TaskView` (`TAGG-03`). Защита внутренней сущности (доменная ошибка при прямой переустановке
  поля) — это **структурная гарантия дизайна** (зеркало `Run`), не выражаемая через публичный
  API, и потому не покрывается отдельным позитивным тестом.
- **§9 «Что НЕ входит в Task» (отсутствие связей с Output/Notion/LLM/Workflow и т. п.).** Это
  утверждения об **отсутствии** ответственности; проверяются **косвенно** — у Task нет таких
  данных/операций в контракте.
- **Терминальность как liveness-свойство (§5 инв. 5).** Доказуема **структурно** (отсутствие
  исходящих рёбер у терминалов + ограниченность попыток `TRW-02`), а не как абстрактная
  «вечность».
- **Механика доставки событий (шины/брокеры).** Сознательно **вне домена**: проверяется только
  **семантика возникновения** события (`TEV-*`), но не его транспорт.

### 4. Подтверждение готовности к pytest

Подтверждаю: после этого документа **можно переходить к написанию pytest-тестов**
(`tests/test_task.py`). Каждый сценарий сформулирован как проверяемое «Дано / Когда / Тогда» с
уникальным идентификатором (прямое отображение «сценарий → тест»). Покрыты все нормативные
требования `TASK_SPEC.md`; ограничения проверяемости явно зафиксированы (пункт 3).

> При расхождении с `TASK_SPEC.md` исправляется **этот** документ: источник истины —
> `TASK_SPEC.md` → `ADR-0004` → `DOMAIN_MODEL.md` → `PROJECT.md` (раздел 17).
