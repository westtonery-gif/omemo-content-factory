## 1. Назначение Schema

**Schema** — типизированное описание формы данных на границе агента и **единый источник истины**
для генерации, валидации и документирования контракта (`DOMAIN_MODEL.md` §2.8; `PROJECT.md` §9:
«нет схемы — нет границы»). Schema делает границу между шагами **валидируемым контрактом**, а не
свободным текстом.

**Почему Schema — самостоятельный Aggregate Root, а не дочерняя сущность.**

- Schema имеет **независимую идентичность и версионирование**; она **референсится** другими
  сущностями (Output, в будущем — Workflow Step), но **не принадлежит** ни одной из них
  (`DOMAIN_MODEL.md` §9.5; `ADR-0008` §2). Её целостность нельзя обеспечивать через чужой корень.
- Schema — сущность **каталога определений**, а не рантайм-состояния прогона. Она существует
  **вне и до** любого Run и переиспользуется многими прогонами и Output.
- В отличие от Run/Task/Output, Schema **не подчинена** идентичности производства (Run): её
  жизненный цикл — отдельная ось (раздел 5; раздел 10).

**Почему Schema важна.**

- Без Schema инвариант Structured Outputs (`PROJECT.md` §9) не реализуем на уровне домена:
  именно Schema превращает результат шага в **проверяемый по контракту** Output.
- Schema — корень дерева зависимостей определений: на неё будут ссылаться Workflow Step и Agent
  (выходной контракт роли). Поэтому она фиксируется раньше прочих определений.

## 2. Ответственность Schema

**Schema обязана:**

- Нести **зафиксированный контракт** — версионируемое описание формы данных на границе агента
  (набор обязательных полей и их доменный смысл — раздел 4).
- Удерживать **ровно один** текущий статус из своего жизненного цикла (раздел 5).
- **Допускать только разрешённые** переходы статуса и **отвергать запрещённые** как ошибку
  предметной области.
- Обеспечивать **неизменяемость принятой версии**: содержание версии не меняется после фиксации;
  новая ревизия — это **новая версия** (`DOMAIN_MODEL.md` §6).
- Предоставлять **детерминированное правило валидации** результата относительно своего контракта
  (раздел 4) — без обращения к внешним системам.

**Schema никогда не должна:**

- **Принимать содержательных/творческих решений** (это делает Agent). Schema задаёт форму, а не
  содержание.
- **Управлять конвейером** — решать, какой шаг следующий, или менять состояние Run/Task/Output.
- **Изменяться в процессе исполнения Run** или быть частью жизненного цикла Run (раздел 10).
- **Зависеть от провайдера, хранилища или инфраструктуры**: системная валидация — чистая
  доменная функция (`PROJECT.md` §9; провайдер-нативная половина «двойной защиты» — инфраструктура
  и в эту сущность не входит).
- Менять своё **зафиксированное содержание версии** или менять статус «сама по себе»: статус
  меняется только по запросу владельца каталога и только по разрешённым рёбрам.

> Кратко: **Schema — это версионируемый, неизменяемый контракт границы и правило его проверки, а
> не исполнитель, не творец и не часть рантайма прогона.**

## 3. Внутренние данные (бизнес-атрибуты)

> Описаны бизнес-свойства, без технических типов. Для каждого: *зачем · обязателен ли · кто
> вправе изменять*. Источник: `DOMAIN_MODEL.md` §2.8, §11; `ADR-0008` §3, §5.

**Идентичность и версия (неизменяемы после создания):**

- **SchemaId** — стабильный идентификатор **логического контракта**, **общий для всех его
  версий**. *Обязателен.* *Изменять: никто* (назначается при создании, далее неизменен).
- **SchemaVersion** — **Value Object** (`DOMAIN_MODEL.md` §11): версия определения; не имеет
  собственной идентичности и жизненного цикла, сравнивается **по значению** («v3 равно v3
  всегда»). *Обязателен.* *Изменять: никто* — новая ревизия порождает **новое значение** версии.
- Пара **(SchemaId, SchemaVersion)** однозначно идентифицирует **одну неизменяемую версию**
  контракта.

**Содержание контракта (неизменяемо после фиксации версии):**

- **Описание контракта** — доменное назначение схемы (что за форму данных она описывает).
  *Обязателен.* *Изменять: никто* после фиксации версии.
- **Набор обязательных полей** — перечень полей контракта и их доменный смысл; «правила
  обязательности» (`DOMAIN_MODEL.md` §2.8). Минимальное достаточное описание границы как
  контракта (раздел 4). *Обязателен.* *Изменять: никто* после фиксации версии.

**Состояние:**

- **Текущий статус** — ровно одно состояние из жизненного цикла (раздел 5). *Обязателен.*
  *Изменять: только владелец каталога определений*, и только по разрешённому переходу.

> Schema **не дублирует** содержимое Output: Output несёт фактическую полезную нагрузку и вердикт
> валидности; Schema несёт **контракт**, относительно которого этот вердикт выносится.

## 4. Контракт и правило валидации

**Контракт (минимальное достаточное описание).** На уровне домена контракт Schema — это
**набор обязательных полей** и их доменный смысл (`DOMAIN_MODEL.md` §2.8; `ADR-0008` §5).
Конкретные технические форматы полей здесь **не проектируются** (`ARCHITECTURE.md` §11): они —
уровень реализации. Обязательный набор полей — это наименьшее достаточное выражение
агент-границы как **проверяемого контракта** (`PROJECT.md` §9).

**Правило валидации (детерминированное, без внешних зависимостей).** Schema выносит по результату
бинарный **вердикт**:

- результат признаётся **валидным** тогда и только тогда, когда **все обязательные поля контракта
  присутствуют и непусты**;
- иначе результат признаётся **невалидным**.

Свойства правила:

- **Детерминированность.** При одном и том же результате и одной и той же версии Schema вердикт
  всегда одинаков.
- **Системность и самодостаточность.** Это валидация **на стороне системы** (`PROJECT.md` §9),
  выполняемая **без обращения к внешним сервисам, провайдеру, сети или хранилищу**.
- **Вердикт — нормальный исход, а не ошибка.** Невалидность — это управляемый результат проверки
  (`PROJECT.md` §10), а не техническое исключение и не нарушение инварианта.
- **Только против `ACTIVE`.** Валидация допустима **исключительно** относительно версии в статусе
  `ACTIVE` (раздел 5; инвариант 6 в разделе 6).

> Вердикт правила валидации проявляется во внешней сущности — **Output** — как его состояние
> `Valid`/`Invalid` (раздел 9). Сама Schema вердикт **не хранит**: она его **выносит**.

## 5. Жизненный цикл

Три состояния. Жизненный цикл Schema — **отдельная ось**, не связанная с жизненным циклом Run
(раздел 10). Принятая версия **неизменяема**; изменение контракта = **новая версия**
(`DOMAIN_MODEL.md` §2.8, §6).

```
   ┌─────────┐
   │  Draft  │  версия контракта оформляется; ещё не пригодна для валидации
   └────┬────┘
        │ (ввод в обращение)
        ▼
   ┌─────────┐
   │ Active  │  единственное состояние, относительно которого валидируется Output
   └────┬────┘
        │ (вывод из обращения)
        ▼
   ┌────────────┐
   │ Deprecated │  терминал: версия выведена из обращения
   └────────────┘
```

**Описание состояний:**

- **Draft** — версия контракта существует, но **не используется** для валидации.
- **Active** — версия введена в обращение; **только** относительно неё допустима валидация Output.
- **Deprecated** — терминал: версия выведена из обращения; новой валидации против неё не
  выполняется.

**Допустимые переходы** (`DOMAIN_MODEL.md` §2.8; `ADR-0008` §4):

- `Draft → Active` (ввод в обращение)
- `Active → Deprecated` (вывод из обращения)

**Запрещённые переходы (нарушение = ошибка предметной области):**

- `Active → Draft`, `Deprecated → Active`, `Deprecated → Draft` и любой иной возврат/обход.
- Любой переход **из** терминала `Deprecated`.
- Любая смена статуса в обход владельца каталога (раздел 7).
- Любой переход при наличии у версии более одного «текущего» статуса.

**Неизменяемость версии.** Содержание принятой версии (описание, набор обязательных полей) после
фиксации **не меняется** ни в одном из состояний. Смена статуса меняет только пригодность версии к
использованию, но **не её содержание**. Любое изменение контракта вводится как **новое значение
SchemaVersion**.

## 6. Инварианты

> Правила, которые **никогда** не нарушаются. Нарушение — дефект, а не «крайний случай».
> Источник: `DOMAIN_MODEL.md` §6 (Schema, Output), §5, §9.5; `ADR-0008`.

1. **Ровно один текущий статус.** В каждый момент версия Schema находится строго в одном
   состоянии из раздела 5.
2. **Неизменяемая идентичность и версия.** `SchemaId` и `SchemaVersion` не меняются на протяжении
   жизни; `SchemaVersion` — Value Object, сравниваемый по значению (`DOMAIN_MODEL.md` §11, §12).
3. **Неизменяемость принятой версии.** Содержание версии (описание, набор обязательных полей)
   неизменно после фиксации; изменение контракта = **новая версия** (`DOMAIN_MODEL.md` §6).
4. **Только разрешённые переходы.** Допустимы лишь рёбра `Draft → Active → Deprecated` (раздел 5).
5. **Единственный распорядитель статуса.** Статус версии меняет только **владелец каталога
   определений**; Content Director Schema **не** распоряжается (раздел 7).
6. **Валидация только против `ACTIVE`.** Результат может быть провалидирован **исключительно**
   относительно версии в статусе `Active`; валидация против `Draft`/`Deprecated` — ошибка
   предметной области.
7. **Детерминированность валидации.** Вердикт зависит только от результата и версии контракта; он
   воспроизводим и не зависит от внешних факторов (раздел 4).
8. **Инвариант валидности Output (межсущностный).** Output **валиден относительно ровно одной
   `ACTIVE` версии Schema** (`DOMAIN_MODEL.md` §5 — 1:1 «Output — Schema (версия)»; §6 — «Output
   валидируется против ровно одной версии Schema; невалидный Output не передаётся дальше»).
   Один Output ссылается на **ровно одну** версию; одна версия валидирует **многие** Output
   (`Schema → Output` — 1:N).
9. **Самостоятельность корня.** Schema не принадлежит Run/Task/Output: она **референсится**, но не
   владеется ими (`DOMAIN_MODEL.md` §9.5).

## 7. Правила изменения

**Разрешено (только владельцем каталога определений — `DOMAIN_MODEL.md` §2.8):**

- Создание новой версии контракта (версия возникает в `Draft`).
- Запрос допустимого перехода статуса версии (`Draft → Active`, `Active → Deprecated`).

**Запрещено:**

- Менять `SchemaId`, `SchemaVersion` или зафиксированное содержание принятой версии.
- Менять статус по неразрешённому ребру или возвращать версию из терминала.
- Менять статус актёром, отличным от владельца каталога (в частности, **через Content Director или
  через Run** — это нарушение границы: Schema им не принадлежит).
- Допускать более одного текущего статуса.
- Валидировать результат против версии, не находящейся в `Active`.

**Ошибки предметной области (domain errors):**

- Запрос недопустимого перехода версии Schema (например, `Active → Draft`, выход из `Deprecated`).
- Попытка изменить неизменяемый атрибут (идентичность, версия, содержание принятой версии).
- Валидация против версии не в статусе `Active`.

> Все перечисленные ошибки — **доменные**: они описывают нарушение бизнес-правил. Невалидный
> вердикт результата (раздел 4) ошибкой **не** является — это нормальный исход проверки.

## 8. Domain Events

На этом этапе Schema **не порождает доменных событий**: `DOMAIN_MODEL.md` §10 не определяет
событий жизненного цикла Schema, и Schema — корень **каталога**, а не дочерняя сущность Run
(собственного журнала событий у неё нет, в журнал Run она ничего не пишет).

Событие **OutputValidated** (`DOMAIN_MODEL.md` §10) принадлежит сущности **Output** и фиксируется
в журнале её Run, а **не** Schema (`ADR-0008` §9). Schema лишь предоставляет правило, по которому
выносится вердикт.

> Если для жизненного цикла Schema (например, ввод/вывод версии) или для невалидного результата
> потребуется отдельное именованное событие, оно вводится через `ADR` и канонизируется в
> `DOMAIN_MODEL.md` §10 — здесь новой функциональности не добавляется.

## 9. Взаимодействие

Schema — **самостоятельный корень каталога**; она **референсится** другими сущностями, но не
владеется ими и их состоянием не управляет.

- **Output.** Output **ссылается на ровно одну версию Schema**: `Output.schema_ref` есть ссылка на
  **SchemaVersion** (`DOMAIN_MODEL.md` §5; `ADR-0008` §7). Валидность Output определяется
  **относительно этой версии** (раздел 6, инвариант 8). Эта спецификация **не вводит новой логики в
  Run/Task/Content Director**: Schema предоставляет правило валидации и является объектом ссылки;
  структура Output (его идентичность, полезная нагрузка, состояние `Valid`/`Invalid`) описана в
  своей сущности и здесь не изменяется.
- **Task.** Прямой связи Task ↔ Schema на этом этапе **нет**. В предметной модели контракты
  вход/выход шага несёт **Workflow Step** (`DOMAIN_MODEL.md` §2.4), который ссылается на Schema;
  поскольку Workflow Step — отдельная сущность вне границ этого документа, связь Task ↔ Schema
  здесь **не описывается**.
- **Content Director.** Content Director **не владеет** Schema и **не распоряжается** её жизненным
  циклом (раздел 7). Он — распорядитель состояния Run и его детей (`DOMAIN_MODEL.md` §9.1), а
  Schema к ним не относится.
- **Run.** Run **не владеет** Schema и **не содержит** её. Run/Task/Output лишь **ссылаются** на
  уже существующую `Active` версию через `Output.schema_ref` (раздел 10).
- **Workflow / Agent (будущие сущности).** Будут **ссылаться** на Schema как на выходной/граничный
  контракт; в рамках данной спецификации не описываются.

---

## 10. Что НЕ входит в Schema

Schema хранит **версионируемый контракт и правило его проверки**, а не чужие данные, не
поведение и не рантайм-состояние прогона. В частности:

- **Schema НЕ является частью жизненного цикла Run.** Её состояния (`Draft`/`Active`/`Deprecated`)
  — отдельная ось; они **не** соответствуют состояниям Run (`Created … Completed/Failed`) и не
  продвигаются «внутри» прогона.
- **Schema НЕ мутирует в процессе исполнения Run.** Run/Task/Output только **читают** (ссылаются
  на) версию; ни один шаг прогона не меняет статус или содержание Schema. Смена статуса Schema —
  действие владельца каталога **вне** оси исполнения.
- **Schema не хранит вердикт.** Состояние валидности (`Valid`/`Invalid`) принадлежит **Output**;
  Schema вердикт выносит, но не удерживает (раздел 4).
- **Schema не управляет Run/Task/Output** и не меняет их статусы — это логика Content Director над
  состоянием Run.
- **Schema не принимает содержательных решений** и не создаёт контент — это роль Agent.
- **Schema не зависит от провайдера/инфраструктуры**: провайдер-нативный structured-output,
  хранилище, сеть, адаптеры — вне сущности (`DOMAIN_MODEL.md` §8).
- **Schema не вводит новых сущностей**: в рамках этого документа существуют только `Schema` и
  связанные Value Objects `SchemaId`, `SchemaVersion`.

## 11. Примеры

> Сценарии описаны как факты предметной области (состояния, вердикты, ссылки) — без реализации и
> без описания потоков исполнения.

### Корректные сценарии

**A. Ввод версии в обращение.**
`create (Draft) → Active`. Версия становится пригодной для валидации Output. *Почему корректно:*
`Draft → Active` — разрешённый переход.

**B. Валидный результат.**
Результат, в котором **все обязательные поля присутствуют и непусты**, проверяется против `Active`
версии и получает вердикт **валиден**. Соответствующий Output фиксируется как `Valid` и ссылается
на эту версию. *Почему корректно:* правило валидации выполнено (раздел 4).

**C. Невалидный результат.**
Результат, в котором **отсутствует или пусто** хотя бы одно обязательное поле, получает вердикт
**невалиден**; соответствующий Output фиксируется как `Invalid` и дальше по конвейеру **не
передаётся**. *Почему корректно:* это нормальный исход проверки (раздел 4; инвариант 8).

**D. Вывод версии из обращения.**
`Active → Deprecated`. Ранее зафиксированные Output, ссылавшиеся на эту версию, **не меняются** —
они привязаны к неизменяемой версии. *Почему корректно:* `Active → Deprecated` разрешён; версия
неизменяема (инвариант 3).

**E. Новая ревизия контракта.**
Изменение контракта оформляется как **новое значение SchemaVersion** (новая версия в `Draft`),
а не как правка существующей. *Почему корректно:* принятая версия неизменяема (инвариант 3).

### Некорректные сценарии

**1. `Active → Draft` (возврат версии).** *Почему ошибка:* недопустимый переход (раздел 5).

**2. Выход из `Deprecated`.** *Почему ошибка:* `Deprecated` — терминал.

**3. Правка содержания принятой версии.** *Почему ошибка:* нарушает неизменяемость версии
(инвариант 3); изменение контракта = новая версия.

**4. Валидация результата против `Draft`/`Deprecated` версии.** *Почему ошибка:* валидация
допустима только против `Active` (инвариант 6).

**5. Смена статуса Schema через Run или Content Director.** *Почему ошибка:* нарушает границу —
Schema самостоятельный корень каталога, ей распоряжается владелец каталога, не оркестратор
(инвариант 5; раздел 7).

**6. Output, валидный относительно более чем одной версии Schema.** *Почему ошибка:* нарушает
инвариант валидности Output (ровно одна версия — инвариант 8).
