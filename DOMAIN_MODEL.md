## 0. Как читать этот документ

- **Доменная сущность** — бизнес-понятие, которое существовало бы независимо от того, на
  чём система реализована. Если заменить Notion, LLM или хранилище — сущность останется.
- **Атрибуты** описаны как **бизнес-свойства**, а не поля БД и не типы данных. Конкретные
  типы, форматы и хранение — уровень реализации/ADR.
- **Актёры домена** (роли, действующие над сущностями): **Content Director** (оркестратор),
  **Agent** (включая **QA Agent**), **Human Reviewer** (человек-рецензент), **Editor/Author**
  (автор брифа), **Analytics Agent**. Внешние системы (Notion, n8n, Google Docs, LLM) —
  **не** актёры домена, а инфраструктура (раздел 8).
- **Категории сущностей:**
  - *Спрос/вход:* Content Brief, Content Type.
  - *Определения (версионируемый каталог конфигурации):* Workflow, Workflow Step, Agent,
    Skill, Prompt, Schema, (Tool — пограничная, см. раздел 8).
  - *Исполнение (рантайм):* Run, Task, Output, Artifact.
  - *Качество и управление:* Evaluation, Human Review.
  - *Наблюдаемость:* Analytics Record.

## 2–4. Каталог сущностей

Для каждой сущности: **Назначение · Ответственность · Жизненный цикл · Владелец данных ·
Создаёт · Изменяет · Читает · Завершает использование · Обязательные атрибуты ·
Необязательные атрибуты · Состояние · Переходы.**

> Сущности-*определения* (Workflow, Agent, Skill, Prompt, Schema, Content Type) —
> **версионируемые**: версия неизменяема (immutable), новая ревизия = новая версия.

---

### 2.1 Content Brief  *(вход/спрос)*

- **Назначение.** Производственное задание — исходный запрос на единицу контента (тема,
  цель, требования, тип контента, редакционные параметры).
- **Ответственность.** Нести бизнес-намерение «что нужно произвести и зачем»; быть
  зафиксированным входом для воспроизводимости (PROJECT.md, раздел 2).
- **Жизненный цикл.** От постановки редактором до завершения производства (или архивации).
- **Владелец данных.** Редакционный домен (Editor/Author). Внутри производства бриф
  считается **неизменяемым снимком** на момент старта Run.
- **Создаёт.** Editor/Author.
- **Изменяет.** Editor/Author — пока бриф в `Draft`. После `Ready` (передан в производство)
  содержательно не меняется; правки — это новая версия брифа.
- **Читает.** Content Director (для запуска Run), Agents (как вход), Human Reviewer
  (как контекст решения).
- **Завершает использование.** Editor/Author (архивация) либо система — при достижении
  Run терминального состояния `Completed`.
- **Обязательные атрибуты.** идентификатор; тема/задача; тип контента (ссылка на Content
  Type); цель; ключевые требования; язык; дата постановки.
- **Необязательные атрибуты.** референсы/источники; ограничения (тон, длина, дисклеймеры);
  приоритет; целевая площадка; заметки редактора.
- **Состояние.** `Draft` · `Ready` · `In Production` · `Fulfilled` · `Archived`.
- **Переходы.** `Draft → Ready → In Production → Fulfilled`; из любого нетерминального →
  `Archived` (отмена). `Fulfilled`/`Archived` — терминальные.

---

### 2.2 Content Type  *(вход/спрос, предлагаемая сущность)*

- **Назначение.** Классификатор формата контента (статья, карусель, AI-видео и т. д.).
  *Зачем нужна:* тип контента — параметр среза, а не ограничение архитектуры (PROJECT.md,
  раздел 2); он связывает бриф с подходящим Workflow и обеспечивает масштабирование на
  новые форматы добавлением, а не переписыванием.
- **Ответственность.** Определять «какого рода материал» и какие Workflow к нему применимы.
- **Жизненный цикл.** Каталожная запись: вводится, используется, при необходимости
  выводится из обращения.
- **Владелец данных.** Системный каталог (архитекторы/редакция).
- **Создаёт / Изменяет.** Владелец каталога (через ADR/конфигурацию).
- **Читает.** Content Brief, Workflow, Content Director, Analytics Agent.
- **Завершает использование.** Владелец каталога (перевод в `Deprecated`).
- **Обязательные атрибуты.** идентификатор; имя; описание.
- **Необязательные атрибуты.** ограничения формата по умолчанию; рекомендуемые Workflow.
- **Состояние.** `Active` · `Deprecated`.
- **Переходы.** `Active → Deprecated`.

---

### 2.3 Workflow  *(определение)*

- **Назначение.** Декларация конвейера для типа контента: какие шаги, в каком порядке,
  какие контрольные точки (PROJECT.md, раздел 18: «Workflow ≠ Orchestrator»).
- **Ответственность.** Описывать **путь** контента (что должно произойти), не исполняя его.
- **Жизненный цикл.** Версионируемое определение: создаётся, активируется, заменяется новой
  версией, выводится из обращения.
- **Владелец данных.** Системный каталог определений.
- **Создаёт / Изменяет.** Архитектор/инженер (через PR/ADR). Изменение = новая версия;
  принятая версия неизменяема.
- **Читает.** Content Director (исполняет), Run (ссылается на версию), Analytics Agent.
- **Завершает использование.** Владелец каталога (перевод версии в `Deprecated`).
- **Обязательные атрибуты.** идентификатор; версия; ссылка на Content Type; упорядоченный
  набор Workflow Step; определение контрольных точек (Approval Gates); терминальные условия.
- **Необязательные атрибуты.** описание; правила маршрутизации/ветвления (для будущих
  этапов); политика повторов по умолчанию.
- **Состояние.** `Draft` · `Active` · `Deprecated`.
- **Переходы.** `Draft → Active → Deprecated`.

---

### 2.4 Workflow Step  *(определение, предлагаемая сущность)*

- **Назначение.** Один шаг внутри Workflow: привязка «позиция в конвейере → роль (Agent) +
  контракт». *Зачем нужна:* отделяет **определение** шага (декларация) от его **исполнения**
  (Task), что критично для масштабирования и переиспользования.
- **Ответственность.** Задавать, какой Agent выполняет шаг, какие у шага входной/выходной
  контракты (Schema) и является ли шаг контрольной точкой.
- **Жизненный цикл.** Часть Workflow (композиция); собственной независимой жизни вне своей
  версии Workflow не имеет.
- **Владелец данных.** Родительский Workflow (агрегат).
- **Создаёт / Изменяет.** Через изменение версии родительского Workflow.
- **Читает.** Content Director (при маршрутизации), Run (порождает Task по шагу).
- **Завершает использование.** Вместе с выводом версии Workflow.
- **Обязательные атрибуты.** идентификатор шага; порядковая позиция; ссылка на Agent;
  входной контракт (Schema); выходной контракт (Schema); признак контрольной точки.
- **Необязательные атрибуты.** условие выполнения/пропуска; локальная политика повторов.
- **Состояние.** Наследует состояние родительского Workflow (отдельного цикла нет).
- **Переходы.** —

---

### 2.5 Agent  *(определение)*

- **Назначение.** Узкая производственная роль («кто и зачем»), компонующая поведение из
  Skills и Tools и возвращающая Output по схеме (PROJECT.md, раздел 8).
- **Ответственность.** Выполнять один тип содержательной работы; **не** управлять конвейером.
- **Жизненный цикл.** Версионируемое определение роли: вводится, активируется, заменяется.
- **Владелец данных.** Системный каталог определений.
- **Создаёт / Изменяет.** Архитектор/инженер (через PR/ADR); привязка «роль → класс модели»
  — внешняя, в конфигурации (PROJECT.md, раздел 5).
- **Читает.** Content Director (вызывает через Task), Workflow Step (ссылается),
  Analytics Agent.
- **Завершает использование.** Владелец каталога (перевод в `Deprecated`).
- **Обязательные атрибуты.** идентификатор; имя роли; зона ответственности; ссылка на
  активный Prompt; выходной контракт (Schema); набор используемых Skills; класс требуемой
  модели (в терминах требований, не имени).
- **Необязательные атрибуты.** набор используемых Tools; лимиты/политики; критерии качества
  (на уровне роли).
- **Состояние.** `Active` · `Deprecated`.
- **Переходы.** `Active → Deprecated`.

---

### 2.6 Skill  *(определение)*

- **Назначение.** Переиспользуемый модуль одной задачи — строительный блок агентов
  (PROJECT.md, раздел 14).
- **Ответственность.** Выполнять одну операцию; **не** знать об агентах и **не** управлять
  конвейером.
- **Жизненный цикл.** Версионируемый модуль каталога: вводится, используется, заменяется.
- **Владелец данных.** Системный каталог определений.
- **Создаёт / Изменяет.** Инженер (через PR/ADR); новая ревизия = новая версия.
- **Читает.** Agents (используют), Analytics Agent.
- **Завершает использование.** Владелец каталога (перевод в `Deprecated`).
- **Обязательные атрибуты.** идентификатор; имя; назначение (одна задача); входной контракт;
  выходной контракт.
- **Необязательные атрибуты.** ограничения/предусловия; зависимость от Tool.
- **Состояние.** `Active` · `Deprecated`.
- **Переходы.** `Active → Deprecated`.

---

### 2.7 Prompt  *(определение)*

- **Назначение.** Версионируемый артефакт поведения агента (System/User, Examples при
  необходимости, Version, Evaluation Criteria — PROJECT.md, раздел 15).
- **Ответственность.** Задавать поведение, ограничения и критерии качества роли; **не**
  управлять конвейером.
- **Жизненный цикл.** Версионируемый: `Draft → Active → Deprecated`; активная версия
  фиксируется в трассировке каждого вызова.
- **Владелец данных.** Принадлежит своему Agent (агрегат «роль»), хранится отдельно от кода.
- **Создаёт / Изменяет.** Инженер/редактор промптов (через PR-ревью).
- **Читает.** Agent (использует), Human Reviewer и Analytics Agent (как контекст версии),
  Evaluation (критерии).
- **Завершает использование.** Перевод версии в `Deprecated` при выходе новой.
- **Обязательные атрибуты.** идентификатор; версия; System Prompt; User Prompt (шаблон);
  ссылка на выходную Schema; критерии оценки (Evaluation Criteria).
- **Необязательные атрибуты.** Examples (few-shot); заметки об изменении.
- **Состояние.** `Draft` · `Active` · `Deprecated`.
- **Переходы.** `Draft → Active → Deprecated`.

---

### 2.8 Schema  *(определение)*

- **Назначение.** Типизированное описание формы данных на границе агента — единый источник
  истины для генерации, валидации и документации контракта (PROJECT.md, раздел 9).
- **Ответственность.** Делать границу между шагами **валидируемым контрактом**, а не
  свободным текстом («нет схемы — нет границы»).
- **Жизненный цикл.** Версионируемый контракт: `Draft → Active → Deprecated`; версия
  неизменяема.
- **Владелец данных.** Системный каталог контрактов.
- **Создаёт / Изменяет.** Инженер/архитектор (через PR/ADR при изменении контракта).
- **Читает.** Agent (формирует Output по схеме), Content Director и система валидации,
  Output (проверяется против схемы).
- **Завершает использование.** Перевод версии в `Deprecated`.
- **Обязательные атрибуты.** идентификатор; версия; описание контракта (набор полей и их
  семантика на доменном уровне); правила обязательности.
- **Необязательные атрибуты.** правила совместимости/миграции с предыдущей версией.
- **Состояние.** `Draft` · `Active` · `Deprecated`.
- **Переходы.** `Draft → Active → Deprecated`.

---

### 2.9 Run  *(исполнение)*

- **Назначение.** Один сквозной проход брифа через Workflow с полной трассировкой —
  единица состояния и аудита (PROJECT.md, раздел 18; ARCHITECTURE.md, раздел 10).
- **Ответственность.** Удерживать состояние и историю производства одной единицы контента.
- **Жизненный цикл.** От создания до терминального состояния (`Completed`/`Failed`).
- **Владелец данных.** Производственный домен; распорядитель состояния — **Content Director**
  (единственный, кто меняет статус). Run — **агрегат**, владеющий своими Task, Output,
  Artifact, Evaluation, Human Review, Analytics Record.
- **Создаёт.** Content Director (по готовому Content Brief).
- **Изменяет (состояние).** Только Content Director.
- **Читает.** Agents, QA, Human Reviewer, Analytics Agent.
- **Завершает использование.** Content Director (перевод в терминальное состояние).
- **Обязательные атрибуты.** идентификатор; ссылка на Content Brief; ссылка на версию
  Workflow; текущий статус; метка времени последнего перехода.
- **Необязательные атрибуты.** счётчик итераций доработки; причина отказа/сбоя; сводные
  ссылки на ключевые артефакты.
- **Состояние.** `Created` · `Queued` · `Running` · `Waiting QA` · `Waiting Human` ·
  `Completed` · `Failed` (ARCHITECTURE.md, раздел 5).
- **Переходы.** `Created → Queued → Running → Waiting QA → Waiting Human → Completed`;
  из `Waiting QA`/`Waiting Human` — возврат в `Running` (доработка); из любого
  нетерминального → `Failed`. `Completed`/`Failed` — терминальные.

### 2.10 Task  *(исполнение, предлагаемая сущность)*

- **Назначение.** Исполнение **одного** Workflow Step в рамках Run — единичная единица
  работы, выполняемая одним Agent. *Зачем нужна:* отделяет рантайм-исполнение шага от его
  определения (Workflow Step) и является носителем метрик и повторов.
- **Ответственность.** Представлять «один вызов одной роли»: вход, исполнение, результат,
  попытки, исход.
- **Жизненный цикл.** От постановки до завершения в рамках своего Run.
- **Владелец данных.** Родительский Run (агрегат).
- **Создаёт.** Content Director (порождает Task по очередному Workflow Step).
- **Изменяет (статус).** Content Director (на основе результата Agent/политики повторов).
- **Читает.** Agent (получает вход), Analytics Agent, Human Reviewer (как контекст).
- **Завершает использование.** Content Director (перевод в терминальный исход).
- **Обязательные атрибуты.** идентификатор; ссылка на Run; ссылка на Workflow Step;
  ссылка на Agent; входные данные (структурированные); статус; число попыток.
- **Необязательные атрибуты.** причина ошибки; контекст доработки (замечания предыдущего
  ревью/QA); ссылка на произведённый Output.
- **Состояние.** `Pending` · `Running` · `Succeeded` · `Failed` · `Skipped`
  (с под-состоянием `Retrying` внутри `Running`).
- **Переходы.** `Pending → Running → Succeeded`; `Running → Retrying → Running`;
  `Running → Failed`; `Pending → Skipped` (по условию шага). `Succeeded`/`Failed`/`Skipped`
  — терминальные для Task.

---

### 2.11 Output  *(исполнение, доменное уточнение «Structured Output»)*

- **Назначение.** Структурированный результат Task, сформированный Agent по своей Schema
  (PROJECT.md, раздел 9). *Отличие от Artifact:* Output — это **контрактный результат шага**
  (контент + метаданные: уверенность, флаги риска, обоснования); Artifact — **единица
  контента**, которую этот Output порождает.
- **Ответственность.** Нести проверяемый по схеме результат шага и служебные метаданные для
  маршрутизации и одобрения.
- **Жизненный цикл.** Создаётся при завершении Task; **неизменяем** после фиксации.
- **Владелец данных.** Родительский Run (через Task).
- **Создаёт.** Agent (при выполнении Task).
- **Изменяет.** Никто (immutable). Новая попытка → новый Output.
- **Читает.** Content Director (валидирует и маршрутизирует), последующие Agents (как вход),
  Evaluation, Analytics Agent.
- **Завершает использование.** Перестаёт быть «текущим», когда Task переходит в терминал;
  сохраняется как часть трассировки.
- **Обязательные атрибуты.** идентификатор; ссылка на Task; ссылка на версию Schema;
  признак валидности; полезная нагрузка (содержание по схеме); метаданные (уверенность,
  флаги риска).
- **Необязательные атрибуты.** обоснования решений; перечень порождённых Artifact.
- **Состояние.** `Pending` (формируется) · `Valid` · `Invalid` (не прошёл валидацию схемы).
- **Переходы.** `Pending → Valid` или `Pending → Invalid`. Терминально и неизменяемо.

---

### 2.12 Artifact  *(исполнение)*

- **Назначение.** Единица контента/данных, порождённая шагом (структура, черновик,
  отредактированный текст, итоговый материал — PROJECT.md, раздел 18).
- **Ответственность.** Представлять собственно **производимый продукт** на разных стадиях
  готовности.
- **Жизненный цикл.** От чернового состояния до одобрения/публикации либо отклонения/замены.
- **Владелец данных.** Родительский Run (агрегат); **каждый Artifact принадлежит ровно
  одному Run** (инвариант, раздел 6).
- **Создаёт.** Agent (как следствие своего Output).
- **Изменяет.** Через порождение **новой версии** Artifact агентом доработки; сам
  зафиксированный Artifact трактуется как неизменяемая версия.
- **Читает.** Последующие Agents, QA (Evaluation), Human Reviewer, Analytics Agent.
- **Завершает использование.** Human Reviewer (Approve → `Approved`/публикация; Reject) или
  система (`Superseded` при появлении новой версии).
- **Обязательные атрибуты.** идентификатор; ссылка на Run; вид артефакта (структура/черновик/
  …); содержание; версия в рамках Run; статус.
- **Необязательные атрибуты.** ссылка на породивший Output; ссылка на предыдущую версию;
  флаги/замечания.
- **Состояние.** `Draft` · `Candidate` · `Approved` · `Rejected` · `Superseded` ·
  `Published`.
- **Переходы.** `Draft → Candidate → Approved → Published`; `Candidate → Rejected`;
  любой рабочий → `Superseded` (при новой версии). `Published`/`Rejected`/`Superseded` —
  терминальные.

---

### 2.13 Evaluation  *(качество и управление)*

- **Назначение.** Оценка Output/Artifact по заданным критериям. Частный случай — **QA-вердикт**
  (качество и комплаенс домена здоровья, `fail closed`); другой — оценка по Evaluation
  Criteria промпта.
- **Ответственность.** Давать структурированное суждение «пройдено / риск / отклонить» с
  замечаниями и флагами — основу для маршрутизации и для человека.
- **Жизненный цикл.** От запроса оценки до вынесения вердикта; вердикт неизменяем.
- **Владелец данных.** Родительский Run (агрегат).
- **Создаёт.** QA Agent (или иной оценивающий Agent), инициируется Content Director.
- **Изменяет.** Никто после вынесения (immutable); повторная оценка = новая Evaluation.
- **Читает.** Content Director (для маршрутизации), Human Reviewer (как контекст),
  Analytics Agent.
- **Завершает использование.** Фиксацией вердикта; сохраняется в трассировке.
- **Обязательные атрибуты.** идентификатор; ссылка на Run; предмет оценки (ссылка на
  Artifact/Output); вид оценки (QA/качество); вердикт; перечень замечаний/флагов.
- **Необязательные атрибуты.** оценка уверенности; ссылка на применённые критерии (версия
  Prompt/правил).
- **Состояние.** `Pending` · `Passed` · `Flagged` · `Failed`.
- **Переходы.** `Pending → Passed` / `Pending → Flagged` / `Pending → Failed`. Терминально.

---

### 2.14 Human Review  *(качество и управление)*

- **Назначение.** Событие обязательного одобрения человеком на Approval Gate (PROJECT.md,
  раздел 12). Носитель решения и его аудита.
- **Ответственность.** Зафиксировать **информированное** решение человека и сделать его
  прослеживаемым; без `Approve` ничего не публикуется.
- **Жизненный цикл.** От перевода Run в `Waiting Human` до вынесения решения.
- **Владелец данных.** Родительский Run (агрегат).
- **Создаёт.** Content Director (открывает контрольную точку); решение вносит Human Reviewer.
- **Изменяет.** Human Reviewer — до момента вынесения решения; после — неизменяемо.
- **Читает.** Content Director (для маршрутизации), Analytics Agent, аудит.
- **Завершает использование.** Human Reviewer (вынесение решения).
- **Обязательные атрибуты.** идентификатор; ссылка на Run; предмет (ссылка на Artifact-
  кандидат); предоставленный контекст (бриф, флаги QA, версии); решение; автор решения;
  метка времени.
- **Необязательные атрибуты.** причина (для Reject); указания (для Request changes);
  комментарии/правки.
- **Состояние.** `Pending` · `Approved` · `Rejected` · `Changes Requested`.
- **Переходы.** `Pending → Approved` / `Pending → Rejected` / `Pending → Changes Requested`.
  Терминально для данного ревью; `Rejected`/`Changes Requested` инициируют новую итерацию Run.

---

### 2.15 Analytics Record  *(наблюдаемость)*

- **Назначение.** Неизменяемая запись метрик по одному вызову (модель, input/output токены,
  стоимость, латентность, число повторов — PROJECT.md, раздел 16).
- **Ответственность.** Образовывать достоверную, **append-only** статистику для оптимизации.
- **Жизненный цикл.** Создаётся в момент вызова; далее неизменяема (только чтение/агрегация).
- **Владелец данных.** Родительский Run (через Task); агрегируется доменом наблюдаемости.
- **Создаёт.** Система при вызове Agent (как часть трассировки Task).
- **Изменяет.** Никто (immutable).
- **Читает.** Analytics Agent (выводы и предложения), люди/ADR (для оптимизации).
- **Завершает использование.** Не завершается активно; может архивироваться по политике
  хранения.
- **Обязательные атрибуты.** идентификатор; ссылка на Task/Run/Agent; модель; входные токены;
  выходные токены; стоимость; время выполнения; число повторов; метка времени.
- **Необязательные атрибуты.** версия Prompt; класс/имя модели; исход вызова.
- **Состояние.** `Recorded` (единственное, неизменяемое).
- **Переходы.** — (append-only).

## 5. Связи между сущностями

> Связи описаны на доменном уровне (кардинальность + причина). Это **не** схема БД.

**Один-к-одному (1:1)**

- **Task — Output (текущий).** Задача порождает один действующий результат. *Причина:* один
  шаг = один контрактный результат; попытки фиксируются отдельно (Analytics Record), а не
  множат «текущий» Output.
- **Output — Schema (версия).** Результат валидируется против ровно одной версии контракта.
  *Причина:* однозначность валидации и аудита.

**Один-ко-многим (1:N)**

- **Content Type → Content Brief / Workflow.** Один тип — много брифов и (потенциально)
  несколько Workflow. *Причина:* тип переиспользуется; формат не ограничивает архитектуру.
- **Content Brief → Run.** Обычно один Run, но допускаются повторные прогоны/итерации.
  *Причина:* воспроизводимость и повторные попытки производства.
- **Workflow → Run.** Одно определение исполняется многими прогонами. *Причина:* определение
  переиспользуется (Workflow ≠ Run).
- **Workflow → Workflow Step.** Конвейер состоит из упорядоченных шагов. *Причина:*
  композиция определения.
- **Run → Task / Artifact / Evaluation / Human Review / Analytics Record.** Run — агрегат,
  владеющий своими дочерними сущностями. *Причина:* единица состояния и аудита.
- **Agent → Task.** Одна роль выполняет много задач (в разных Run). *Причина:* роль —
  переиспользуемое определение.
- **Agent → Prompt (версии).** У роли множество версий промпта, одна активна. *Причина:*
  версионирование поведения.
- **Schema → Output.** Один контракт валидирует множество результатов. *Причина:*
  переиспользование контракта.
- **Artifact → Evaluation / Human Review.** Артефакт может оцениваться и ревьюиться
  многократно (итерации доработки). *Причина:* итеративность качества.
- **Task → Analytics Record.** Несколько попыток/вызовов → несколько записей метрик.
  *Причина:* учёт каждого вызова отдельно.

**Многие-ко-многим (N:M)**

- **Agent — Skill.** Агент использует много Skills; Skill используется многими агентами.
  *Причина:* переиспользование строительных блоков (PROJECT.md, раздел 14).
- **Agent — Tool.** Агент пользуется многими возможностями; одна возможность доступна многим
  агентам. *Причина:* общие операционные возможности (Tool как capability).
- **Workflow — Agent** (через Workflow Step). Один Workflow задействует много агентов; один
  агент участвует во многих Workflow. *Причина:* агенты — переиспользуемые роли в разных
  конвейерах.

## 6. Domain Invariants

> Правила, которые **никогда** не должны нарушаться. Нарушение — дефект, а не «крайний случай».

**Run**
- Run всегда имеет **ровно один** текущий статус.
- Run всегда ссылается на **ровно один** Content Brief и **ровно одну** версию Workflow.
- Статус Run меняет **только** Content Director.
- Run обязан достигать **терминального** состояния (`Completed`/`Failed`) — «вечный» Run
  недопустим.
- Переход к публикации возможен **только** через `Approve` соответствующего Human Review.

**Workflow / Workflow Step**
- Workflow не может быть `Active` без хотя бы одного шага и **определённого терминального
  условия** (Workflow не может завершаться без финального состояния).
- Каждый Workflow Step привязан **ровно к одному** Agent и имеет входной и выходной контракты.
- Принятая версия Workflow **неизменяема**; изменение = новая версия.

**Task**
- Task всегда выполняется **ровно одним** Agent.
- Task всегда принадлежит **ровно одному** Run и **ровно одному** Workflow Step.
- Task завершается **терминальным** исходом (`Succeeded`/`Failed`/`Skipped`); число попыток
  ограничено политикой повторов (бесконечные ретраи запрещены).

**Output**
- Output всегда **валидируется против ровно одной** версии Schema; невалидный Output **не
  передаётся** дальше по конвейеру.
- Output **неизменяем** после фиксации; новая попытка порождает новый Output.
- Между агентами передаётся **только** структурированный Output — свободный текст запрещён.

**Artifact**
- Artifact всегда принадлежит **ровно одному** Run.
- Каждый Artifact порождён шагом (имеет происхождение); «бесхозных» артефактов нет.
- Ни один Artifact не получает статус `Published` без предшествующего `Approve` (Human Review).
- Зафиксированная версия Artifact неизменяема; правка = новая версия (`Superseded`).

**Agent / Skill / Prompt / Schema (определения)**
- Каждый Agent имеет **ровно один активный** Prompt и определённый выходной контракт (Schema).
- Skill **не** ссылается на агентов и **не** управляет конвейером.
- Привязка «роль → модель» — **внешняя** (в конфигурации), не зашита в Agent/Prompt.
- Принятые версии Prompt и Schema **неизменяемы**.

**Evaluation**
- Evaluation всегда относится **ровно к одному** Run и имеет **ровно один** вердикт.
- Вердикт `Failed`/`Flagged` (риск) обязывает систему **не** пропускать контент дальше
  (`fail closed`) и эскалировать.
- Вердикт неизменяем после вынесения.

**Human Review**
- Human Review всегда имеет **зафиксированного автора** решения и метку времени (аудит).
- Решение принимается **информированно**: рецензенту предоставлен контекст (бриф, флаги QA,
  версии).
- Решение неизменяемо после вынесения; `Reject`/`Changes Requested` **никогда** не ведут
  к публикации.

**Analytics Record**
- Analytics Record **неизменяема** (append-only) и привязана к конкретному Task/Run/Agent.
- Каждый вызов Agent обязан порождать запись метрик; вызов без записи считается несделанным.

**Content Brief**
- После перевода в производство (`Ready`/`In Production`) бриф трактуется как **неизменяемый
  снимок** входа Run (воспроизводимость).

## 7. Жизненный цикл контента

Путь одного материала — от идеи до аналитики. Стрелки — поток создания/преобразования; все
содержательные решения принимают агенты и человек, а маршрутизацию — Content Director.

```
   Идея
     │  (замысел редакции)
     ▼
   Content Brief            создаётся редактором; фиксируется как снимок входа
     │  (Ready → передан в производство)
     ▼
   Workflow                 выбирается по Content Type (декларация пути)
     │  (Content Director запускает исполнение)
     ▼
   Run                      создаётся; ведёт состояние и трассировку
     │  (по каждому Workflow Step порождается)
     ▼
   Task                     единица работы одного шага
     │  (выполняется одной ролью)
     ▼
   Agent                    производит структурированный Output
     │  (Output → порождает)
     ▼
   Artifact                 единица контента (черновик → кандидат)
     │  (оценка качества/комплаенса)
     ▼
   QA  (Evaluation)         вердикт: пройдено / риск (fail closed)
     │  (пройдено → на ревью)
     ▼
   Human Review             человек: Approve / Reject / Request changes
     │  (Approve)
     ▼
   Published                Artifact → состояние «Published» (публикация разрешена)
     │
     ▼
   Analytics                накопление метрик прогона → Analytics Agent → оптимизация

   Ветви возврата: Reject / Request changes / риск QA → новая итерация Run (Running);
   неустранимая ошибка / исчерпание повторов → Run = Failed.
```

---

## 8. Что НЕ является доменной сущностью

Перечисленное — **инфраструктура и механизмы**, а не бизнес-сущности. Критерий: если элемент
можно заменить, не изменив смысла предметной области, — он не часть доменной модели.

- **Notion** — внешняя система постановки задач/статусов. Домену принадлежит **Content Brief**,
  а не инструмент, в котором его завели. Notion заменяем.
- **n8n** — событийный транспорт/триггеры. Это механизм запуска, а не бизнес-понятие.
- **Google Docs** — площадка ревью. Домену принадлежит **Human Review** (решение и аудит),
  а не редактор, где оно сделано.
- **GitHub** — хостинг кода/процессов разработки. К предметной области производства контента
  отношения не имеет.
- **Claude / GPT и любые LLM-модели** — взаимозаменяемая инфраструктура исполнения. Домену
  принадлежат **Agent**, **Prompt**, **Output**; конкретная модель — деталь конфигурации
  (PROJECT.md, раздел 5 — провайдер-независимость).
- **API** — способ взаимодействия между системами. Контракт домена — это **Schema/Output**,
  а не транспортный интерфейс.
- **База данных / хранилище** — механизм персистентности. Домен оперирует **Run/Artifact/…**;
  то, *где* они хранятся, доменно безразлично.
- **Adapter, Tool, CI, pre-commit, очереди** — архитектурные и инженерные механизмы
  (ARCHITECTURE.md), а не бизнес-сущности.

> Принцип: доменная модель должна **пережить замену** любого из этих элементов. Поэтому они
> остаются за её границей — в архитектуре и инфраструктуре, но не в предметной области.

---

## 9. Aggregate Roots

**Aggregate Root** — сущность, через которую обеспечивается целостность группы связанных
объектов (агрегата). Дочерние сущности не существуют и не изменяются «в обход» корня. Это
тактическое усиление уже описанного владения данными (разделы 2–4), без изменения сущностей.

> **Принцип агрегатов.**
> **Все изменения дочерних сущностей происходят только через соответствующий Aggregate
> Root.** Внешние актёры не обращаются к дочерним сущностям напрямую — только к корню,
> который гарантирует инварианты (раздел 6).

### 9.1 Run — корень исполнения

- **Почему root.** Run — единица состояния и аудита; его инварианты (ровно один статус,
  терминальность, публикация только через Approve) можно гарантировать лишь в одной точке.
- **Владеет.** Task, Output, Artifact, Evaluation, Human Review, Analytics Record.
- **Только через Run разрешено.** Создание/переход Task; фиксация Output; создание новой
  версии Artifact и смена его статуса; открытие Evaluation и Human Review; запись Analytics
  Record; любой переход статуса самого Run (распорядитель — Content Director).

### 9.2 Workflow — корень определения конвейера

- **Почему root.** Workflow Step не имеет смысла и идентичности вне своей версии Workflow;
  целостность упорядоченного конвейера и его контрольных точек держит корень.
- **Владеет.** Workflow Step.
- **Только через Workflow разрешено.** Добавление/изменение/упорядочивание шагов; задание
  контрольных точек и терминальных условий — исключительно как новая версия Workflow.

### 9.3 Agent — корень роли

- **Почему root.** Поведение роли и её контракт целостны только вместе с активным промптом;
  версии Prompt принадлежат роли.
- **Владеет.** Prompt (его версии).
- **Только через Agent разрешено.** Введение новой версии Prompt и назначение активной;
  определение выходного контракта роли и набора используемых Skills.
- *Примечание.* Skill и Tool — **референсы** (N:M), а не собственность Agent; они являются
  самостоятельными корнями (см. 9.5).

### 9.4 Content Brief — корень входа

- **Почему root.** Бриф — самостоятельный бизнес-объект спроса с собственной идентичностью и
  инвариантом неизменяемости после передачи в производство.
- **Владеет.** Собственным содержанием (агрегат из одной сущности; дочерних нет).
- **Только через Content Brief разрешено.** Изменение содержания брифа (пока он в `Draft`);
  смена его статуса жизненного цикла.

### 9.5 Дополнительные корни (предлагаются)

Самостоятельные сущности-определения каталога, у которых **нет** родителя и которые лишь
**референсятся** другими, — это отдельные (одиночные) Aggregate Roots:

- **Skill** — корень переиспользуемого модуля; владеет своими версиями.
- **Schema** — корень контракта данных; владеет своими версиями.
- **Content Type** — корень классификатора формата.

*Обоснование.* Они имеют независимую идентичность и версионирование, переиспользуются
многими агентами/воркфлоу и не подчинены ни одной другой сущности; их целостность нельзя
обеспечивать через чужой корень.

> Сводно: корни-исполнения — **Run**; корни-определения — **Workflow**, **Agent**, **Skill**,
> **Schema**, **Content Type**; корень-вход — **Content Brief**.

## 10. Domain Events

**Domain Event** — факт, который **уже произошёл** в предметной области и значим для бизнеса.
Здесь описаны только бизнес-события (что и когда произошло), без механики доставки.
События отражают **уже определённые** переходы (разделы 2–4, 7) и не вводят новой
функциональности.

> Не описываются: реализация, Event Bus, брокеры сообщений, формат полезной нагрузки. Только
> предметный смысл события.

| Событие | Когда возникает | Инициатор | Затрагиваемые сущности |
|---|---|---|---|
| **ContentBriefCreated** | Бриф оформлен и готов к производству (`Ready`) | Editor/Author | Content Brief |
| **RunCreated** | Создан прогон по готовому брифу | Content Director | Run, Content Brief |
| **RunQueued** | Прогон поставлен в очередь на исполнение | Content Director | Run |
| **RunStarted** | Прогон начал исполнение (`Running`) | Content Director | Run |
| **TaskCreated** | Порождена задача по очередному Workflow Step | Content Director | Task, Run, Workflow Step |
| **TaskStarted** | Задача передана в исполнение роли | Content Director | Task, Agent |
| **TaskCompleted** | Задача успешно завершена | Content Director | Task, Output |
| **TaskFailed** | Задача завершилась ошибкой/исчерпала повторы | Content Director | Task, Run |
| **OutputValidated** | Результат прошёл валидацию по версии Schema (`Valid`) | Content Director | Output, Schema, Task |
| **ArtifactCreated** | Шаг породил единицу контента | Agent (через Content Director) | Artifact, Run |
| **EvaluationCompleted** | Вынесен вердикт QA/качества | QA Agent (через Content Director) | Evaluation, Artifact, Run |
| **HumanReviewRequested** | Прогон переведён на Approval Gate (`Waiting Human`) | Content Director | Human Review, Artifact, Run |
| **ArtifactApproved** | Кандидат одобрен человеком | Human Reviewer | Artifact, Human Review |
| **HumanReviewApproved** | Решение человека — Approve | Human Reviewer | Human Review, Run |
| **HumanReviewRejected** | Решение человека — Reject | Human Reviewer | Human Review, Run |
| **RunCompleted** | Прогон достиг `Completed` (артефакт одобрен) | Content Director | Run, Artifact |
| **RunFailed** | Прогон достиг `Failed` | Content Director | Run |
| **AnalyticsRecordCaptured** | Зафиксированы метрики вызова | Система (трассировка Task) | Analytics Record, Task |

*Согласованность с разделом 7.* `HumanReviewRejected` (как и «Changes Requested») инициирует
новую итерацию Run (`Running`), а не публикацию; `ArtifactApproved` обязательно предшествует
переходу артефакта в `Published`. События лишь именуют уже зафиксированные переходы.

## 11. Value Objects

**Value Object (VO)** — объект предметной области, который **не имеет собственной идентичности**,
**неизменяем** и **сравнивается по значению** (два VO с одинаковыми значениями
взаимозаменяемы). VO описывают *характеристики* сущностей, а не самостоятельные бизнес-объекты.

| Value Object | Назначение | Почему VO, а не Entity |
|---|---|---|
| **Version** | Версия определения (Workflow/Agent/Prompt/Schema/…) | Само значение версии не имеет жизненного цикла и идентичности; «v3» равно «v3» всегда |
| **Language** | Язык контента/брифа | Классифицирующее значение; не отслеживается во времени, сравнивается по значению |
| **Confidence Score** | Уверенность агента/оценки | Чистая величина; одинаковое значение взаимозаменяемо, своей жизни не имеет |
| **Token Usage** | Входные/выходные токены вызова | Замер-величина; не изменяется и не имеет идентичности отдельно от записи |
| **Cost** | Стоимость вызова | Денежная величина; определяется значением, неизменяема |
| **Retry Policy** | Правила повторов (предел, задержки) | Набор параметров-значений; две одинаковые политики эквивалентны |
| **Time Range** | Интервал (например, длительность вызова) | Пара значений «начало/конец»; сравнивается по значению, без идентичности |
| **Verdict** *(доп.)* | Исход Evaluation (пройдено/риск/отклонить) + флаги | Значение-суждение; не объект со своей судьбой, эквивалентно по значению |
| **Review Decision** *(доп.)* | Решение человека (Approve/Reject/Changes) + причина | Значение-решение; идентичность несёт само Human Review, не его «решение» |
| **Risk Flag** *(доп.)* | Признак риска/замечание | Помечающее значение; одинаковые флаги взаимозаменяемы |

> Связь с сущностями: например, **Analytics Record** (Entity) описывается значениями
> **Token Usage**, **Cost**, **Time Range**; **Output** несёт **Confidence Score** и **Risk
> Flag**. Меняется не VO «внутри», а сущность получает **новое значение** VO.

## 12. Identity Rules

Правила идентичности отделяют **Entity** (объект с непрерывной идентичностью во времени) от
**Value Object** (объект, определяемый только значением).

**Общие правила идентичности Entity:**

- Идентификатор Entity **никогда не изменяется** на протяжении её жизни.
- **Идентичность отделена от состояния:** Entity остаётся той же при любых сменах статуса и
  атрибутов (Run в `Running` и в `Completed` — одна и та же Run).
- **Изменение атрибутов не создаёт новую Entity** — это та же сущность в новом состоянии.
- **Новая Entity создаётся только при появлении нового бизнес-объекта** (новый бриф → новый
  Content Brief; новый прогон → новый Run; новая попытка шага → новый Output).
- Версионируемые определения: каждая **версия** — это неизменяемое значение определения;
  сама роль/контракт (Agent/Schema/…) сохраняет идентичность поверх своих версий.

**Entity против Value Object:**

| Признак | Entity | Value Object |
|---|---|---|
| Идентичность | Есть (стабильный идентификатор) | Нет |
| Изменяемость | Меняет состояние, оставаясь собой | Неизменяем |
| Сравнение | По идентичности | По значению |
| Жизненный цикл | Есть (состояния/переходы) | Нет |
| Примеры | Run, Task, Artifact, Agent, Human Review | Version, Cost, Token Usage, Confidence Score |

> Практическое следствие: чтобы «изменить» Value Object у сущности, ей присваивают **новое
> значение** VO; чтобы изменить Entity — меняют её **состояние**, сохраняя идентификатор.
> Это согласуется с инвариантами раздела 6 (неизменяемость Output, версий Prompt/Schema и
> записей Analytics Record).
