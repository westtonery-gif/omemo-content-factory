# PROVIDER_MODEL_SPEC.md — Реализационный контракт владения выбором provider/model

> **Место в иерархии документации (PROJECT.md §17).**
> `PROJECT.md` → `ARCHITECTURE.md` → `ROADMAP.md` → `ADR-0016` (ownership of provider/model
> selection) → **`PROVIDER_MODEL_SPEC.md`** → `PROVIDER_MODEL_ACCEPTANCE.md` → `Implementation (Code)`.
>
> **Это НЕ архитектурное и НЕ доменное решение.** ADR-0016 уже решил *кто* владеет выбором
> provider/model (Content Factory). Здесь фиксируется **форма**, в которой это владение приводится
> в существование: конфигурационная поверхность role→provider/model, контракт разрешения
> «роль → готовый `LLMClient`» и границы. Нового архитектурного/доменного решения не вводится.
>
> **Статус:** Proposed. **Дата:** 2026-07-12.

---

## 1. Назначение и границы

### 1.1. Что фиксирует документ

1. **Конфигурационная поверхность выбора** — *семантическую форму*: привязку `role → model`
   (PROJECT §5) и владение выбором **провайдера** (включая fake для keyless); значения берутся из
   окружения (PROJECT §5, §6 — секреты только из окружения, никогда в репозитории). **Гранулярность
   провайдера** (per-role или на уровне адаптера) — предмет реализации, здесь не фиксируется.
2. **Контракт разрешения** — внутри Content Factory, по роли, выдать **готовый `LLMClient`** за уже
   существующим портом (`ADR-0014` §2), сконструированный под выбранные провайдера и модель. Выбор
   живёт **в фабрике**; вызывающая сторона не передаёт ни модель, ни клиент.

### 1.2. Что НЕ входит

- **Точные имена ключей/переменных окружения, формат/сериализация конфигурации, внутреннее
  устройство фабрики клиента.** Это уровень реализации; SPEC фиксирует форму, не код (прецедент:
  `RUN_RESTORE_SPEC` §1.2 отдаёт сериализацию `RunStore`).
- **Добавление/смена/вывод провайдеров или адаптеров.** Переиспользуются существующие
  `AnthropicLLMClient` и `FakeLLMClient` (Slice A). Речь только о **выборе среди существующих**
  (`ADR-0016` Out of scope).
- **Контракт Composition Root.** Не меняется (`ADR-0012`): CR остаётся dumb-компилятором и
  selection-логики не получает.
- **Домен, Workflow, Runtime.** Не затрагиваются.
- **Метрики стоимости/латентности** (PROJECT §16 — отдельный предмет).
- **Сторона вызывающего (main-core): снятие shim выбора модели.** Это **парное, отдельно
  управляемое** изменение вызывающей стороны; настоящий контракт — только CF-сторона (граница
  односторонняя; CF не знает про caller — `ADR-0001` §13.4). См. §6.

### 1.3. Принцип аддитивности

Реализация — **чисто аддитивное** расширение: порт `LLMClient`, адаптер `AnthropicLLMClient`,
`FakeLLMClient` и сигнатуры Composition Root **не меняются** (PROJECT §4.11; `ADR-0016`). Новых
доменных сущностей нет — provider/model это инфраструктура/конфигурация. Выбор провайдера/модели
получает **дом** внутри фабрики, которого раньше не было (`config.py` его явно исключал).

## 2. Почему выбор на стороне вызывающего недостаточен

Сегодня вызывающая сторона читает модель из своего окружения и собирает клиент, инъектируя его в
композицию (`ADR-0016` Context). Это **не реализует** PROJECT §5 («привязка role→model живёт в
конфигурации фабрики; провайдер сменяем без правки логики») и держит знание модели **над** границей
(нарушение `ADR-0001` §13.3). Форма §3–§4 доказывает, что для владения достаточно **конфигурации +
контракта разрешения**, без новых доменных типов и без изменения порта/CR.

| Что отсутствует сегодня | Последствие | Что фиксирует SPEC |
|---|---|---|
| привязка `role → provider/model` внутри CF | модель выбирается снаружи, не конфигурацией фабрики | §3 |
| разрешение «роль → клиент» внутри CF | клиент строит caller; §13.3 «дырявый» | §4 |
| keyless как выбор провайдера (fake) | keyless живёт тумблером у caller, не в конфиге фабрики | §3, §4.2 |

---

## 3. Форма конфигурации `role → provider/model`

- **Состав.** Конфигурация выражает как минимум привязку `role → model` (PROJECT §5): для роли
  (`agent_ref`) — **модель** (строковый идентификатор класса/версии). Выбор **провайдера** — одной из
  существующих реализаций порта `LLMClient` (Anthropic; либо **fake** для keyless) — тоже во владении
  фабрики; **на какой гранулярности** (per-role или на уровне адаптера) — предмет реализации.
- **Источник.** Значения читаются из **окружения** (PROJECT §5, §6). Секреты (ключи API) — только из
  окружения, не из репозитория и не из этого контракта.
- **Keyless — это выбор провайдера.** Выбор **fake**-провайдера (для роли или системно) даёт
  детерминированный keyless-прогон без ключа/сети. Keyless перестаёт быть решением вызывающей
  стороны и становится профилем конфигурации фабрики.
- **Точные ключи/формат — вне SPEC** (§1.2). Фиксируется *что* конфигурация выражает, не *как* она
  записана.

## 4. Контракт разрешения «роль → `LLMClient`»

### 4.1. Роль и вход/выход

Внутри Content Factory существует разрешение (отдельное от Composition Root), которое по **роли**
(`agent_ref`) возвращает **готовый `LLMClient`**, построенный под выбранные конфигурацией провайдера
и модель. Вход — роль; выход — клиент за существующим портом. Внутреннее устройство разрешения —
предмет реализации (§1.2); контракт — только вход/выход и свойства ниже.

### 4.2. Свойства (инварианты формы)

- **Provider-agnostic.** Результат — всегда объект за портом `LLMClient`; выбор провайдера скрыт за
  портом (PROJECT §4.7; `ADR-0014` §2). Смена провайдера/модели не затрагивает агентов, промпты,
  схемы, оркестрацию, домен.
- **Config-driven, без хардкода.** Модель/провайдер берутся только из конфигурации; **ни одна модель
  не зашита** в логику разрешения или агента (PROJECT §5).
- **Keyless.** При выбранном fake-провайдере результат — keyless-клиент; прогон не требует ключа/сети.
- **Fail-closed на отсутствие/некорректность.** Нет привязки для роли или значение некорректно →
  **явный, управляемый отказ**; молчаливого дефолта к захардкоженной модели быть не должно (PROJECT
  §5 «не хардкодится»; §10 «прогон в явном состоянии»).

### 4.3. Постусловие

Исполнитель, собранный через Composition Root, использует провайдера/модель, выбранные
**конфигурацией фабрики** (не переданные на вход для этой роли). Полное отсутствие чтения модели
**над** границей — итог парного снятия shim на стороне main-core (§6), а не только этого контракта.

---

## 5. Чего контракт НЕ делает

- Не добавляет и не удаляет провайдеров/адаптеров (переиспользует существующие).
- Не меняет порт `LLMClient`, `AnthropicLLMClient`, `FakeLLMClient` и сигнатуры Composition Root.
- Не вводит доменных сущностей, состояний, событий, lifecycle.
- Не трогает Workflow, Runtime, финализацию Output.
- Не решает и не описывает **как** устроена сторона вызывающего (main-core) — см. §6.

---

## 6. Связь с вызывающей стороной (для полноты, вне предмета контракта)

Полное владение достигается парой изменений: (а) CF получает форму §3–§4 (этот контракт); (б)
вызывающая сторона (main-core) **перестаёт** читать модель из своего окружения и собирать клиент —
снятие shim (`ADR-0001` §11 п.2). Пункт (б) — **отдельно управляемое** изменение main-core-стороны с
собственной приёмкой; настоящий SPEC его не проектирует и не предписывает (§13.4). Здесь он назван
лишь для полноты картины.
