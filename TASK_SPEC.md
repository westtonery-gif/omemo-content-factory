## 1. Назначение Task

**Task** — исполнение **одного** Workflow Step в рамках Run: единичная единица работы,
выполняемая **одним** Agent (`DOMAIN_MODEL.md` §2.10). Task отделяет рантайм-исполнение шага
(`Task`) от его определения (`Workflow Step`) и является **носителем попыток (attempts) и
исхода** одного вызова одной роли.

**Почему Task — дочерняя Entity, а не Aggregate Root.**

- Task **не существует вне Run**: он порождается по очередному Workflow Step **только через
  Run** и привязан ровно к одному Run (`DOMAIN_MODEL.md` §9.1; `ADR-0004` §9). Его целостность
  нельзя гарантировать «в обход» корня.
- Граница согласованности — это **Run** (см. `RUN_SPEC.md` §1). Task живёт внутри этой границы:
  его создание и любые изменения проходят через Run как корень агрегата.
- Task несёт **идентичность одного исполнения шага**: к нему привязываются вход, статус,
  число попыток и (в будущих этапах) произведённый Output/метрики. Но эта идентичность
  подчинена идентичности производства (Run).

**Почему Task важен.**

- Task — это «один вызов одной роли»: именно на его уровне фиксируются вход шага, попытки
  (retry) и терминальный исход. Без Task жизненный цикл Run (`Running`) был бы «пустым»
  внутри.
- Исход Task служит **основанием** для решений Content Director о переходах самого Run
  (`RUN_SPEC.md` §8): успех/провал задач определяет движение производства.

## 2. Ответственность Task

**Task обязан:**

- Хранить **зафиксированный вход** шага (ссылку на владеющий Run, ссылку на Workflow Step,
  ссылку на Agent и входные данные) **неизменным** на протяжении всей жизни.
- Удерживать **ровно один** текущий статус из своего жизненного цикла (раздел 4).
- **Допускать только разрешённые** переходы статуса и **отвергать запрещённые** как ошибку
  предметной области.
- Учитывать **число попыток (attempts)** и **не превышать** предел, заданный политикой
  повторов (бесконечные ретраи запрещены — `PROJECT.md` §10).
- Достигать **терминального** исхода (`Succeeded`/`Failed`/`Skipped`) — не «зависать» навсегда.
- Существовать и изменяться **только через свой Run** (граница агрегата, `INV-07`).

**Task никогда не должен:**

- **Существовать вне Run** или быть созданным/изменённым в обход корня агрегата.
- Принимать **творческих/содержательных** решений (это делает Agent). Task хранит вход и
  исход, но не создаёт контент.
- **Управлять конвейером** — решать, какой шаг следующий, или менять статус Run; это логика
  Content Director над состоянием Run.
- **Выбирать Agent или модель**, вызывать LLM, обращаться к внешним системам.
- Менять свой **зафиксированный вход** или менять статус «сам по себе»: статус меняется только
  по запросу уполномоченного актёра (Content Director) **через Run** и только по разрешённым
  рёбрам.

> Кратко: **Task — это согласованное состояние и история одного исполнения шага внутри Run, а
> не исполнитель и не творец.**

## 3. Внутренние данные (бизнес-атрибуты)

> Описаны бизнес-свойства, без технических типов. Для каждого: *зачем · обязателен ли · кто
> вправе изменять*. Источник: `DOMAIN_MODEL.md` §2.10; `ADR-0004` §3, §6.

**Идентичность и вход (неизменяемы после создания):**

- **Идентификатор Task** — стабильная идентичность одного исполнения шага. *Обязателен.*
  *Изменять: никто* (назначается при создании через Run, далее неизменен).
- **Ссылка на владеющий Run** — какому Run принадлежит Task; равна идентификатору этого Run
  (`INV-07`). *Обязателен.* *Изменять: никто* после создания.
- **Ссылка на Workflow Step** — какой шаг определения исполняется. *Обязателен.* *Изменять:
  никто* после создания.
- **Ссылка на Agent** — какая роль выполняет шаг. *Обязателен.* *Изменять: никто* после
  создания.
- **Входные данные шага** — зафиксированный структурированный вход исполнения. *Обязателен.*
  *Изменять: никто* после создания.

**Состояние и исполнение:**

- **Текущий статус** — ровно одно состояние из жизненного цикла (раздел 4). *Обязателен.*
  *Изменять: только Content Director через Run*, и только по разрешённому переходу.
- **Число попыток (attempt count)** — сколько раз исполнение входило в `Running` (первый старт
  и каждый повтор). *Обязателен* (стартует с нуля, растёт при входе в `Running`). *Изменять:
  только Content Director через Run* (в рамках перехода в `Running`).
- **Причина ошибки** — почему наступил `Failed`. *Необязателен* (только при `Failed`).
  *Изменять: только Content Director через Run* при терминальном переходе в `Failed`.

> Task **не дублирует** содержимое других сущностей (Output, Artifact, Analytics Record) — он
> держит **принадлежность, вход и состояние одного исполнения шага**; результаты и метрики
> живут в своих сущностях (вне границ этого этапа реализации).

## 4. Жизненный цикл

Пять состояний. `Succeeded`, `Failed` и `Skipped` — **альтернативные терминальные** исходы.
`Retrying` **не является отдельным состоянием**: повтор моделируется как **счётчик попыток
внутри `Running`** (`ADR-0004` §2; ср. `rework_count` у Run, `RUN_SPEC.md` §4).

```
        ┌─────────┐
        │ Pending │  Task создан по Workflow Step, вход зафиксирован, ещё не исполняется
        └────┬────┘
             │
   ┌─────────┼─────────────────────┐
   │ (старт) │                     │ (условие шага)
   ▼         │                     ▼
┌─────────┐  │                ┌─────────┐
│ Running │◄─┘   повтор       │ Skipped │  терминал: шаг пропущен по условию
│         │──────(retry)──┐   └─────────┘
└────┬────┘   attempt +1  │
     │   ▲                │
     │   └────────────────┘
     │
     ├───────────────► ┌───────────┐
     │   (успех)       │ Succeeded │  терминал-успех: исполнение завершено
     │                 └───────────┘
     │
     └───────────────► ┌────────┐
         (ошибка/       │ Failed │  терминал-сбой: исчерпаны попытки / неустранимая ошибка
          исчерпание)   └────────┘
```

**Описание состояний:**

- **Pending** — Task существует, вход зафиксирован; исполнение ещё не начато.
- **Running** — исполнение идёт; число попыток ≥ 1. Повтор (retry) — это **повторный вход** в
  `Running` с увеличением счётчика попыток, а не отдельное состояние.
- **Succeeded** — терминал-успех: шаг исполнен.
- **Failed** — терминал-сбой: зафиксированное наблюдаемое состояние с причиной (исчерпаны
  попытки или неустранимая ошибка).
- **Skipped** — терминал: шаг пропущен по условию (`DOMAIN_MODEL.md` §2.10).

**Допустимые переходы** (`DOMAIN_MODEL.md` §2.10; `ADR-0004` §5):

- `Pending → Running` (первый старт; счётчик попыток → 1)
- `Pending → Skipped` (пропуск по условию шага)
- `Running → Running` (повтор; счётчик попыток +1, ограничен политикой)
- `Running → Succeeded`
- `Running → Failed`

**Запрещённые переходы (нарушение = ошибка предметной области):**

- Любой переход **из** терминала (`Succeeded`/`Failed`/`Skipped`) — терминалы окончательны;
  для новой работы создаётся **новый** Task.
- `Pending → Succeeded` и `Pending → Failed` напрямую (исполнение должно пройти через
  `Running`).
- `Running → Skipped` (пропуск возможен только из `Pending`).
- `Running → Pending` (возврат к ожиданию запрещён).
- Повтор (`Running → Running`), который **превысил бы** предел числа попыток политики.
- Любая смена статуса **не** через Run и **не** Content Director.
- Любой переход при наличии у Task более одного «текущего» статуса.

## 5. Инварианты

> Правила, которые **никогда** не нарушаются. Нарушение — дефект, а не «крайний случай».
> Источник: `DOMAIN_MODEL.md` §6 (Task); `ADR-0004` §9; расширение инвариантов Run на Task.

1. **Ровно один текущий статус.** В каждый момент Task находится строго в одном состоянии.
   *(`INV-04` расширяется на Task; ср. `RUN_SPEC.md` §5 п. 1.)*
2. **Неизменяемый вход.** Идентификатор, владеющий Run, Workflow Step, Agent и входные данные
   не меняются после создания. *Почему:* воспроизводимость и стабильная идентичность исполнения.
3. **Единственный распорядитель статуса.** Статус Task меняет только Content Director — и
   только **через Run** как корень агрегата. *Почему:* управление потоком сосредоточено в
   оркестраторе; агенты не управляют конвейером (`PROJECT.md` §4 п. 2).
4. **Только разрешённые переходы.** Допустимы лишь рёбра из раздела 4.
5. **Обязательная терминальность.** Task обязан достигать `Succeeded`/`Failed`/`Skipped` —
   «вечный» Task недопустим.
6. **Ограниченность попыток.** Число попыток ограничено политикой повторов; повтор сверх
   предела отвергается, а легальным следующим шагом является перевод в `Failed`. *Почему:*
   защита от бесконечных ретраев и неконтролируемой стоимости (`PROJECT.md` §10).
7. **Целостность агрегата (`INV-07`).** Каждый Task принадлежит **ровно одному** Run, его
   ссылка на Run равна идентификатору этого Run, и он **создаётся/изменяется только через**
   Run. «Бесхозных» или сторонне-изменяемых Task не существует.
8. **Прослеживаемость по умолчанию (`INV-09`).** Значимые факты жизни Task фиксируются как
   доменные события **в общем журнале событий своего Run** (раздел 7).
9. **Согласованность с Run (межсущностный инвариант).** Run **не должен** достигать
   `Completed` при наличии **нетерминальных** Task (`ADR-0004` §9). *Почему:* производство не
   завершено успехом, пока его шаги не доведены до исхода.

## 6. Правила изменения

**Разрешено (исключительно через Run как корень — `RUN_SPEC.md` §6):**

- Content Director **открывает** новый Task по очередному Workflow Step (Task возникает в
  `Pending`).
- Content Director **запрашивает** допустимый переход статуса Task; Run (корень) проводит
  проверку авторизации, допустимости ребра и предела попыток, после чего фиксирует переход.
- Обновление **производных** атрибутов Task в рамках перехода: число попыток (при входе в
  `Running`), причина (при `Failed`).

**Запрещено:**

- Создавать или изменять Task **в обход** Run.
- Менять идентификатор Task или его зафиксированный вход (Run, Workflow Step, Agent, входные
  данные).
- Менять статус не через Content Director или по неразрешённому ребру.
- Выполнять повтор сверх предела политики.
- Возобновлять терминальный Task.
- Допускать более одного текущего статуса.

**Ошибки предметной области (domain errors):**

- Запрос недопустимого перехода Task (например, `Pending → Succeeded`, `Running → Skipped`,
  выход из терминала).
- Повтор, превышающий предел числа попыток.
- Смена статуса/открытие Task актёром, отличным от Content Director.
- Попытка изменить неизменяемый атрибут Task.

> Все перечисленные ошибки — **доменные**: они описывают нарушение бизнес-правил, а не
> технические сбои. Технический сбой исполнения шага ведёт к управляемому повтору или к
> `Failed` (с причиной), но не нарушает инварианты.

## 7. Domain Events

Task порождает следующие бизнес-события (`DOMAIN_MODEL.md` §10; `ADR-0004` §7). Все события
Task **фиксируются в общем журнале событий владеющего Run** (`ADR-0004` §7): отдельной шины у
Task нет.

- **TaskCreated** — когда Task создан по очередному Workflow Step (`Pending`). *Инициатор:*
  Content Director (через Run). *Несёт:* ссылку на Run, идентификатор Task, ссылку на Workflow
  Step и ссылку на Agent.
- **TaskStarted** — при **первом** входе в исполнение `Pending → Running`. *Инициатор:*
  Content Director. *Повтор (`Running → Running`) события `TaskStarted` НЕ порождает* (ср.
  `RunStarted` у Run, `RUN_SPEC.md` §7 / `EV-03`).
- **TaskCompleted** — при переходе в `Succeeded`. *Инициатор:* Content Director.
- **TaskFailed** — при переходе в `Failed`. *Инициатор:* Content Director. *Несёт:* причину
  сбоя.

**Без выделенного события:** переход `Pending → Skipped` на этом этапе **не** порождает
отдельного доменного события (`ADR-0004` §7); повтор (`Running → Running`) также не порождает
события.

> Если для пропуска (`Skipped`) или для повтора потребуется отдельное именованное событие, оно
> вводится через `ADR` и канонизируется в `DOMAIN_MODEL.md` §10 — здесь новой функциональности
> не добавляется.

## 8. Взаимодействие

Во всех случаях Task — **дочерняя Entity агрегата Run**: он существует внутри Run и изменяется
**только через него**; смену состояния инициирует Content Director.

- **Run (корень).** Run **владеет** Task. По очередному Workflow Step через Run создаётся Task;
  исход Task (успех/провал) служит основанием для переходов статуса самого Run
  (`Running → Waiting QA`, `Running → Failed`) — но это решение принимает Content Director над
  состоянием Run, а не сам Task. Run **не исполняет** Task и **не выбирает** Agent. Run **не
  завершается** `Completed` при наличии нетерминальных Task (раздел 5 п. 9).
- **Workflow Step.** Task — **исполнение** одного Workflow Step; Task ссылается на шаг, но не
  содержит его определения.
- **Agent.** Task ссылается на **ровно одну** роль (Agent), которая исполняет шаг. Task не
  владеет Agent и не управляет им.
- **Content Director.** Единственный актёр, который открывает Task и запрашивает его переходы
  (через Run).
- **Output / Artifact / Analytics Record.** Произведённый результат шага, единицы контента и
  метрики — **отдельные сущности** (`DOMAIN_MODEL.md` §2.11, §2.12, §2.15), порождаемые в
  будущих этапах. На текущем этапе Task их **не** хранит и **не** реализует.

---

## 9. Что НЕ входит в Task

Task хранит **принадлежность, вход и состояние одного исполнения шага**, а не чужие данные и не
поведение. В частности:

- **Task не управляет Run** и не меняет статус Run — это логика Content Director над состоянием
  Run.
- **Task не выбирает Agent/модель** и не вызывает LLM — это конфигурация и рассуждение Agent.
- **Task не содержит Workflow Step** — он **ссылается** на шаг; определение живёт в Workflow.
- **Task не хранит Output/Artifact/Evaluation/Analytics Record** — это отдельные сущности
  агрегата Run (будущие этапы).
- **Task не знает про Notion / Google Docs / n8n / API / БД / адаптеры** — это инфраструктура
  (`DOMAIN_MODEL.md` §8).
- **Task не существует и не изменяется вне Run** — нет публичной фабрики Task и нет изменяемого
  Task-объекта, доступного снаружи корня (`ADR-0004` §4).
- **Task не принимает содержательных решений** (что написать/как оценить) — он хранит вход и
  исход, но не создаёт контент.

## 10. Примеры

> Сценарии описаны как последовательности состояний и доменных фактов — без реализации.

### Корректные сценарии

**A. Прямой успех.**
`open (Pending) → Running → Succeeded`. Порождены события `TaskCreated`, `TaskStarted`,
`TaskCompleted`; число попыток = 1. *Почему корректно:* пройдены обязательные шаги, достигнут
терминал.

**B. Повтор, затем успех.**
`open (Pending) → Running → Running (retry, попытка 2) → Succeeded`. Второй вход в `Running`
**не** порождает `TaskStarted`; число попыток = 2. *Почему корректно:* повтор в пределах
политики, затем нормальный успех.

**C. Пропуск шага.**
`open (Pending) → Skipped`. Терминал без выделенного события; исполнение не запускалось.
*Почему корректно:* `Pending → Skipped` — разрешённый переход.

**D. Исчерпание попыток.**
`open (Pending) → Running → Running … → (повтор сверх предела отвергнут) → Failed (причина)`.
*Почему корректно:* повтор сверх предела не выполняется; легальный следующий шаг — `Failed`.

### Некорректные сценарии

**1. `Pending → Succeeded` напрямую.** *Почему ошибка:* минует исполнение (`Running`).

**2. `Running → Skipped`.** *Почему ошибка:* пропуск возможен только из `Pending`.

**3. `Succeeded → Running` (повторное открытие терминала).** *Почему ошибка:* терминал
окончателен; для новой работы — новый Task.

**4. Повтор сверх предела политики.** *Почему ошибка:* нарушает ограниченность попыток
(`PROJECT.md` §10).

**5. Смена статуса Task актёром, отличным от Content Director.** *Почему ошибка:* нарушает
инвариант единственного распорядителя статуса.

**6. Создание/изменение Task в обход Run.** *Почему ошибка:* нарушает границу агрегата
(`INV-07`).

**7. Завершение Run в `Completed` при нетерминальном Task.** *Почему ошибка:* производство не
доведено до исхода по шагам (раздел 5 п. 9).
