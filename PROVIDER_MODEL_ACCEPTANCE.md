# PROVIDER_MODEL_ACCEPTANCE.md — Критерии приёмки владения выбором provider/model

> Приёмка реализации `PROVIDER_MODEL_SPEC.md` (по `ADR-0016`). Каждый критерий —
> **фальсифицируемый**: сформулирован так, что при неверной реализации тест **падает**. Проверяется
> **CF-сторона** (capability); парное снятие shim в main-core — отдельная приёмка (§6).
>
> **Статус:** Proposed. **Дата:** 2026-07-12.

---

## 0. Соглашения

- Прогоны **детерминированы**, без реальной сети/ключей. Конфигурация `role → provider/model`
  инъектируется через окружение (или явную подстановку — DI, PROJECT §6), не через глобальное
  состояние.
- «Роль» — `agent_ref` (напр. `script_writer@v1`). «Клиент» — объект за портом `LLMClient`.
- Реальный вызов провайдера не выполняется; проверяется **выбор и форма**, не качество генерации.

---

## 1. Ownership: выбор берётся из конфигурации фабрики

- **Сценарий.** Конфигурация задаёт для роли `R` модель `M` (и провайдера `P`, выбираемого фабрикой).
  Запрашивается разрешение по `R`. Значение модели вызывающей стороной **не передаётся**.
- **Ожидание.** CF возвращает клиент для `R`, построенный под `P`/`M` (при `P=Anthropic` — адаптер с
  моделью `M`).
- **Что доказывает / как падает.** CF владеет выбором. Падает, если разрешение игнорирует конфиг,
  использует захардкоженную модель или требует модель на входе.

## 2. Config-driven: swap-инвариант

- **Сценарий.** Меняется **только** конфигурация роли (`M`→`M'`, либо `P`→другой существующий
  провайдер). Агенты, промпты, схемы, оркестрация, домен — не трогаются.
- **Ожидание.** Разрешённый клиент отражает новые `P'/M'`; ничего, кроме конфигурации, менять не
  потребовалось.
- **Что доказывает / как падает.** PROJECT §5 (модель сменяема без правки логики). Падает, если смена
  модели/провайдера потребовала изменить агент/промпт/схему/оркестратор.

## 3. Keyless как выбор провайдера

- **Сценарий.** Конфигурация выбирает **fake**-провайдера (для роли `R` или системно). `ANTHROPIC_API_KEY` отсутствует.
- **Ожидание.** CF возвращает keyless-клиент; полный прогон роли `R` завершается без ключа/сети,
  вывод валиден по схеме роли.
- **Что доказывает / как падает.** Keyless — профиль конфигурации фабрики (регресс dev-режима, ср.
  Slice A AT-1). Падает, если keyless всё ещё требует ключ или задаётся не конфигом фабрики.

## 4. Fail-closed на отсутствие/некорректность конфигурации

- **Сценарий.** Для роли `R` привязка отсутствует (или значение некорректно). Запрашивается
  разрешение по `R`.
- **Ожидание.** **Явный, управляемый отказ**; никакого молчаливого дефолта к захардкоженной модели.
- **Что доказывает / как падает.** PROJECT §5 («не хардкодится»), §10 (явное состояние). Падает, если
  разрешение молча подставляет какую-либо модель по умолчанию.

## 5. Границы и инварианты сохранены

- **Порт неизменен.** Сигнатура `LLMClient.complete(...)` и типы `AnthropicLLMClient`/`FakeLLMClient`
  не изменены (статическая проверка). Падает при изменении порта.
- **Composition Root неизменен.** CR остаётся dumb-компилятором, **selection-логики не содержит**
  (`ADR-0012`); клиент он получает готовым. Падает, если в CR появилась логика выбора модели.
- **Граница.** Ни один доменный тип CF не пересекает границу из-за этого изменения; направление
  `caller → CF` сохранено; data-контракт (`brief`+контекст ↓, `FactoryResult` ↑) не тронут.
- **Аддитивность.** Существующие тесты CF остаются зелёными (никакой домен/Runtime/Workflow не
  затронут).

## 6. Что приёмка НЕ проверяет (границы)

- **Снятие shim на стороне main-core** (вызывающая сторона перестаёт читать модель/собирать клиент)
  — это **отдельная приёмка main-core-стороны** (`ADR-0001` §11 п.2; §13.4). Здесь не проверяется.
- Точные имена ключей/формат конфигурации (уровень реализации; SPEC §1.2).
- Добавление новых провайдеров/адаптеров (вне scope `ADR-0016`).
- Стоимость/латентность (PROJECT §16 — отдельный предмет).
