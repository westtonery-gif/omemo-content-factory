## 1. Назначение Run

**Run** — один сквозной проход производственного задания (Content Brief) через выбранный
Workflow, от создания до терминального исхода, с полной прослеживаемостью. Это **единица
исполнения, состояния и аудита** всей фабрики контента: «одно производство одной единицы
контента».

**Почему Run — Aggregate Root.**

- Run задаёт **границу согласованности** для всего, что порождается в ходе производства:
  Task, Output, Artifact, Evaluation, Human Review, Analytics Record. Эти сущности не имеют
  смысла вне своего Run и не должны изменяться «в обход» него.
- Только в одной точке можно гарантировать ключевые инварианты (ровно один статус,
  терминальность, «публикация лишь после Approve»). Эта точка — Run как корень.
- Run несёт **идентичность производства**: к нему привязываются все промежуточные и
  итоговые результаты, решения людей и метрики. Без единого корня прослеживаемость
  распадается.

**Почему система строится вокруг Run.**

- Любой содержательный результат (Artifact), любое суждение (Evaluation), любое решение
  человека (Human Review) и любая метрика (Analytics Record) существуют **внутри** какого-то
  Run. Run — это «контейнер истории» одной единицы контента.
- Оркестратор (Content Director) управляет именно жизненным циклом Run; агенты работают в
  контексте Run; человек одобряет результат Run; аналитика агрегируется по Run.
- Воспроизводимость, аудит и «человек в контуре» — свойства, держащиеся на Run. Поэтому Run
  — центральная сущность домена.

## 2. Ответственность Run

**Run обязан:**

- Хранить **зафиксированный вход** производства (ссылку на Content Brief и версию Workflow)
  неизменным на протяжении всей жизни.
- Удерживать **ровно один** текущий статус и историю переходов между состояниями.
- Быть **границей согласованности** своего агрегата: все дочерние сущности создаются и
  изменяются только через Run.
- **Допускать только разрешённые** переходы состояния и **отвергать запрещённые** (как
  ошибку предметной области).
- Обеспечивать **полную прослеживаемость**: связывать с собой все Task, Artifact, Evaluation,
  Human Review, Analytics Record.
- Гарантировать инвариант публикации: переход к успешному завершению возможен **только**
  после явного Approve соответствующего Human Review.
- Достигать **терминального** состояния (`Completed`/`Failed`) — не «зависать» навсегда.

**Run никогда не должен:**

- Принимать **творческих/содержательных** решений (это делают агенты). Run хранит результат,
  но не создаёт его.
- **Управлять конвейером по существу** — решать стратегию маршрутизации; это логика Content
  Director, действующего над состоянием Run и Workflow.
- **Управлять агентами**, вызывать LLM, обращаться к внешним системам (Notion, Google Docs,
  n8n) или выполнять публикацию во внешний мир.
- **Содержать** Workflow, Prompt, Schema, Skill, Tool — он лишь **ссылается** на нужные
  определения/версии.
- **Оценивать качество** контента самостоятельно (это Evaluation/QA).
- Менять свой **зафиксированный вход** или менять статус «сам по себе» — статус меняется
  только по запросу уполномоченного актёра (Content Director) и только по разрешённым рёбрам.

> Кратко: **Run — это согласованное состояние и история производства, а не исполнитель и не
> творец.** Он хранит и охраняет, но не сочиняет и не маршрутизирует.

## 3. Внутренние данные (бизнес-атрибуты)

> Описаны бизнес-свойства, без технических типов. Для каждого: *зачем · обязателен ли · кто
> вправе изменять*.

**Идентичность и вход (неизменяемы после создания):**

- **Идентификатор Run** — стабильная идентичность производства, к которой привязывается всё
  остальное. *Обязателен.* *Изменять: никто* (назначается при создании, далее неизменен).
- **Ссылка на Content Brief** — какое задание производится; зафиксированный вход для
  воспроизводимости. *Обязателен.* *Изменять: никто* после создания.
- **Ссылка на версию Workflow** — по какому именно определению конвейера идёт производство.
  *Обязателен.* *Изменять: никто* после создания (смена пути = новый Run).

**Состояние и история:**

- **Текущий статус** — ровно одно состояние из жизненного цикла (раздел 4). *Обязателен.*
  *Изменять: только Content Director*, и только по разрешённому переходу (через Run как корень).
- **Момент создания** — когда Run возник (`Created`); опора аудита. *Обязателен.*
  *Изменять: никто* после создания.
- **Момент последнего перехода** — отметка времени последней смены статуса. *Обязателен.*
  *Изменять:* обновляется самим фактом перехода (инициатор перехода — Content Director).
- **Момент достижения терминала** — когда Run достиг `Completed`/`Failed`. *Необязателен*
  (появляется при терминировании). *Изменять:* фиксируется при терминальном переходе, далее
  неизменен.

**Контекст доработки и исхода:**

- **Счётчик итераций доработки** — сколько раз Run возвращался на доработку (из `Waiting QA`/
  `Waiting Human`); защита от бесконечных циклов и материал аналитики. *Необязателен* (растёт
  по мере возвратов). *Изменять: только Content Director* при возврате в `Running`.
- **Причина отказа/сбоя** — почему наступил `Failed` или почему была отклонена итерация
  (Reject). *Необязателен* (только при соответствующем событии). *Изменять: только Content
  Director* при терминальном/возвратном переходе.

**Состав агрегата (принадлежность, не «копии»):**

- **Связанные Task** — задачи-шаги данного Run. *Обязателен* как коллекция (может быть пустой
  в самом начале). *Изменять:* пополняется **только через Run** (инициатор — Content Director).
- **Связанные Artifact** — единицы контента, порождённые в этом Run. *Необязателен* (до
  первого результата отсутствуют). *Изменять:* только через Run.
- **Связанные Evaluation** — вердикты QA/качества в этом Run. *Необязателен.* *Изменять:*
  только через Run.
- **Связанные Human Review** — решения человека по этому Run. *Необязателен.* *Изменять:*
  только через Run.
- **Связанные Analytics Record** — метрики вызовов этого Run. *Необязателен* (append-only).
  *Изменять:* только пополнение, через Run.

> Run **не дублирует** содержимое дочерних сущностей и не хранит чужих данных (Prompt,
> Workflow, модель и т. п.) — он держит **принадлежность и состояние**, а детали живут в
> своих сущностях (раздел 9).

## 4. Жизненный цикл

Семь состояний. `Completed` и `Failed` — **альтернативные терминальные** исходы (не
последовательные): успешное завершение **или** управляемый сбой.

```
        ┌─────────┐
        │ Created │  Run создан, вход зафиксирован, ещё не в очереди
        └────┬────┘
             ▼
        ┌─────────┐
        │ Queued  │  поставлен в очередь на исполнение
        └────┬────┘
             ▼
        ┌─────────┐ ◄───────────── возврат на доработку ─────────────┐
        │ Running │  исполняется конвейер (идут Task)                 │
        └────┬────┘                                                   │
             ▼                                                        │
      ┌──────────────┐  QA: риск/возврат                              │
      │  Waiting QA  │ ───────────────────────────────────────────────┤
      └──────┬───────┘                                                │
             │ QA: пройдено                                            │
             ▼                                                        │
     ┌─────────────────┐  Reject / Request changes                    │
     │  Waiting Human  │ ─────────────────────────────────────────────┘
     │ (Approval Gate) │
     └────────┬────────┘
              │ Approve
              ▼
        ┌───────────┐
        │ Completed │  терминал-успех: артефакт одобрен, публикация разрешена
        └───────────┘

   Из ЛЮБОГО нетерминального состояния при неустранимой ошибке /
   исчерпании повторов / фатальном сбое:
              ▼
         ┌────────┐
         │ Failed │  терминал-сбой, зафиксирован с причиной
         └────────┘
```

**Описание состояний:**

- **Created** — Run существует, вход (Content Brief + версия Workflow) зафиксирован;
  исполнение ещё не начато.
- **Queued** — ожидает исполнения; ресурсы пока не заняты.
- **Running** — выполняется конвейер: создаются и исполняются Task, формируются Output и
  Artifact.
- **Waiting QA** — **пауза**: результат передан на контроль качества/комплаенса (Evaluation).
  Run может находиться здесь сколь угодно долго.
- **Waiting Human** — **пауза** на Approval Gate: ожидается решение человека (Human Review).
- **Completed** — терминал-успех: получен одобренный человеком артефакт; публикация разрешена.
- **Failed** — терминал-сбой: зафиксированное наблюдаемое состояние с причиной.

**Допустимые переходы:**

- `Created → Queued`
- `Queued → Running`
- `Running → Waiting QA`
- `Waiting QA → Waiting Human` (QA пройдено)
- `Waiting QA → Running` (QA-риск/возврат — доработка)
- `Waiting Human → Completed` (Approve)
- `Waiting Human → Running` (Reject / Request changes — доработка)
- `{Created | Queued | Running | Waiting QA | Waiting Human} → Failed` (из любого
  нетерминального при неустранимой ошибке/исчерпании повторов)

**Запрещённые переходы (нарушение = ошибка предметной области):**

- Любой переход **из** `Completed` или `Failed` (терминалы окончательны; повторное открытие
  запрещено — нужен **новый** Run).
- `Waiting Human → Completed` **без** Approve (нет «авто-одобрения», истечения по таймауту и
  т. п.).
- `Running → Completed` и `Waiting QA → Completed` напрямую (нельзя миновать QA и/или
  Human Approval).
- `Running → Waiting Human` напрямую (Approval Gate достигается только после `Waiting QA`).
- Пропуск шагов очереди исполнения: `Created → Running` (минуя `Queued`), `Queued → Waiting
  QA` (минуя `Running`) и любые иные рёбра вне списка допустимых.
- Возврат в `Created` из любого состояния.
- Любая смена статуса **не** через Content Director.
- Любой переход при наличии у Run более чем одного «текущего» статуса.

## 4a. Восстановление Run (контракт восстановления — ADR-0015)

> Аддитивно к разделам 4–6 (ADR-0015). Не меняет `create`, таблицу переходов или сигнатуры Run.
> Механизм сохранения/восстановления и хранилище — **вне** этого раздела (см. раздел 9; это уровень
> реализации).

ADR-0003 вводит Run только в начальном состоянии (`create`) и продвигает его сторожевыми переходами
(раздел 4). Восстановимость исполнения (ADR-0015; инвариант раздела 5 п. 2) требует, чтобы Run мог
снова существовать — в новом процессе — в **сохранённом** (неначальном) состоянии. Раздел задаёт
**доменный контракт** такого появления.

**Восстановленный Run** — тот же агрегат: тот же `run_id`, тип и инварианты, что и Run, чья истина
была сохранена. Он приводится в существование непосредственно в сохранённом состоянии вместе со
своими дочерними сущностями и историей событий и далее продвигается **неизменным** контрактом
переходов раздела 4. Это не новая сущность и не новое состояние — тот же Run, появившийся иным
способом.

**Контракт допущения (verify/reject).** Восстановленный Run допускается **только** если сохранённое
состояние удовлетворяет всем инвариантам Run (раздел 5) — в частности: ровно один текущий статус;
неизменяемый вход на месте; целостность агрегата (все дети принадлежат этому `run_id`); статус
согласован с составом/исходами детей в той мере, в какой это задают инварианты. Состояние, не
удовлетворяющее инвариантам, **не является восстановимым Run**: оно **отвергается** (доменная ошибка),
а не устанавливается «как есть». Восстановление **не** обходит сторожей переходов: оно не *совершает*
переход в сохранённое состояние, а **переустанавливает** состояние, которого оригинал уже достиг
легально (раздел 5 п. 4 + ADR-0013); легальность **проверяется**, а не предполагается.

**После восстановления** Run неотличим от Run, чья истина была сохранена: те же допустимые/запрещённые
переходы (раздел 4), те же правила изменения (раздел 6), те же доменные ошибки. Терминальный Run
остаётся терминальным (возобновление терминального запрещено).

**Доменные ошибки (дополнительно к разделу 6):**

- Допуск восстановленного Run в состоянии, нарушающем инварианты раздела 5 (отвергается).
- Появление двух «живых» Run с одним `run_id` (нарушение идентичности/целостности).

## 5. Инварианты

> Правила, которые **никогда** не нарушаются. Нарушение — дефект, а не «крайний случай».

1. **Ровно один текущий статус.** В каждый момент Run находится строго в одном состоянии.
   *Почему:* состояние — основа маршрутизации и аудита; неоднозначность статуса делает
   систему неуправляемой.
2. **Неизменяемый вход.** Ссылка на Content Brief и ссылка на версию Workflow не меняются
   после создания. *Почему:* воспроизводимость — любой Run должен быть восстановим из
   зафиксированного входа (PROJECT.md §2).
3. **Единственный распорядитель статуса.** Статус Run меняет только Content Director.
   *Почему:* детерминированное управление потоком сосредоточено в оркестраторе; агенты не
   управляют конвейером (PROJECT.md §4 п. 2).
4. **Только разрешённые переходы.** Допустимы лишь рёбра из раздела 4. *Почему:* порядок
   производства (исполнение → QA → человек) защищает качество и «человека в контуре».
5. **Обязательная терминальность.** Run обязан достигать `Completed`/`Failed`. *Почему:*
   «вечный» Run — это утечка состояния и невозможность учёта; каждое производство должно
   иметь исход.
6. **Публикация только через Approve.** Переход в `Completed` (и, как следствие, разрешение
   публикации артефакта) возможен исключительно после явного Approve соответствующего
   Human Review. *Почему:* ключевой инвариант «человек в контуре» (PROJECT.md §12).
7. **Целостность агрегата.** Все дочерние сущности (Task, Output, Artifact, Evaluation,
   Human Review, Analytics Record) принадлежат ровно одному Run и создаются/изменяются только
   через него. *Почему:* Run — граница согласованности; «бесхозных» или сторонне-изменяемых
   детей быть не может.
8. **Ограниченность доработок.** Число итераций доработки и повторов ограничено политикой;
   бесконечные циклы запрещены. *Почему:* защита от «вечной» доработки и неконтролируемой
   стоимости (PROJECT.md §10).
9. **Прослеживаемость по умолчанию.** Каждый значимый шаг и метрика фиксируются в составе
   Run. *Почему:* без трассировки шаг считается несделанным (PROJECT.md §4 п. 5, §16).

## 6. Правила изменения

**Разрешено:**

- Content Director **запрашивает** допустимый переход статуса; Run его **принимает** (как
  корень-страж) и фиксирует историю.
- Пополнение агрегата **через Run**: создание Task по очередному шагу; фиксация Output;
  создание новой версии Artifact и смена его статуса; открытие Evaluation и Human Review;
  запись Analytics Record.
- Обновление **производных** атрибутов Run в рамках перехода: момент последнего перехода,
  счётчик итераций доработки (при возврате), причина (при `Failed`/Reject), момент терминала.

**Запрещено:**

- Менять идентификатор Run или его зафиксированный вход (Content Brief, версия Workflow).
- Менять статус не через Content Director или по неразрешённому ребру.
- Создавать/изменять дочерние сущности «снаружи», в обход Run.
- Завершать Run успехом без предшествующего Approve.
- Возобновлять терминальный Run.
- Допускать более одного текущего статуса или «застревание» без пути к терминалу.

**Ошибки предметной области (domain errors):**

- Запрос недопустимого перехода (например, `Running → Completed`).
- Попытка изменить терминальный Run.
- Создание дочерней сущности вне границы Run.
- Достижение `Completed` без Approve.
- Смена статуса актёром, отличным от Content Director.
- Нарушение «ровно один статус» или попытка изменить неизменяемый вход.

> Все перечисленные ошибки — **доменные**: они описывают нарушение бизнес-правил, а не
> технические сбои. Технические сбои (таймауты, недоступность) ведут к управляемому
> `Failed`/повтору, но не нарушают инварианты.

## 7. Domain Events

Run порождает следующие бизнес-события (только смысл «что произошло»; механика доставки,
шины и брокеры — вне домена, согласовано с `DOMAIN_MODEL.md` §10):

- **RunCreated** — когда Run создан по готовому Content Brief (`Created`). *Инициатор:*
  Content Director.
- **RunQueued** — когда Run поставлен в очередь (`Queued`). *Инициатор:* Content Director.
- **RunStarted** — когда Run начал исполнение (`Running`). *Инициатор:* Content Director.
- **RunCompleted** — когда Run достиг `Completed` (после Approve). *Инициатор:* Content
  Director. *Затрагивает:* Run и одобренный Artifact.
- **RunFailed** — когда Run достиг `Failed`. *Инициатор:* Content Director. *Несёт:* причину
  сбоя.

**Возврат на доработку.** Повторный вход в `Running` из `Waiting QA`/`Waiting Human`
происходит **вследствие** вышестоящих событий жизненного цикла (вердикт Evaluation с риском;
`HumanReviewRejected` / запрос изменений — см. `DOMAIN_MODEL.md` §10), а не как самостоятельное
«творческое» решение Run. Если для возврата потребуется отдельное именованное событие (напр.
`RunReturnedToRework`), оно вводится через `ADR` и канонизируется в `DOMAIN_MODEL.md` §10 —
здесь новой функциональности не добавляется.

## 8. Взаимодействие

Во всех случаях Run выступает **корнем агрегата**: дочерние сущности существуют внутри него и
изменяются **только через него**; смену состояния Run инициирует Content Director.

- **Task.** Run **владеет** задачами. По очередному Workflow Step через Run создаётся Task;
  исход Task (успех/ошибка) служит основанием для перехода статуса Run (`Running → Waiting QA`,
  `Running → Failed`). Run хранит принадлежность Task, но **не исполняет** их и **не выбирает**
  агентов.
- **Artifact.** Run **владеет** артефактами. Результаты шагов порождают версии Artifact внутри
  Run; именно артефакт Run выносится на QA и Human Review и может стать `Published` — но лишь
  после Approve. Каждый Artifact принадлежит ровно одному Run.
- **Evaluation.** Run **владеет** вердиктами. На `Waiting QA` открывается Evaluation; её
  вердикт (пройдено/риск) определяет переход Run (`Waiting QA → Waiting Human` либо возврат в
  `Running`). Run хранит вердикт, но **не оценивает** контент сам.
- **Human Review.** Run **владеет** решениями человека. На `Waiting Human` открывается Human
  Review; его исход (Approve/Reject/Changes) задаёт терминал или доработку. `Approve` —
  **единственный** путь к `Completed`.
- **Analytics Record.** Run **владеет** метриками. По каждому вызову в рамках Task через Run
  фиксируется запись метрик (append-only). Run хранит принадлежность, но **не анализирует** —
  выводы строит Analytics Agent.

## 9. Что НЕ входит в Run

Run хранит **принадлежность и состояние**, а не чужие данные и не поведение. В частности:

- **Run не хранит Prompt** — промпты принадлежат Agent (их версии); Run лишь опосредованно
  ссылается через Task/Agent.
- **Run не содержит Workflow** — он **ссылается** на конкретную версию Workflow; само
  определение и его шаги живут в Workflow.
- **Run не управляет Agent** и не выбирает модель — это конфигурация и логика Content Director.
- **Run не вызывает LLM** и вообще не выполняет инференс.
- **Run не знает про Notion** — домену принадлежит Content Brief, а не инструмент его
  постановки.
- **Run не знает про Google Docs** — домену принадлежит Human Review (решение/аудит), а не
  площадка ревью.
- **Run не знает про n8n / API / БД / адаптеры** — это инфраструктура (DOMAIN_MODEL §8).
- **Run не хранит Schema/Skill/Tool** — он ссылается на версии контрактов и использует
  результаты, но определения принадлежат своим сущностям.
- **Run не принимает содержательных решений** (что написать, как оценить) и **не маршрутизирует
  по существу** — он охраняет переходы, а стратегию задаёт Content Director над его состоянием.

## 10. Примеры

> Сценарии описаны как последовательности состояний и доменных фактов — без реализации.

### Корректные сценарии

**A. Прямой путь (happy path).**
`Created → Queued → Running → Waiting QA → (QA пройдено) → Waiting Human → (Approve) →
Completed`. Артефакт получает `Published`. *Почему корректно:* пройдены все обязательные шаги,
публикация — после Approve, достигнут терминал.

**B. Возврат после QA-риска.**
`… → Running → Waiting QA → (риск, fail closed) → Running → Waiting QA → (пройдено) →
Waiting Human → (Approve) → Completed`. *Почему корректно:* QA-риск останавливает выпуск и
возвращает на доработку (PROJECT.md §10); затем нормальный путь к одобрению. Счётчик итераций
увеличен.

**C. Возврат после Reject человеком, затем одобрение.**
`… → Waiting Human → (Reject, причина зафиксирована) → Running → Waiting QA → (пройдено) →
Waiting Human → (Approve) → Completed`. *Почему корректно:* Reject ведёт к доработке, а не к
публикации; финальное `Completed` — только после последующего Approve.

**D. Управляемый сбой.**
`Created → Queued → Running → (исчерпаны повторы Task) → Failed` (с причиной). *Почему
корректно:* при неустранимой ошибке Run достигает наблюдаемого терминала `Failed`, инварианты
не нарушены.

### Некорректные сценарии

**1. `Running → Completed` напрямую.** Минует QA и Human Approval. *Почему ошибка:* нарушает
порядок производства и инвариант «публикация только через Approve».

**2. `Waiting Human → Completed` без Approve** (например, авто-одобрение по таймауту).
*Почему ошибка:* прямое нарушение «человека в контуре» (PROJECT.md §12).

**3. `Completed → Running` (повторное открытие терминала).** *Почему ошибка:* терминалы
окончательны; для новой работы создаётся **новый** Run.

**4. Смена ссылки на Content Brief в середине Run.** *Почему ошибка:* нарушает неизменяемость
входа и воспроизводимость.

**5. Два «текущих» статуса одновременно** (например, «Running и Waiting QA»). *Почему ошибка:*
нарушает инвариант «ровно один статус».

**6. Создание Task в обход Run** (напрямую, без корня). *Почему ошибка:* нарушает границу
агрегата и целостность данных.

**7. Статус Run меняет Agent, а не Content Director.** *Почему ошибка:* нарушает инвариант
единственного распорядителя статуса; агент не управляет конвейером.
