# RUN_ACCEPTANCE.md — Критерии приёмки реализации **Run**

> **Место в иерархии документации (PROJECT.md, раздел 17).**
> `PROJECT.md` → `ARCHITECTURE.md` → `ROADMAP.md` → `DOMAIN_MODEL.md` → `RUN_SPEC.md` →
> **`RUN_ACCEPTANCE.md`** → `ADR` → `Implementation (Code)`.
>
> Документ отвечает на вопрос: **«Как мы поймём, что реализация Run полностью соответствует
> `RUN_SPEC.md`?»** Это набор **проверяемых сценариев приёмки** на языке предметной области.
> Он не содержит кода, pytest и технических деталей и становится **прямой основой для
> `test_run.py`** (Этап 2 ROADMAP): один сценарий → один тест (или группа тестов).
>
> **Источник истины:** `RUN_SPEC.md` (по сущности Run) → `DOMAIN_MODEL.md` → `PROJECT.md`.

## 0. Соглашения

- Формат сценария — **Дано / Когда / Тогда** (Given / When / Then) на языке домена.
- Каждый сценарий имеет **уникальный идентификатор** (например, `HP-01`) для прямого
  отображения в будущий тест.
- Актёр, инициирующий смену статуса, — **Content Director** (если не указано иное). Изменения
  дочерних сущностей идут **только через Run** (корень агрегата).
- «Отвергается как ошибка предметной области» означает: операция **не выполняется**, состояние
  Run **не изменяется**, фиксируется доменная ошибка (не технический сбой).
- Идентификаторы блоков: `AC` — критерии; `HP` — happy path; `RW` — доработка; `FL` —
  ошибки/сбои; `INV` — инварианты; `AGG` — границы агрегата; `ID` — идентичность; `EV` —
  события; `REG` — регресс.

## 1. Acceptance Criteria

Проверяемые критерии, выведенные из требований `RUN_SPEC.md`. Каждый критерий проверяется
одним или несколькими сценариями разделов 2–8 (ссылки указаны).

- **AC-01. Ровно один статус.** В любой момент у Run строго одно текущее состояние. ⟶ `INV-01`
- **AC-02. Неизменяемый вход — Content Brief.** Ссылку на Content Brief нельзя изменить после
  создания. ⟶ `FL-05`, `INV-02`, `ID-02`
- **AC-03. Неизменяемый вход — версия Workflow.** Ссылку на версию Workflow нельзя изменить
  после создания. ⟶ `FL-06`, `INV-02`
- **AC-04. Единственный распорядитель статуса.** Статус меняет только Content Director. ⟶
  `FL-08`, `INV-03`
- **AC-05. Только разрешённые переходы.** Допустимы лишь рёбра из `RUN_SPEC.md` §4; остальные
  отвергаются. ⟶ `FL-01`, `FL-09`, `FL-10`, `INV-04`
- **AC-06. Терминальность.** Run обязан иметь путь к `Completed`/`Failed`; терминал
  окончателен. ⟶ `FL-03`, `FL-04`, `INV-05`
- **AC-07. Публикация только через Approve.** `Completed` (и `Published` артефакта) достижимо
  лишь после явного Approve. ⟶ `FL-02`, `INV-06`, `EV-04`
- **AC-08. Целостность агрегата.** Дочерние сущности принадлежат ровно одному Run и
  создаются/меняются только через него. ⟶ `AGG-01..06`, `INV-07`
- **AC-09. Ограниченность доработок.** Число итераций доработки/повторов ограничено; при
  исчерпании — `Failed`. ⟶ `RW-04`, `INV-08`
- **AC-10. Прослеживаемость по умолчанию.** Каждый значимый шаг и метрика фиксируются в составе
  Run. ⟶ `INV-09`, `AGG-05`
- **AC-11. Стабильная идентичность.** Идентификатор Run неизменен; смена атрибутов/статуса не
  создаёт новую Entity. ⟶ `ID-01`, `ID-03`
- **AC-12. События отражают факты.** Каждое доменное событие Run возникает строго в свой момент
  и не возникает в иных. ⟶ `EV-01..05`

*Критериев приёмки: **12** (AC-01 … AC-12).*

## 2. Happy Path Scenarios

**HP-01. Прямой успешный путь.**
- *Дано:* существует готовый Content Brief (`Ready`) и активная версия Workflow.
- *Когда:* Run создаётся и проходит `Created → Queued → Running → Waiting QA → (QA пройдено)
  → Waiting Human → (Approve) → Completed`.
- *Тогда:* Run в `Completed`; одобренный Artifact получает `Published`; зафиксирован момент
  терминала; порождены события `RunCreated`, `RunQueued`, `RunStarted`, `RunCompleted`.
- *Ожидаемый результат:* полный успешный прогон без доработок и ошибок.

**HP-02. `Completed` достигается только после Approve.**
- *Дано:* Run в `Waiting Human` с артефактом-кандидатом.
- *Когда:* человек выносит **Approve**.
- *Тогда:* Run переходит в `Completed`, артефакт становится `Published`, возникает
  `RunCompleted`.
- *Ожидаемый результат:* успешное завершение наступает исключительно вследствие Approve.

*Сценариев happy path: **2**.*

## 3. Rework Scenarios

**RW-01. Возврат после QA-риска.**
- *Дано:* Run в `Waiting QA`.
- *Когда:* QA выносит вердикт «риск» (fail closed).
- *Тогда:* Run возвращается в `Running` (доработка), счётчик итераций увеличивается; затем
  `Waiting QA → (пройдено) → Waiting Human → (Approve) → Completed`.
- *Ожидаемый результат:* риск останавливает выпуск и инициирует доработку, а не публикацию.

**RW-02. Возврат после Reject человеком.**
- *Дано:* Run в `Waiting Human`.
- *Когда:* человек выносит **Reject** с причиной.
- *Тогда:* причина фиксируется; Run возвращается в `Running`; далее нормальный путь до
  `Approve → Completed`.
- *Ожидаемый результат:* Reject ведёт к доработке, а не к завершению.

**RW-03. Возврат после Request changes.**
- *Дано:* Run в `Waiting Human`.
- *Когда:* человек выносит **Request changes** с указаниями.
- *Тогда:* указания фиксируются как контекст доработки; Run возвращается в `Running`; далее до
  `Approve → Completed`.
- *Ожидаемый результат:* запрос изменений возвращает на доработку с сохранением указаний.

**RW-04. Исчерпание лимита доработок/повторов.**
- *Дано:* Run многократно возвращался на доработку и достиг предела политики.
- *Когда:* предел итераций/повторов исчерпан.
- *Тогда:* Run переходит в `Failed` с причиной; дальнейшие доработки невозможны.
- *Ожидаемый результат:* бесконечные циклы исключены; наступает управляемый терминал.

*Сценариев доработки: **4**.*

## 4. Failure Scenarios (ошибки предметной области)

**FL-01. Запрещённый переход `Running → Completed`.**
- *Когда:* запрашивается прямой переход из `Running` в `Completed`.
- *Тогда:* отвергается как доменная ошибка; статус не меняется. *Почему:* минует QA и Approval.

**FL-02. `Waiting Human → Completed` без Approve.**
- *Когда:* делается попытка завершить Run без явного Approve (напр., «по таймауту»).
- *Тогда:* отвергается; Run остаётся в `Waiting Human`. *Почему:* нарушает «человека в контуре».

**FL-03. Повторное открытие `Completed`.**
- *Дано:* Run в `Completed`.
- *Когда:* запрашивается любой переход из `Completed`.
- *Тогда:* отвергается. *Почему:* терминал окончателен; нужна новая Run.

**FL-04. Повторное открытие `Failed`.**
- *Дано:* Run в `Failed`.
- *Когда:* запрашивается любой переход из `Failed`.
- *Тогда:* отвергается. *Почему:* терминал окончателен.

**FL-05. Изменение ссылки на Content Brief.**
- *Когда:* делается попытка сменить Content Brief действующего Run.
- *Тогда:* отвергается. *Почему:* вход неизменяем (воспроизводимость).

**FL-06. Изменение ссылки на версию Workflow.**
- *Когда:* делается попытка сменить версию Workflow действующего Run.
- *Тогда:* отвергается. *Почему:* вход неизменяем; смена пути = новая Run.

**FL-07. Изменение идентификатора Run.**
- *Когда:* делается попытка изменить идентификатор.
- *Тогда:* отвергается. *Почему:* идентичность неизменна.

**FL-08. Смена статуса не Content Director.**
- *Когда:* статус пытается сменить иной актёр (например, Agent).
- *Тогда:* отвергается. *Почему:* единственный распорядитель статуса — Content Director.

**FL-09. Пропуск состояния (`Created → Running`).**
- *Когда:* запрашивается переход, минующий `Queued`.
- *Тогда:* отвергается. *Почему:* допустимы только рёбра из §4.

**FL-10. Пропуск QA (`Running → Waiting Human`).**
- *Когда:* запрашивается прямой переход в `Waiting Human` минуя `Waiting QA`.
- *Тогда:* отвергается. *Почему:* Approval Gate достигается только после QA.

**FL-11. Более одного текущего статуса.**
- *Когда:* возникает попытка одновременно держать два статуса.
- *Тогда:* невозможно/отвергается. *Почему:* инвариант «ровно один статус».

**FL-12. Технический сбой → управляемый `Failed`.**
- *Дано:* Run в `Running`, у задач исчерпаны повторы из-за технической ошибки.
- *Когда:* неустранимый сбой/исчерпание повторов.
- *Тогда:* Run переходит в `Failed` с причиной (наблюдаемо), инварианты не нарушены.
- *Примечание:* это **управляемое** поведение, а не нарушение домена.

*Сценариев ошибок/сбоев: **12**.*

## 5. Invariant Tests

По одному сценарию проверки на каждый инвариант `RUN_SPEC.md` §5.

**INV-01. Ровно один статус.** В любой достижимой точке у Run строго одно состояние.
**INV-02. Неизменяемый вход.** Content Brief и версия Workflow не меняются после создания.
**INV-03. Единственный распорядитель статуса.** Любая смена статуса возможна только Content
Director.
**INV-04. Только разрешённые переходы.** Полный перебор показывает: допустимы лишь рёбра из §4,
прочие отвергаются.
**INV-05. Терминальность.** Из любого нетерминального состояния существует путь к
`Completed`/`Failed`; терминал не имеет исходящих переходов.
**INV-06. Публикация только через Approve.** Ни один путь, кроме Approve, не приводит к
`Completed`/`Published`.
**INV-07. Целостность агрегата.** Любая дочерняя сущность принадлежит ровно одному Run и
изменяется только через него.
**INV-08. Ограниченность доработок.** Существует предел итераций/повторов; его достижение ведёт
к `Failed`.
**INV-09. Прослеживаемость.** Каждый значимый шаг и каждый вызов сопровождается записью
(Task/Analytics Record) в составе Run.

*Сценариев инвариантов: **9** (по числу инвариантов).*

## 6. Aggregate Boundary Tests

**AGG-01. Task вне Run не существует.** Task всегда принадлежит ровно одному Run; создать Task
без Run невозможно.
**AGG-02. Artifact вне Run не существует.** Каждый Artifact принадлежит ровно одному Run.
**AGG-03. Evaluation вне Run не существует.** Вердикт всегда привязан к одному Run.
**AGG-04. Human Review вне Run не существует.** Решение всегда привязано к одному Run.
**AGG-05. Analytics Record вне Run не существует.** Метрика всегда привязана к одному
Run/Task.
**AGG-06. Изменение детей только через корень.** Любое создание/изменение дочерней сущности
выполняется только через Run; «обходные» изменения отвергаются.

*Сценариев границ агрегата: **6**.*

---

## 7. Identity Tests

**ID-01. Неизменяемость идентификатора.** Идентификатор Run остаётся прежним на всём
жизненном цикле (через все переходы состояний).
**ID-02. Неизменяемость входа.** Зафиксированные Content Brief и версия Workflow не меняются
(подтверждает AC-02/AC-03 с точки зрения идентичности).
**ID-03. Смена состояния не создаёт новую Entity.** Run в `Running` и тот же Run в `Completed`
— одна и та же сущность с тем же идентификатором.
**ID-04. Новая Entity — только новый бизнес-объект.** Новый бриф или повторный прогон порождает
**новый** Run, а не мутацию существующего.

*Сценариев идентичности: **4**.*

---

## 8. Event Tests

Для каждого события Run: **когда возникает** и **когда возникать не должен**.

**EV-01. RunCreated.**
- *Возникает:* в момент создания Run по готовому брифу (`Created`).
- *Не возникает:* при последующих переходах или повторно для того же Run.

**EV-02. RunQueued.**
- *Возникает:* при переходе `Created → Queued`.
- *Не возникает:* при иных переходах; повторно при возврате на доработку.

**EV-03. RunStarted.**
- *Возникает:* при первом входе в исполнение `Queued → Running`.
- *Не возникает:* при повторном входе в `Running` по доработке (возврат вызывается
  вышестоящими событиями — вердиктом Evaluation / `HumanReviewRejected`, см. `RUN_SPEC.md` §7),
  и не возникает в терминальных состояниях.

**EV-04. RunCompleted.**
- *Возникает:* при переходе в `Completed` (после Approve).
- *Не возникает:* без Approve; никогда — при `Failed`.

**EV-05. RunFailed.**
- *Возникает:* при переходе в `Failed`; несёт причину.
- *Не возникает:* при `Completed`; не возникает, если Run ещё нетерминален.

*Сценариев событий: **5**.*

## 9. Regression Tests

Минимальный набор, который обязан проходить **после любого изменения** Run (ядро поведения).
Новых сценариев не вводит — ссылается на существующие:

- **Поток:** `HP-01` (прямой успех), `RW-01` (возврат после QA), `RW-02` (возврат после
  Reject).
- **Запреты:** `FL-01` (запрещённый переход), `FL-02` (завершение без Approve), `FL-03`
  (повторное открытие терминала), `FL-05` (изменение входа), `FL-08` (смена статуса чужим
  актёром).
- **Границы и идентичность:** `AGG-01` (Task вне Run), `ID-01` (неизменность идентификатора).
- **События:** `EV-04` (RunCompleted только после Approve).

*Регрессионный набор: **11** сценариев (подмножество, без новых ID).*

### 3. Требования RUN_SPEC.md, которые невозможно проверить напрямую

Честно отмечаем ограничения **сценарной (примерной)** приёмки:

- **Терминальность как liveness-свойство (§5, инв. 5).** «Run *в итоге* достигает терминала»
  — свойство живучести; примерным сценарием доказуемо лишь **структурно** (наличие пути к
  терминалу из каждого состояния + ограниченность доработок `RW-04`/`INV-08`), но не как общий
  «вечный» прогон. Проверяем достижимость и ограниченность, а не абстрактную бесконечность.
- **§9 «Что НЕ входит в Run» (отсутствие связей с Prompt/Notion/Google Docs/LLM/Workflow).**
  Это утверждения об **отсутствии** ответственности. Они проверяются **косвенно**: у Run нет
  таких данных/операций в контракте, и попытки их «вменить» отсутствуют как сценарии. Прямой
  позитивный тест на «отсутствие» сформулировать нельзя — это свойство дизайна модели.
- **Механика доставки событий (шины/брокеры).** Сознательно **вне домена** (`RUN_SPEC.md` §7).
  Проверяется только **семантика возникновения** события (`EV-*`), но не его транспорт.

Все прочие требования (§1–§4, §6, §8, §10 и большинство §5) — **полностью проверяемы**
сценарно.

### 4. Подтверждение готовности к pytest

Подтверждаю: после этого документа **можно переходить к написанию pytest-тестов** (`test_run.py`,
Этап 2 ROADMAP). Каждый сценарий сформулирован в виде проверяемого «Дано / Когда / Тогда» с
уникальным идентификатором, что даёт прямое отображение «сценарий → тест». Покрыты все
нормативные требования `RUN_SPEC.md`; ограничения проверяемости явно зафиксированы (пункт 3),
чтобы они не превратились в ложные ожидания к тестам.

> При расхождении с `RUN_SPEC.md` исправляется **этот** документ: источник истины —
> `RUN_SPEC.md` → `DOMAIN_MODEL.md` → `PROJECT.md` (раздел 17).
