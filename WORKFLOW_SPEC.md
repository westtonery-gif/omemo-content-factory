## 1. Назначение

**Workflow** — **декларация** конвейера для типа контента: какие шаги, в каком порядке
(`DOMAIN_MODEL.md` §2.3; `PROJECT.md` §18: «Workflow ≠ Orchestrator»). **Workflow Step** — один
шаг внутри Workflow: привязка «позиция в конвейере → роль (Agent) + контракт» (`DOMAIN_MODEL.md`
§2.4).

**Природа: чистый декларативный план (v1 static model).** Workflow **описывает** порядок работы и
**ничего не исполняет** (`ADR-0009` §1, §11). Это **данные**, а не движок: он не знает Run, Task и
Output, не содержит логики исполнения, не имеет runtime-состояния.

**Почему Workflow — самостоятельный корень, владеющий шагами.**

- Workflow Step **не имеет идентичности и существования вне** своей Workflow; целостность
  упорядоченного конвейера держит корень (`DOMAIN_MODEL.md` §9.2; `ADR-0009` §2, §3).
- Workflow — сущность **каталога определений**, в одном ряду с Schema/Agent, а не часть оси
  исполнения (Run).

**Почему это важно.** Workflow — это переход от императивной к **декларативной** постановке
конвейера: один и тот же execution core разворачивает разные планы, не меняясь.

## 2. Ответственность

**Workflow / Workflow Step обязаны:**

- Нести **зафиксированное декларативное описание** конвейера: упорядоченный набор шагов и их
  привязки (роль, контракт), как **неизменяемые данные** (раздел 3).
- Гарантировать **целостность данных** на момент создания: непустой набор шагов, уникальные
  идентификаторы шагов, ссылочную корректность `depends_on` (раздел 5).
- Сохранять **строгий порядок шагов** как единственную модель порядка (раздел 4).

**Workflow / Workflow Step никогда не должны:**

- **Исполнять** что-либо, влиять на runtime напрямую или менять состояние Run/Task/Output/Schema.
- **Содержать execution-семантику**: планирование, маршрутизацию по условию, реактивность, DAG.
- **Использовать `depends_on`** для порядка/планирования/оптимизации (раздел 4).
- **Знать** про Run, Task, Output, Content Director или инфраструктуру.
- **Изменяться после создания**: новое расположение шагов — это **новое значение** Workflow.

> Кратко: **Workflow — это неизменяемое декларативное описание плана, а не исполнитель, не граф и
> не движок.**

## 3. Внутренние данные (бизнес-атрибуты)

> Бизнес-свойства, без технических типов. Для каждого: *зачем · обязателен ли · кто вправе
> изменять*. Источник: `DOMAIN_MODEL.md` §2.3, §2.4; `ADR-0009` §4.

**Workflow (неизменяем после создания):**

- **Идентификатор Workflow** — стабильная идентичность плана. *Обязателен.* *Изменять: никто.*
- **Имя (name)** — человекочитаемое название плана. *Обязателен.* *Изменять: никто.*
- **Шаги (steps)** — **упорядоченный непустой** набор Workflow Step. *Обязателен.* *Изменять:
  никто* (новое расположение = новый Workflow).

**Workflow Step (неизменяем; существует только в составе Workflow):**

- **Идентификатор шага (step_id)** — **уникален в пределах** своей Workflow. *Обязателен.*
  *Изменять: никто.*
- **Тип задачи (task_type)** — род работы шага. *Обязателен.* *Изменять: никто.*
- **Ссылка на роль (agent_ref)** — **ровно одна** роль (Agent), исполняющая шаг
  (`DOMAIN_MODEL.md` §6); непрозрачная ссылка. *Обязателен.* *Изменять: никто.*
- **Ссылка на контракт (schema_ref)** — **декларативный planning-hint** на выходной контракт шага
  (версия Schema; непрозрачная ссылка). Только planning / consistency-check; в **runtime не участвует
  и на результат не влияет**. Authority над контрактом — **Schema layer** (`ADR-0008`); runtime-ссылку
  на применяемую Schema несёт `Prompt.schema_ref` (passive artifact, `ADR-0011`; AGENT_SPEC §4).
  *Обязателен.* *Изменять: никто.*
- **Зависимости (depends_on)** — перечень `step_id`, на которые ссылается шаг; **только
  описательная метаданная** (раздел 4). *Необязателен* (по умолчанию пуст). *Изменять: никто.*

> Workflow Step **не хранит** runtime-данные (вход исполнения, статус, попытки, результат) — это
> принадлежит Task на оси исполнения и вне данной сущности.

## 4. Модель порядка исполнения

**Порядок исполнения строго определяется порядком элементов в `steps`** (`ADR-0009` §6).

- `depends_on` — **только метаданная** и **НЕ влияет** на порядок исполнения, планирование или
  поток управления. В частности, `depends_on` **не используется**:
  - как **DAG**;
  - для **планирования**;
  - для **оптимизации/переупорядочивания** исполнения.
- Разворачивание плана выполняет **Content Director как детерминированный планировщик**, который
  отображает `steps → последовательность Task` в **строгом порядке списка** и **не** интерпретирует
  `depends_on`, **не** строит граф, **не** оптимизирует порядок (раздел 8; `ADR-0009` §8).
- Зависимости-через-`depends_on`, параллельные ветви, условная маршрутизация и обнаружение циклов
  — **вне** этой модели (раздел 9).

---

## 5. Инварианты

> Правила, которые **никогда** не нарушаются. Нарушение — дефект. Источник: `DOMAIN_MODEL.md` §6;
> `ADR-0009` §2, §3, §5.

1. **Неизменяемость.** Workflow и Workflow Step неизменяемы после создания; изменение плана = **новое
   значение** Workflow. *(Сильнее, чем «принятая версия неизменяема», `DOMAIN_MODEL.md` §6.)*
2. **Непустой план.** Workflow содержит **хотя бы один** шаг (`DOMAIN_MODEL.md` §6).
3. **Уникальность шагов.** `step_id` уникальны в пределах одной Workflow.
4. **Ссылочная корректность `depends_on`.** Каждый идентификатор в `depends_on` ссылается на
   существующий `step_id` той же Workflow. *(Только проверка данных; без анализа порядка/циклов.)*
5. **Ровно одна роль на шаг.** Каждый Workflow Step привязан **ровно к одному** Agent
   (`DOMAIN_MODEL.md` §6).
6. **Целостность агрегата.** Workflow Step существует **только** в составе своей Workflow и не имеет
   идентичности вне неё (`DOMAIN_MODEL.md` §9.2).
7. **Строгий порядок.** Единственная модель порядка — порядок `steps`; `depends_on` на порядок не
   влияет (раздел 4).
8. **Отсутствие runtime-состояния.** Workflow не несёт состояния исполнения и не участвует в
   жизненном цикле Run (раздел 9).

## 6. Правила изменения

**Разрешено:**

- Создать **новую** Workflow (с непустым, валидным набором шагов).
- Сформировать **новое значение** Workflow для иного плана (иные шаги/порядок).

**Запрещено:**

- Менять идентификатор, имя или шаги существующей Workflow; менять любой атрибут Workflow Step.
- Использовать `depends_on` для порядка/планирования/оптимизации.
- Наделять Workflow/Workflow Step поведением исполнения.

**Ошибки предметной области (domain errors):**

- Создание Workflow **без шагов** (пустой план).
- **Неуникальные** `step_id` в пределах Workflow.
- `depends_on`, ссылающийся на **несуществующий** `step_id`.

> Все перечисленные ошибки — **доменные**: нарушение правил данных при создании плана, а не
> технический сбой и не runtime-исключение.

---

## 7. Domain Events

Workflow и Workflow Step **не порождают доменных событий**: `DOMAIN_MODEL.md` §10 не определяет
событий для них, и Workflow — корень **каталога**, а не дочерняя сущность Run (журнала событий не
имеет; в журнал Run ничего не пишет) — `ADR-0009` §7.

> Если для жизненного цикла Workflow (ввод/вывод версии) понадобится событие, оно вводится через
> `ADR` и канонизируется в `DOMAIN_MODEL.md` §10 — здесь новой функциональности не добавляется.

## 8. Взаимодействие

Workflow — **декларация**, разворачиваемая в исполнение **только на уровне Content Director**;
сама она исполнения не касается (`ADR-0009` §8, §11).

- **Content Director.** **Детерминированный планировщик**: отображает `Workflow.steps →
  последовательность Task` в **строгом порядке списка**, вызывая **существующий** execution
  pipeline без изменений. Не интерпретирует `depends_on`, не строит граф, не оптимизирует порядок
  — только разворачивает линейный план. Content Director остаётся **единственной точкой
  исполнения**.
- **Workflow Step ↔ Agent.** Шаг **ссылается** на ровно одну роль (`agent_ref`), но не владеет ею
  и не управляет.
- **Workflow Step ↔ Schema.** Шаг **ссылается** на выходной контракт (`schema_ref`, версия
  Schema); Schema остаётся самостоятельным корнем (`SCHEMA_SPEC.md`), Workflow его не меняет.
- **Run / Task / Output.** Workflow их **не знает** и **не меняет**; они появляются на оси
  исполнения, которую ведёт Content Director, а не Workflow.

---

## 9. Что НЕ входит в Workflow / Workflow Step

Workflow хранит **неизменяемое декларативное описание плана**, а не поведение и не runtime. В
частности:

- **Не исполняет** и **не влияет на runtime напрямую** — разворачивается в Task только Content
  Director'ом (`ADR-0009` §11).
- **Не является engine / DAG / scheduler** — **none of**: DAG engine, scheduler, reactive execution
  graph, dependency-resolution engine, workflow runtime system (`ADR-0009` Non-goals).
- **Нет runtime-состояния, жизненного цикла, версионирования, планирования и execution-семантики**
  (v1 static model; `ADR-0009` §7). Отложены: `Draft/Active/Deprecated`, `version`, ссылка на
  Content Type, Approval Gates, терминальные условия, **входной** контракт шага и признак
  контрольной точки (`DOMAIN_MODEL.md` §2.3/§2.4 — документированная лексика, не моделируется).
- **Не управляет** Run/Task/Output и не меняет их статусы — это логика Content Director над
  состоянием Run.
- **Не знает** про Notion / Google Docs / n8n / LLM / адаптеры (`DOMAIN_MODEL.md` §8).

## 10. Примеры

> Сценарии описаны как факты предметной области (структура данных, порядок, ошибки) — без
> реализации и без описания потоков исполнения.

### Корректные сценарии

**A. Линейный план.**
Workflow с шагами `[research, write, edit]` (каждый с `agent_ref` и `schema_ref`). *Почему
корректно:* непустой набор, уникальные `step_id`. **Порядок исполнения = `research → write →
edit`** (порядок списка).

**B. `depends_on` присутствует, но инертен.**
Шаг `edit` имеет `depends_on = [write]`. Порядок исполнения **всё равно** определяется списком
(`research → write → edit`); `depends_on` — только метаданная. *Почему корректно:* `depends_on`
ссылается на существующий шаг и на порядок не влияет (раздел 4).

**C. Новый план — новое значение.**
Иной порядок/набор шагов оформляется как **новая** Workflow, а не правка существующей. *Почему
корректно:* Workflow неизменяема (инвариант 1).

### Некорректные сценарии

**1. Пустой план.** Workflow без шагов. *Почему ошибка:* нарушает «непустой план» (инвариант 2).

**2. Дублирующиеся `step_id`.** Два шага с одинаковым `step_id`. *Почему ошибка:* нарушает
уникальность (инвариант 3).

**3. Висячая зависимость.** `depends_on` ссылается на `step_id`, которого нет в Workflow. *Почему
ошибка:* нарушает ссылочную корректность (инвариант 4).

**4. Использование `depends_on` для переупорядочивания исполнения.** *Почему ошибка:* единственная
модель порядка — список `steps`; `depends_on` на порядок не влияет (раздел 4; инвариант 7).

**5. Изменение шагов существующей Workflow.** *Почему ошибка:* нарушает неизменяемость
(инвариант 1); изменение плана = новое значение Workflow.
