## 1. System Overview

OMEMO Content Factory — промышленная **фабрика контента** с человеком в контуре. Система
принимает производственное задание (бриф) из рабочего пространства редакции, проводит его через
детерминированно управляемый конвейер специализированных AI-агентов, обеспечивает контроль
качества и **обязательное одобрение человеком**, после чего публикует одобренный артефакт и
накапливает статистику для непрерывного улучшения.

Система делится на три макрослоя:

1. **Внешний слой (Edge).** Системы, с которыми работают люди и которые служат источником
   заданий и местом ревью: **Notion** (постановка задач, бриф, статусы), **n8n** (триггеры и
   связующая автоматизация), **Google Docs** (площадка для ревью и одобрения человеком).
2. **Ядро производства (Core).** **Python Agent Service** — приложение, внутри которого живут
   **Content Director** (оркестратор), **Agents**, **Skills**, **Tools** и слой **Adapters**.
   Здесь происходит фактическое производство контента и управление прогоном (`Run`).
3. **Слой данных и наблюдаемости (Data & Observability).** Хранилище состояния прогонов и
   артефактов, а также **Analytics** — сбор метрик стоимости/производительности/качества и
   аналитический агент для оптимизации.

**Главные компоненты и их ответственность (одной строкой):**

| Компонент | Ответственность |
|---|---|
| **Notion** | Источник брифов, метаданных задания и статусов; «фронт» редакции |
| **n8n** | Событийная автоматизация и мост между Notion/Google Docs и ядром |
| **Python Agent Service** | Среда исполнения ядра: оркестратор, агенты, skills, tools, адаптеры |
| **Content Director (Orchestrator)** | Детерминированное управление `Run`: запуск Workflow, состояние, маршрутизация, повторы, Human Approval, завершение |
| **Agents** | Узкие роли производства (планирование, написание, проверка фактов, редактура и т. д.) |
| **Skills** | Переиспользуемые модули одной задачи — строительные блоки агентов |
| **Tools** | Возможности, которые агент вызывает в ходе рассуждения для действия/получения данных |
| **Adapters** | Изоляция от внешних поставщиков (LLM, Notion, Google Docs, хранилище, аналитика) |
| **QA** | Контроль качества и соответствия (включая домен здоровья) перед выходом к человеку |
| **Analytics** | Сбор метрик и аналитический агент для оптимизации стоимости/качества |
| **Google Docs** | Площадка ревью и фиксации решения человека (Approve/Reject/Changes) |

> **Инвариант (PROJECT.md, разделы 1, 12).** Никакой артефакт не покидает систему и не
> считается готовым без явного **одобрения человека**. Это свойство архитектуры, а не настройка.

## 2. High-Level Architecture

Поток производства — от постановки задачи в Notion до накопления аналитики. Внешние системы
(Notion, n8n, Google Docs) взаимодействуют с ядром **только через адаптеры** (раздел 9).

```
                          ┌──────────────────────────────────────────────┐
                          │                  EDGE (люди)                  │
                          │   Notion  ·  n8n  ·  Google Docs              │
                          └──────────────────────────────────────────────┘

   Notion                 Постановка задачи, бриф, метаданные
      │
      ▼
   n8n                    Триггер: «бриф готов к производству» → вызов ядра
      │
      ▼
   Python Agent Service   Точка входа ядра: принимает задание, создаёт Run
      │
      ▼
   Content Director       Запускает Workflow, управляет состоянием Run
      │
      ▼
   Agents                 Конвейер ролей: Researcher → Writer → … (по Workflow)
      │
      ▼
   QA                     Контроль качества/комплаенса (fail closed)
      │
      ▼
   Google Docs            Публикация черновика-кандидата на ревью
      │
      ▼
   Human Approval         Approve / Reject / Request changes
      │
      ▼
   Analytics              Сбор метрик прогона; основа для оптимизации
```

**Поток управления (кто чем управляет):**

```
        ┌───────────────────────────── Content Director ─────────────────────────────┐
        │  (детерминированный оркестратор — единственный владелец потока и состояния)  │
        └───────────────────────────────────────────────────────────────────────────-┘
            │ запускает          │ маршрутизирует      │ объявляет паузу      │ завершает
            ▼                    ▼                     ▼                      ▼
        Workflow             Agent → Agent          QA / Human            Run → Completed
        (декларация)         (по контрактам)        (Approval Gates)        / Failed
```

**Ключевые наблюдения по схеме:**

- Стрелки «вниз» — это **поток данных**; решает «что дальше» исключительно **Content Director**,
  а не сами агенты (PROJECT.md, раздел 4, п. 2).
- Между агентами передаются **только Structured Outputs** (раздел 11). Свободный текст между
  агентами запрещён.
- Внешние системы отделены от ядра **адаптерами**, поэтому Notion/Google Docs/n8n заменяемы без
  изменения ядра.

## 3. Core Components

Для каждого компонента: **Ответственность · Вход · Выход · Зависимости**.

### 3.1 Notion
- **Ответственность:** «фронт» редакции — хранит бриф, тему, тип контента, требования,
  редакционные параметры и статус задания. Точка, где человек *ставит* задачу и *видит* итог.
- **Вход:** действия редакторов (создание/изменение задания), обратные статусы из ядра.
- **Выход:** структурированный бриф и метаданные задания; сигнал готовности к производству.
- **Зависимости:** n8n (триггеры/синхронизация), Notion Adapter со стороны ядра.

### 3.2 n8n
- **Ответственность:** событийная автоматизация и интеграционный «клей»: ловит событие из Notion
  («бриф готов»), вызывает ядро, переносит статусы и ссылки между Notion, ядром и Google Docs.
- **Вход:** события Notion/Google Docs; вызовы/уведомления от ядра.
- **Выход:** инициирующий вызов в Python Agent Service; обновления статусов во внешних системах.
- **Зависимости:** Notion, Google Docs, точка входа Python Agent Service.
- **Принцип:** n8n — **транспорт и триггеры**, а **не место бизнес-логики производства**. Логика
  конвейера живёт только в Content Director.

### 3.3 Python Agent Service
- **Ответственность:** среда исполнения ядра. Принимает задание, создаёт и ведёт `Run`, содержит
  Content Director, агентов, skills, tools и адаптеры; предоставляет точки входа для запуска
  прогонов и для приёма решений Human Approval.
- **Вход:** запрос на производство (из n8n), решения человека (через адаптеры), конфигурация.
- **Выход:** артефакты, обновления состояния `Run`, события для внешних систем, метрики.
- **Зависимости:** все внутренние слои; внешние системы — только через адаптеры.

### 3.4 Content Director (Orchestrator)
- **Ответственность:** детерминированное управление прогоном (полностью раскрыто в разделе 4).
- **Вход:** созданный `Run`, выбранный `Workflow`, структурированные результаты агентов и QA,
  решения человека.
- **Выход:** переходы состояния `Run`, вызовы агентов в заданном порядке, контрольные точки,
  финализация прогона.
- **Зависимости:** Workflow (декларация), агенты, хранилище состояния (через Storage Adapter),
  QA, механизм Human Approval.

### 3.5 Agents
- **Ответственность:** узкие производственные роли; каждый выполняет один этап и возвращает
  Structured Output. Не управляют пайплайном.
- **Вход:** структурированный вход шага (от Content Director, согласно контракту).
- **Выход:** Structured Output по схеме своей роли (контент + метаданные: уверенность, флаги).
- **Зависимости:** свой Prompt, набор Skills и Tools, LLM Adapter (через слой адаптеров).

### 3.6 Skills
- **Ответственность:** переиспользуемые модули одной задачи — строительные блоки агентов
  (раздел 7).
- **Вход/Выход:** типизированный вход/выход одной операции.
- **Зависимости:** при необходимости — Tools/Adapters; **никогда** — другие агенты или
  состояние пайплайна.

### 3.7 Tools
- **Ответственность:** операционные возможности, вызываемые **агентом во время рассуждения**
  (раздел 8).
- **Вход/Выход:** запрос агента → результат действия/данных.
- **Зависимости:** при необходимости — адаптеры (например, доступ к внешнему источнику).

### 3.8 Adapters
- **Ответственность:** изоляция ядра от внешних поставщиков и инфраструктуры (раздел 9).
- **Вход/Выход:** внутренний контракт системы ↔ внешний API поставщика.
- **Зависимости:** конкретный внешний сервис (LLM-провайдер, Notion, Google Docs, хранилище,
  аналитика).

### 3.9 QA
- **Ответственность:** контроль качества и соответствия (включая медицинский комплаенс домена
  здоровья) до выхода артефакта к человеку. Реализуется как специализированный агент(ы) QA в
  конвейере. Работает по принципу **fail closed**.
- **Вход:** артефакт-кандидат и сопровождающие метаданные/флаги предыдущих шагов.
- **Выход:** Structured Output с вердиктом (прошёл/риск/отклонить), перечнем замечаний и флагов
  риска.
- **Зависимости:** свои Prompt/Skills/Tools; правила домена; Content Director (как потребитель
  вердикта для маршрутизации).

### 3.10 Analytics
- **Ответственность:** сбор метрик стоимости/производительности/качества по каждому вызову
  агента и прогону; аналитический агент формирует предложения по оптимизации (раздел 14).
- **Вход:** метрики вызовов, исходы прогонов, решения человека.
- **Выход:** агрегаты, отчёты, предложения по оптимизации (для человека/ADR).
- **Зависимости:** Storage/Analytics Adapter; данные трассировки `Run`.

### 3.11 Google Docs
- **Ответственность:** площадка ревью — здесь человек видит артефакт-кандидат с контекстом и
  принимает решение (Approve/Reject/Request changes).
- **Вход:** артефакт-кандидат и сопровождающий контекст (бриф, флаги QA, версии).
- **Выход:** решение человека и (при необходимости) правки/комментарии.
- **Зависимости:** Google Docs Adapter, n8n для связи статусов.

## 4. Orchestrator — Content Director

`Content Director` — **детерминированный исполняющий слой** и единственный владелец потока и
состояния прогона. Это сердце системы по управлению, но **не по творчеству**.

**Content Director отвечает исключительно за:**

1. **Запуск Workflow** — выбор и инициирование декларативного конвейера, соответствующего типу
   контента задания.
2. **Управление состоянием** — ведение объекта `Run` через его жизненный цикл (раздел 5, 10),
   надёжная фиксация каждого перехода.
3. **Маршрутизацию** — передачу управления и структурированных данных от шага к шагу согласно
   Workflow и в зависимости от вердиктов (например, QA-риск → возврат на доработку или эскалация).
4. **Повторные попытки** — применение политики ретраев для транзиентных ошибок и ошибок
   контракта (PROJECT.md, раздел 10), с учётом и записью числа попыток.
5. **Human Approval** — перевод `Run` в состояние ожидания человека на Approval Gate и приём
   решения (раздел 13). Без одобрения — никаких внешних публикаций.
6. **Завершение Run** — финализацию прогона в терминальное состояние (`Completed`/`Failed`) с
   полной трассировкой.

**Чего Content Director НЕ делает (границы ответственности):**

```
   ┌─────────────────────────── Content Director ───────────────────────────┐
   │  ДЕЛАЕТ:  запуск Workflow · состояние · маршрутизация · ретраи ·        │
   │           контрольные точки Human Approval · завершение Run            │
   │                                                                        │
   │  НЕ ДЕЛАЕТ: не пишет и не редактирует контент · не оценивает качество  │
   │             по существу · не «придумывает» решения · не вызывает LLM    │
   │             ради творческой работы · не зашивает бизнес-смысл в себя    │
   └────────────────────────────────────────────────────────────────────────┘
```

> **Инвариант (PROJECT.md, раздел 4, п. 2).** **Оркестратор никогда не принимает творческих
> решений.** Все содержательные решения — у агентов; Content Director лишь решает «кого и когда
> запустить» на основе их **структурированных** результатов и правил Workflow. Бизнес-логику
> конвейера запрещено «зашивать» в промпт; равно запрещено зашивать творчество в оркестратор.

**Принцип детерминизма.** При одинаковых входных данных, одной версии Workflow и зафиксированных
версиях промптов/моделей маршрутизация Content Director воспроизводима. Недетерминизм
сосредоточен внутри агентов (LLM), а не в управлении.

## 5. Workflow Engine — жизненный цикл Run

`Workflow` — **декларация** конвейера (какие шаги/агенты, порядок, контрольные точки). `Workflow
Engine` — часть Content Director, которая *исполняет* эту декларацию, продвигая `Run` по
состояниям. Engine исполняет, Workflow описывает (PROJECT.md: Workflow ≠ Orchestrator).

**Жизненный цикл Run (основной путь и ветви):**

```
            ┌─────────┐
            │ Created │   Run создан из брифа; вход зафиксирован
            └────┬────┘
                 ▼
            ┌─────────┐
            │ Queued  │   Поставлен в очередь на исполнение
            └────┬────┘
                 ▼
            ┌─────────┐        транзиентная/контрактная ошибка
            │ Running │◀──────────────── retry (по политике) ──────────┐
            └────┬────┘                                                 │
                 │ агенты конвейера отработали                          │
                 ▼                                                      │
         ┌──────────────┐    QA: риск/отклонение                       │
         │  Waiting QA  │───────────────► (возврат на доработку) ──────┤
         └──────┬───────┘                                              │
                │ QA: пройдено                                          │
                ▼                                                      │
       ┌─────────────────┐   Reject ─► (возврат на доработку) ─────────┘
       │  Waiting Human  │
       │  (Approval Gate)│   Request changes ─► (доработка с указаниями)
       └────────┬────────┘
                │ Approve
                ▼
          ┌───────────┐
          │ Completed │   Артефакт одобрен; публикация разрешена; метрики собраны
          └───────────┘

   В любой точке при неустранимой ошибке / исчерпании ретраев / фатальном сбое:
                ▼
           ┌────────┐
           │ Failed │   Зафиксированное терминальное состояние с трассировкой
           └────────┘
```

**Описание состояний и переходов:**

| Состояние | Смысл | Выходные переходы |
|---|---|---|
| **Created** | `Run` создан, вход (бриф) зафиксирован | → Queued |
| **Queued** | Ожидает исполнения | → Running |
| **Running** | Исполняется конвейер агентов | → Waiting QA; → Running (retry); → Failed |
| **Waiting QA** | Артефакт у контроля качества/комплаенса | → Waiting Human (пройдено); → Running/доработка (риск); → Failed |
| **Waiting Human** | Approval Gate: ожидание решения человека | → Completed (Approve); → доработка (Reject/Changes); → Failed |
| **Completed** | Терминальное: одобрено, публикация разрешена | — |
| **Failed** | Терминальное: управляемый сбой с трассировкой | — (возможен ручной перезапуск нового `Run`/возобновление) |

**Правила переходов (инварианты):**

- Каждый переход **атомарно фиксируется** в хранилище состояния (раздел 10). «Молчаливых»
  переходов нет.
- Переход `Waiting Human → Completed` возможен **только** по явному `Approve`. Никакой иной путь
  не ведёт к публикации (PROJECT.md, раздел 12).
- `Waiting QA`/`Waiting Human` — это **паузы**: `Run` может находиться в них сколь угодно долго,
  не удерживая исполняющие ресурсы.
- `Failed` — всегда **зафиксированное наблюдаемое** состояние, а не аварийное падение процесса.
- Возвраты на доработку (из QA или от человека) повторно входят в `Running` с дополнительным
  контекстом замечаний; счётчики итераций/ретраев фиксируются (для аналитики и защиты от циклов).

## 6. Agent Architecture

Любой агент имеет единую внутреннюю структуру. Это обеспечивает единообразие, заменяемость и
тестируемость (PROJECT.md, разделы 8, 15).

**Каждый агент состоит из пяти элементов:**

```
                       ┌──────────────────────────────────────────┐
                       │                  AGENT                    │
                       │            (узкая роль · контракт)        │
                       ├──────────────────────────────────────────┤
                       │                                          │
   вход (Structured)   │  ┌────────────────┐   ┌───────────────┐  │
        ──────────────►│  │ Configuration  │   │    Prompt     │  │
                       │  │ роль·модель·   │   │ System/User · │  │
                       │  │ лимиты·политики│   │ Examples·Ver. │  │
                       │  └────────────────┘   └───────────────┘  │
                       │                                          │
                       │  ┌────────────────┐   ┌───────────────┐  │
                       │  │     Skills     │   │     Tools     │  │
                       │  │ переиспользуе- │   │ возможности,  │  │
                       │  │ мые операции   │   │ вызываемые    │  │
                       │  │                │   │ при рассужд.  │  │
                       │  └────────────────┘   └───────────────┘  │
                       │                                          │
                       │            ┌───────────────────┐         │
                       │            │   Output Schema   │         │  выход (Structured)
                       │            │ форма результата  │ ───────────────────►
                       │            │ (обязательна)     │         │
                       │            └───────────────────┘         │
                       └──────────────────────────────────────────┘
```

**Назначение элементов:**

- **Configuration** — внешняя конфигурация роли: привязка «роль → класс модели» (без жёсткого
  имени модели — PROJECT.md, раздел 5), лимиты, политики ретраев, параметры поведения. Меняется
  без правки логики агента.
- **Prompt** — промпт-архитектура (System Prompt, User Prompt, Examples при необходимости,
  Version, Evaluation Criteria — PROJECT.md, раздел 15). Хранится отдельно от кода, версионируется.
- **Skills** — набор переиспользуемых операций, из которых агент компонует поведение (раздел 7).
- **Tools** — возможности, доступные модели агента во время рассуждения (раздел 8).
- **Output Schema** — обязательная схема структурированного результата (раздел 11; PROJECT.md,
  раздел 9). Промпт обязывает её соблюдать; результат дополнительно валидируется системой.

**Инварианты агента:** узкая роль; без скрытого состояния (весь контекст — на входе); только
структурированный вывод; модель — снаружи в конфигурации; не управляет пайплайном; для домена
здоровья — `fail closed` (PROJECT.md, раздел 8).

## 7. Skills Layer

**Skill** — переиспользуемый, независимый и тестируемый модуль выполнения **одной задачи**;
строительный блок агентов (PROJECT.md, раздел 14). Слой Skills — это **библиотека** таких
модулей, общая для всех агентов.

**Принцип повторного использования:**

```
        Skill A ─────┬───────────────┬──────────────► используется многими агентами
                     │               │
   Agent 1 ──► [ Skill A · Skill B ]                  (многие-ко-многим)
   Agent 2 ──► [ Skill A · Skill C ]
   Agent 3 ──► [ Skill B · Skill C · Skill D ]
                     │       │
        Skill B ─────┘       └──────────────────────► один агент использует несколько Skills
```

- Один агент использует **несколько** Skills; один Skill используется **несколькими** агентами.
- Skills **ничего не знают об агентах** и **не управляют пайплайном**.
- **Повторяющаяся логика всегда выносится в Skill.** Дублирование между агентами — дефект и
  кандидат на немедленное выделение в Skill.
- Skills изолированы и покрываются модульными тестами вне агентов и оркестратора.
- Расширение библиотеки — **добавлением** нового Skill, не модификацией существующих под чужую
  задачу (PROJECT.md, раздел 4, п. 11).

**Примеры будущих Skills (иллюстративно, не исчерпывающе):**

- извлечение ключевых тезисов из брифа;
- построение структуры материала по редакционному шаблону;
- нормализация и проверка терминологии (в т. ч. медицинской);
- сегментация длинного текста на смысловые блоки;
- проверка наличия обязательных дисклеймеров для домена здоровья;
- адаптация одного смыслового ядра под формат (статья → тезисы для карусели → сценарий видео);
- оценка читаемости/тональности относительно brand voice;
- сбор и нормализация цитат/источников для фактчекинга.

> Skills образуют «общий словарь операций» фабрики: новый тип контента собирается во многом из
> уже существующих Skills, что и обеспечивает дешёвое горизонтальное масштабирование.

## 8. Tool Layer

**Tool** — операционная возможность, которую **агент вызывает во время своего рассуждения**, для
действия или получения данных. Важно не смешивать Tool со Skill.

**Различие Tool и Skill (PROJECT.md, раздел 18 — «Skill ≠ Tool»):**

```
   ┌──────────────── Skill ────────────────┐   ┌──────────────── Tool ─────────────────┐
   │ Внутренний модуль системы              │   │ Возможность, вызываемая МОДЕЛЬЮ агента │
   │ Детерминированная переиспользуемая     │   │ во время инференса/рассуждения         │
   │ операция (логика)                      │   │ (действие или доступ к данным)         │
   │ Вызывается в коде агента/системы       │   │ Инициируется решением модели           │
   │ Назначение: переиспользование логики   │   │ Назначение: дать агенту «руки и глаза» │
   └────────────────────────────────────────┘   └────────────────────────────────────────┘
```

**Как агент использует Tool:**

```
   Agent (рассуждение по Prompt)
        │  «мне нужно действие/данные»
        ▼
     Tool вызывается ──► (при необходимости) Adapter ──► внешний источник/сервис
        │
        ▼
     результат возвращается в рассуждение агента ──► агент формирует Output Schema
```

- Tool — это контролируемая граница действия: агент не «ходит во внешний мир» напрямую, а через
  объявленный Tool (который, в свою очередь, при выходе наружу опирается на Adapter).
- Набор доступных агенту Tools задаётся его конфигурацией; агент не может выйти за пределы
  выданных ему возможностей.
- Tool не управляет пайплайном и не подменяет оркестратор — он обслуживает **один шаг рассуждения**
  агента.

## 9. Adapter Layer

**Adapter** — слой изоляции ядра от внешних поставщиков и инфраструктуры. Адаптеры реализуют
**внутренний контракт системы поверх чужого API** и обеспечивают заменяемость без изменения ядра
(PROJECT.md, раздел 4, п. 7; «Adapter ≠ Tool» — раздел 18).

```
        ЯДРО (Agents · Skills · Tools · Content Director)
                       │  внутренний контракт
        ┌──────────────┼───────────────┬───────────────┬───────────────┐
        ▼              ▼               ▼               ▼               ▼
   LLM Adapter   Notion Adapter  Google Docs Ad.  Storage Adapter  Analytics Ad.
        │              │               │               │               │
        ▼              ▼               ▼               ▼               ▼
   LLM-провайдер     Notion        Google Docs       Хранилище     Аналитич. сток
   (по умолч.        API            API              состояния/    /метрики
    Anthropic,                                       артефактов
    заменяем)
```

**Адаптеры и их назначение:**

- **LLM Adapter** — единая внутренняя точка доступа к моделям. Делает систему **независимой от
  поставщика**: смена модели/провайдера — это смена/конфигурация адаптера, без правок агентов и
  оркестратора. Поставщик по умолчанию — Anthropic, но заменяем на любого (PROJECT.md, раздел 5).
- **Notion Adapter** — чтение брифов/метаданных и запись статусов; изолирует ядро от специфики
  Notion.
- **Google Docs Adapter** — публикация артефакта-кандидата на ревью и считывание решения/правок
  человека; изолирует ядро от специфики Google Docs.
- **Storage Adapter** — персистентность состояния `Run`, артефактов и трассировки; позволяет
  менять реализацию хранилища (от простого на MVP до промышленного) без затрагивания логики.
- **Analytics Adapter** — отправка метрик стоимости/производительности/качества в аналитический
  сток; изолирует ядро от конкретной аналитической реализации.

**Зачем нужны адаптеры:**

- **Заменяемость поставщиков** без изменения ядра (ключ к независимости от моделей и к
  долгосрочной расширяемости).
- **Тестируемость**: ядро тестируется против контрактов адаптеров, а не «живых» внешних систем.
- **Локализация изменений**: любое изменение внешнего API затрагивает только свой адаптер.
- **Соблюдение направления зависимостей** (инфраструктура зависит от домена, не наоборот —
  PROJECT.md, раздел 7).

## 10. State Management — объект Run

`Run` — один сквозной проход брифа через `Workflow` с полной трассировкой. Это **единица
состояния и аудита** всей системы.

**Что концептуально хранит Run (категории данных, без определения схемы):**

- **Идентичность и привязка:** идентификатор прогона, ссылка на задание (бриф) и тип контента,
  версия использованного `Workflow`.
- **Входные данные:** зафиксированный бриф и метаданные на момент старта (для воспроизводимости).
- **Текущее состояние:** позиция в жизненном цикле (раздел 5) и метка последнего перехода.
- **Артефакты:** промежуточные и итоговый `Artifact` каждого шага (структура, черновик,
  отредактированный текст, кандидат на ревью).
- **Трассировка шагов:** для каждого вызова агента — какой агент, версия его `Prompt`, факт
  использованная модель, входные/выходные структурированные данные, число повторов, статус.
- **Метрики стоимости/производительности:** по каждому вызову — модель, input/output токены,
  стоимость, латентность, retries (раздел 14; PROJECT.md, раздел 16).
- **Вердикты QA:** результаты контроля качества/комплаенса и флаги риска.
- **Решения человека:** кто, когда и какое решение принял (Approve/Reject/Request changes),
  с причиной/указаниями (раздел 13).

**Как меняется состояние Run:**

```
   Событие (агент отработал · QA вынес вердикт · человек решил · ошибка)
        │
        ▼
   Content Director вычисляет следующий переход (по Workflow и вердиктам)
        │
        ▼
   Атомарная фиксация нового состояния + трассировки в хранилище (через Storage Adapter)
        │
        ▼
   Состояние Run наблюдаемо и восстановимо в любой момент
```

**Инварианты состояния:**

- **Единственный владелец переходов — Content Director.** Никакой агент/skill/tool не меняет
  состояние `Run` напрямую.
- **Прослеживаемость по умолчанию:** шаг без записи трассировки и метрик считается несделанным
  (PROJECT.md, разделы 4 п. 5, 16).
- **Идемпотентное возобновление:** `Run` восстанавливается из сохранённого состояния без
  дублирования уже выполненной работы (PROJECT.md, раздел 10).
- **Никаких секретов/PII** в трассировке и логах.

## 11. Data Contracts — взаимодействие между агентами

Всё межкомпонентное взаимодействие в конвейере — **исключительно через Structured Outputs**.
**Свободный текст между агентами запрещён** (PROJECT.md, раздел 9: «нет схемы — нет границы»).

```
   Agent A ──[ Structured Output по Schema A ]──► Content Director
                                                      │  валидация по Schema A
                                                      ▼
                                                  маршрутизация
                                                      │
   Content Director ──[ Structured Input по Schema B ]──► Agent B
```

**Принципы контрактов:**

- **Граница = типизированная схема.** Каждый агент имеет строго описанные вход и выход; граница
  между шагами — валидируемый контракт данных, а не текст.
- **Двойная защита.** Используются нативные механизмы структурированного вывода поставщика
  (через LLM Adapter) **и** независимая валидация результата системой. Невалидный вывод **не
  проходит дальше** по конвейеру и обрабатывается как ошибка контракта (раздел 5, ретраи).
- **Schema — единый источник истины** формы данных: используется для генерации, валидации и
  документирования контракта одновременно.
- **Метаданные в контракте.** Помимо «контента» структурированный вывод несёт служебные данные:
  уверенность, флаги рисков, обоснования — то, что нужно для маршрутизации и для Human Approval.
- **Версионирование контрактов.** Изменение контракта агента — это изменение его `Schema`,
  проводимое осознанно (совместимость/миграция, при необходимости ADR).
- **Свободный текст** допустим **только внутри полей** схемы (как значение), но **не как способ
  передачи** между агентами.

> Конкретные поля схем здесь **не проектируются** — это уровень реализации/ADR. Архитектурно
> важно одно: граница без схемы не существует.

## 12. External Integrations

Внешние интеграции идут **только через адаптеры** (раздел 9) и **триггеры n8n**. Ядро не знает
специфики внешних систем.

**Notion (источник заданий).**
```
   Редактор ─► Notion (бриф готов) ─► n8n (триггер) ─► Python Agent Service (создать Run)
   Ядро ─► Notion Adapter ─► Notion (обновить статус задания)
```
- Роль: постановка задачи, бриф/метаданные, отображение статуса. Источник и «витрина» задания.

**n8n (событийная связка).**
```
   Notion event ─► n8n ─► вызов ядра
   Ядро event ─► n8n ─► обновление Notion / Google Docs
```
- Роль: триггеры и перенос статусов/ссылок между Edge-системами и ядром. **Не содержит** бизнес-
  логики производства.

**Google Docs (ревью и одобрение).**
```
   Ядро ─► Google Docs Adapter ─► Google Docs (опубликовать кандидата на ревью)
   Человек ─► решение/правки ─► (n8n / Google Docs Adapter) ─► ядро (Approve/Reject/Changes)
```
- Роль: площадка, где человек видит артефакт-кандидат с контекстом и фиксирует решение.

**Общие правила интеграций:**

- Любая внешняя система **заменяема**: замена Notion/Google Docs/n8n затрагивает только
  соответствующий адаптер/триггер, но не ядро.
- **Никаких внешних публикаций без Approve** (раздел 13).
- Здесь описывается **взаимодействие**, а не реализация — без API-спецификаций (по требованиям).

## 13. Human Approval

Human Approval — архитектурный **инвариант** (PROJECT.md, раздел 12). Система спроектирована так,
что обойти его нельзя.

**Что происходит после QA:**

```
   QA вердикт
     ├─ «пройдено»  ─► Content Director публикует кандидата в Google Docs
     │                 ─► Run переходит в Waiting Human (Approval Gate)
     │
     └─ «риск/отклонить» (fail closed)
                       ─► возврат на доработку или принудительная эскалация человеку
```

**Как человек утверждает результат:**

- На Approval Gate человеку предоставляется **не только контент**, но и контекст решения:
  исходный бриф, ключевые промежуточные артефакты, флаги/замечания QA и фактчекинга, версии
  использованных моделей/промптов (PROJECT.md, раздел 12).
- Возможные решения: **Approve**, **Reject** (с причиной), **Request changes** (с указаниями).
- Решение и его автор **фиксируются** в `Run` (аудит): кто, когда, что и почему.

```
   Waiting Human
       ├─ Approve         ─► Run → Completed ─► публикация разрешена ─► Analytics
       ├─ Reject          ─► Run → доработка (Running) с зафиксированной причиной
       └─ Request changes ─► Run → доработка (Running) с конкретными указаниями
```

**Что происходит после Reject:**

- `Run` **не завершается успехом** и **не публикуется наружу**. Он возвращается на доработку в
  `Running` с зафиксированной причиной отказа; соответствующие агенты получают этот контекст как
  дополнительный вход на повторной итерации.
- Число итераций доработки фиксируется (для аналитики и защиты от бесконечных циклов); при
  исчерпании установленного предела `Run` переводится в `Failed`/эскалацию по правилам Workflow.

**Инвариант.** Переход к публикации возможен **только** через `Approve`. `Reject`/`Request
changes` никогда не ведут наружу. **«Автоматизируем производство, но не ответственность.»**

## 14. Analytics

Аналитический блок превращает фабрику в **самоулучшающуюся** систему. Метрики собираются по
умолчанию и неизбежно (PROJECT.md, раздел 16).

**Какие данные собираются (по каждому вызову агента, как часть трассировки Run):**

- модель (имя/версия фактически использованной модели и поставщик);
- количество входных токенов (input);
- количество выходных токенов (output);
- стоимость вызова;
- время выполнения (латентность);
- количество повторных попыток (retries).

Дополнительно на уровне прогона: исход `Run` (Completed/Failed), число итераций доработки,
вердикты QA, решения человека (включая долю Reject/Changes).

**Архитектура потока аналитики:**

```
   Каждый вызов агента ─► метрики ─► Analytics Adapter ─► аналитический сток/хранилище
                                                              │
                                                              ▼
                                                    Аналитический агент
                                                   (агрегация · выводы · предложения)
                                                              │
                                                              ▼
                                          Предложения по оптимизации ─► человек / ADR
```

**Какие выводы строятся:**

- где переназначить роль на **более экономичный класс модели** без потери качества;
- где сокращать токены/повторы (узкие места по стоимости);
- где узкие места по латентности;
- какие промпты/версии дают регрессии или, наоборот, улучшения качества (связь с Evaluation
  Criteria, PROJECT.md, раздел 15);
- какие шаги чаще всего вызывают Reject/доработку (точки улучшения качества).

**Как система будет улучшаться:**

- Аналитический агент **предлагает**, но **не меняет** систему самовольно. Любая оптимизация
  (смена привязки «роль → модель», правка промпта) проходит обычный путь согласования и
  версионирования — **через ADR** и ревью (PROJECT.md, разделы 16, 11).
- Накопленная статистика — фактологическая база цикла непрерывного улучшения фабрики и материал
  для будущих ADR.

## 15. Repository Architecture

Здесь фиксируются **архитектурные принципы** организации репозитория. **Структура папок не
создаётся** (по требованию) — это уровень реализации/ADR. Принципы наследуют PROJECT.md, раздел 7.

**Принципы:**

- **Разделение по слоям ответственности** с зависимостями, направленными **внутрь**
  (инфраструктура зависит от домена, не наоборот):
  ```
        Точки входа / интеграции
                 │
        Оркестрация (Content Director · Workflow Engine)
                 │
        Агенты ── Skills ── Tools
                 │
        Адаптеры (LLM · Notion · Google Docs · Storage · Analytics)
                 │
        Домен / Контракты (Schemas, типы артефактов, состояние Run)   ◄── ни от чего не зависит
  ```
- **Промпты, конфигурация и код — раздельно.** Промпты версионируются и ревьюятся как код
  (PROJECT.md, раздел 15).
- **Один агент — одна изолированная заменяемая единица** с явным контрактом.
- **Skills — общая библиотека**, не привязанная к конкретному агенту.
- **Адаптеры — единственная точка контакта с внешним миром.** Прямые обращения ядра к внешним
  API вне адаптеров запрещены.
- **Тесты располагаются рядом со своей областью** и являются частью определения «готово»;
  Skills/Tools/контракты — модульно тестируемы изолированно.
- **Workflow — декларации, отделённые от исполняющего кода** оркестратора.
- Соответствие иерархии документации (PROJECT.md, раздел 17): код не противоречит ARCHITECTURE.md
  и PROJECT.md; при конфликте — приоритет за документацией.

## 16. Future Extensions

Система расширяется **добавлением модулей, а не изменением существующих** (PROJECT.md, раздел 4,
п. 11 — Open/Closed). Ниже — как именно подключается новое.

**Новый агент.**
```
   Добавить: Configuration + Prompt + (ссылки на существующие/новые Skills и Tools) + Output Schema
   Подключить: как новый шаг в нужный Workflow
   Не менять: Content Director, существующие агенты, существующие контракты
```
- Новый агент реализует существующий механизм агента (раздел 6) и встаёт в Workflow как ещё один
  шаг с собственным контрактом.

**Новый Skill.**
```
   Добавить: изолированный, тестируемый модуль одной задачи в библиотеку Skills
   Подключить: указать его в нужных агентах
   Не менять: другие Skills и агентов, которые его не используют
```
- Повторяющаяся логика выносится в новый Skill; существующие Skills остаются нетронутыми.

**Новый тип контента.**
```
   Добавить: новый Workflow (декларация шагов/агентов/контрольных точек для формата)
   Переиспользовать: существующие агенты, Skills, Tools, адаптеры, оркестратор
   Не менять: ядро (тип контента — не архитектурное ограничение; PROJECT.md, раздел 2)
```
- Статьи, карусели, AI-видео и прочие форматы — это **новые Workflow**, собираемые во многом из
  уже существующих строительных блоков.

**Новая модель / новый поставщик.**
```
   Изменить/добавить: конфигурацию привязки «роль → класс модели» и/или реализацию LLM Adapter
   Не менять: агентов, Skills, оркестратор, контракты
```
- Независимость от поставщика обеспечивается LLM Adapter и внешней конфигурацией (PROJECT.md,
  раздел 5). Anthropic — по умолчанию, но заменяем.

**Новая интеграция / внешний сервис.**
```
   Добавить: новый Adapter (+ при необходимости триггер n8n и/или Tool для агентов)
   Не менять: ядро и существующие адаптеры
```
- Любой внешний сервис входит в систему **через свой адаптер**, сохраняя изоляцию ядра.

**Общий инвариант расширяемости.** Если для подключения нового агента/Skill/типа/модели/
интеграции приходится **менять оркестратор или существующие контракты** — это **сигнал дефекта
архитектуры**, а не нормальный путь развития (PROJECT.md, раздел 4, п. 11).
