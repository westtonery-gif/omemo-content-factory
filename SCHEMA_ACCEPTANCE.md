# SCHEMA_ACCEPTANCE.md — Критерии приёмки реализации **Schema**

> **Место в иерархии документации (PROJECT.md, раздел 17).**
> `PROJECT.md` → `ARCHITECTURE.md` → `ROADMAP.md` → `DOMAIN_MODEL.md` → `SCHEMA_SPEC.md` →
> **`SCHEMA_ACCEPTANCE.md`** → `ADR` → `Implementation (Code)`.
>
> Документ отвечает на вопрос: **«Как мы поймём, что реализация Schema полностью соответствует
> `SCHEMA_SPEC.md` и `ADR-0008`?»** Это набор **проверяемых сценариев приёмки** на языке
> предметной области. Он не содержит кода, pytest и технических деталей и становится **прямой
> основой для `tests/test_schema.py`**: один сценарий → один тест (или группа тестов).
>
> Документ описывает **только наблюдаемое доменное поведение самой Schema**. Поведение
> соседних сущностей (Output, Run) и слоёв (оркестрация) — **не** предмет этой приёмки; то, что
> относится к ним, вынесено в раздел 8 как будущая ответственность.
>
> **Источник истины:** `SCHEMA_SPEC.md` (по сущности Schema) → `ADR-0008` (технический контракт) →
> `DOMAIN_MODEL.md` → `PROJECT.md`.

## 0. Соглашения

- Формат сценария — **Дано / Когда / Тогда** (Given / When / Then) на языке домена.
- Каждый сценарий имеет **уникальный идентификатор** (например, `SHP-01`) для прямого
  отображения в будущий тест.
- Критерии проверяют **только наблюдаемое поведение** Schema через её публичный контракт, а **не
  внутреннюю реализацию** (типы, хранение, структуры данных) и **не** соседние сущности/слои.
- **Распоряжение жизненным циклом.** Создание Schema и переходы `Draft → Active → Deprecated`
  относятся к **владельцу каталога определений** (`SCHEMA_SPEC.md` §7) — это *процессное
  полномочие* (кому позволено). На этом этапе `ADR-0008` §10 **не вводит рантайм-актора** для
  каталога: переход защищён **таблицей допустимых переходов**, а не проверкой личности актёра.
  Поэтому «единственный распорядитель статуса» здесь **не** выражается как сценарий отклонения по
  актёру (см. раздел 10, флаг **G-1**) и **не выдумывается**.
- «Отвергается как ошибка предметной области» означает: операция **не выполняется**, состояние
  Schema **не изменяется**, фиксируется **доменная** ошибка (не технический сбой).
- **Вердикт валидации — не ошибка.** Невалидный результат — это нормальный наблюдаемый исход
  проверки (`SCHEMA_SPEC.md` §4), а не доменная ошибка.
- Доступ к Schema — **только для чтения** (снимок), без изменяемого объекта снаружи корня.
- Префиксы блоков: `SAC` — критерии приёмки; `SHP` — жизненный цикл; `SFL` — запрещённые
  переходы; `SID` — идентичность и версионность; `SVAL` — валидация; `SINV` — инварианты
  (+ таблица трассируемости); `SISO` — изоляция Schema от исполнения.

## 1. Acceptance Criteria

Проверяемые критерии, выведенные из `SCHEMA_SPEC.md` и `ADR-0008`. Каждый покрывается одним или
несколькими сценариями разделов 2–7 (ссылки указаны).

- **SAC-01. Создание в `Draft`.** Новая версия Schema возникает в `Draft` с идентичностью,
  версией и контрактом; доступна для чтения. ⟶ `SHP-01`
- **SAC-02. Идентичность и версионность.** `SchemaId` стабилен; `SchemaVersion` сравнивается по
  значению; новая ревизия — новая версия. ⟶ `SID-01..03`
- **SAC-03. Жизненный цикл.** Допустимы переходы `Draft → Active → Deprecated`. ⟶ `SHP-02`,
  `SHP-03`
- **SAC-04. Только разрешённые переходы.** Любой переход вне `SCHEMA_SPEC.md` §5 отвергается. ⟶
  `SFL-01`
- **SAC-05. Неизменяемость принятой версии.** Содержание версии не меняется ни в одном статусе;
  изменение контракта = новая версия. ⟶ `SINV-01`, `SID-03`
- **SAC-06. Вердикт валидации.** Результат, удовлетворяющий контракту, признаётся валидным;
  иначе — невалидным. ⟶ `SVAL-01`, `SVAL-02`
- **SAC-07. Детерминированность и повторяемость валидации.** Один и тот же результат против одной
  и той же версии всегда даёт один и тот же вердикт, без побочного эффекта. ⟶ `SVAL-03`, `SVAL-04`
- **SAC-08. Валидация только против `Active`.** Проверка допустима лишь относительно `Active`
  версии. ⟶ `SVAL-05`
- **SAC-09. Самостоятельность корня и изоляция от исполнения.** Schema создаётся и проходит
  жизненный цикл независимо от Run; нет операции изменения Schema через исполнение. ⟶ `SISO-01`,
  таблица §6 (инв. 9)

*Критериев приёмки: **9** (SAC-01 … SAC-09).*

> Критерий «единственный распорядитель статуса» (`SCHEMA_SPEC.md` §6, инв. 5) **намеренно не
> включён** как исполняемый: на этом этапе нет рантайм-актора каталога (`ADR-0008` §10). См.
> раздел 10, флаг **G-1**.

## 2. Lifecycle Scenarios

**SHP-01. Создание новой версии Schema.**
- *Дано:* каталог определений.
- *Когда:* создаётся версия Schema с описанием и набором обязательных полей.
- *Тогда:* возникает версия в `Draft` с `SchemaId` и `SchemaVersion`; описание и обязательные
  поля доступны для чтения; текущий статус — ровно один (`Draft`).

**SHP-02. Ввод версии в обращение.**
- *Дано:* версия Schema в `Draft`.
- *Когда:* выполняется переход `Draft → Active`.
- *Тогда:* версия в `Active` (ровно один статус); относительно неё допустима валидация (раздел 5).

**SHP-03. Вывод версии из обращения.**
- *Дано:* версия Schema в `Active`.
- *Когда:* выполняется переход `Active → Deprecated`.
- *Тогда:* версия в `Deprecated` (терминал); новой валидации против неё не выполняется (раздел 5).

*Сценариев жизненного цикла: **3**.*

---

## 3. Forbidden-Transition Scenarios (ошибки предметной области)

**SFL-01. Запрещённые переходы статуса отвергаются.**
- *Когда:* запрашивается любой переход, отсутствующий в `SCHEMA_SPEC.md` §5 — в частности:
  `Draft → Deprecated` (минуя `Active`), `Active → Draft`, `Deprecated → Active`,
  `Deprecated → Draft`, любой переход **из** терминала `Deprecated`, а также переход в тот же
  статус.
- *Тогда:* отвергается как доменная ошибка; статус версии не меняется. *(Полный перебор рёбер.)*

*Сценариев запретов: **1** (полный перебор недопустимых рёбер).*

---

## 4. Identity & Versioning Scenarios

**SID-01. Неизменяемость `SchemaId`.** Идентификатор логического контракта остаётся прежним на
всём жизненном цикле версии (через все переходы статусов).

**SID-02. `SchemaVersion` сравнивается по значению.** Две версии с одинаковым значением версии
равны по версии («v3 равно v3»); версии с разными значениями различны (свойство Value Object,
`DOMAIN_MODEL.md` §11).

**SID-03. Новая ревизия — новая версия.** Создание новой версии того же контракта даёт **другое**
значение `SchemaVersion` при **том же** `SchemaId`; содержание ранее принятой версии при этом
**не изменяется**.

*Сценариев идентичности/версионности: **3**.*

## 5. Validation Scenarios

> Schema проверяет переданный ей структурированный результат и **выносит вердикт**. Вердикт —
> собственный наблюдаемый результат Schema; как он впоследствии отражается на Output, описано в
> разделе 8 (будущая ответственность Output), а не здесь.

**SVAL-01. Валидный результат.**
- *Дано:* версия Schema в `Active` с заданным набором обязательных полей.
- *Когда:* проверяется результат, в котором **все обязательные поля присутствуют и непусты**.
- *Тогда:* вердикт — **валиден**.

**SVAL-02. Невалидный результат.**
- *Дано:* версия Schema в `Active` с заданным набором обязательных полей.
- *Когда:* проверяется результат, в котором **отсутствует или пусто** хотя бы одно обязательное
  поле.
- *Тогда:* вердикт — **невалиден** (нормальный исход, не доменная ошибка).

**SVAL-03. Детерминированность.**
- *Дано:* версия Schema в `Active`.
- *Когда:* один и тот же результат проверяется относительно этой версии.
- *Тогда:* вердикт совпадает с вердиктом любой другой проверки того же результата против той же
  версии (одинаковый вход → одинаковый исход).

**SVAL-04. Повторяемость без побочного эффекта.**
- *Дано:* версия Schema в `Active`.
- *Когда:* один и тот же результат проверяется **повторно**.
- *Тогда:* исход каждой проверки **идентичен**; статус и содержание Schema после проверок
  **не изменяются** (валидация не имеет наблюдаемого побочного эффекта).

**SVAL-05. Валидация только против `Active`.**
- *Дано:* версия Schema в `Draft` **или** в `Deprecated`.
- *Когда:* запрашивается валидация результата относительно неё.
- *Тогда:* отвергается как доменная ошибка; вердикт не выносится.

*Сценариев валидации: **5**.*

## 6. Invariants

После поэлементной проверки инвариантов `SCHEMA_SPEC.md` §6 уникальную, не покрытую иными
сценариями проверку даёт только инвариант 3 (неизменяемость содержания версии **через смену
статусов**) — он оставлен явным сценарием. Остальные инварианты покрываются сценариями разделов
2–5/7 и сведены в таблицу трассируемости (без дублирующих сценариев).

**SINV-01. Неизменяемость содержания версии через жизненный цикл.**
- *Дано:* версия Schema с заданными описанием и набором обязательных полей.
- *Когда:* версия проходит `Draft → Active → Deprecated`.
- *Тогда:* описание и набор обязательных полей на каждом шаге **неизменны**; смена статуса меняет
  только пригодность версии к использованию, но не её содержание.

**Таблица трассируемости инвариантов (`SCHEMA_SPEC.md` §6):**

| Инвариант | Покрытие |
|---|---|
| 1. Ровно один текущий статус | `SHP-01/02/03` (после каждого перехода — ровно один статус) |
| 2. Неизменная идентичность и версия | `SID-01`, `SID-02` |
| 3. Неизменяемость принятой версии | **`SINV-01`** (через жизненный цикл) + `SID-03` (новая ревизия) |
| 4. Только разрешённые переходы | `SFL-01` |
| 5. Единственный распорядитель статуса | флаг **G-1** (процессное полномочие; не принуждается на этом этапе) |
| 6. Валидация только против `Active` | `SVAL-05` |
| 7. Детерминированность валидации | `SVAL-03`, `SVAL-04` |
| 8. Инвариант валидности Output | **вне Schema** → раздел 8 (будущий `OUTPUT_ACCEPTANCE`) |
| 9. Самостоятельность корня | `SISO-01` + структурно (нет операции изменения Schema через исполнение) |

*Уникальных сценариев инвариантов: **1** (`SINV-01`); остальные — в таблице трассируемости.*

## 7. Isolation Scenarios

**SISO-01. Schema независима от Run.**
- *Дано:* каталог определений.
- *Когда:* версия Schema создаётся и проходит `Draft → Active → Deprecated`.
- *Тогда:* для этих операций **не требуется** ни один Run; Schema существует и меняет статус вне
  оси исполнения. Это же демонстрирует самостоятельность корня (`SCHEMA_SPEC.md` §6 инв. 9): нет
  операции, которой исполнение прогона меняло бы Schema.

*Сценариев изоляции: **1**.*

## 8. Будущая ответственность (вне приёмки Schema)

Перечисленное — наблюдаемое поведение **соседних сущностей и слоёв**, не Schema. Оно
проверяется в соответствующих acceptance-документах того среза, где будет реализовано, и здесь
**намеренно не является критерием Schema** (граница ответственности — `SCHEMA_SPEC.md` §9, §10).

- **Связь Output ↔ SchemaVersion и статус Output (`Valid`/`Invalid`).** Что зафиксированный
  Output ссылается на ровно одну `SchemaVersion` и несёт статус согласно вердикту — это
  **инвариант Output** (`DOMAIN_MODEL.md` §5, §6 — раздел «Output»). ⟶ будущий `OUTPUT_ACCEPTANCE`.
- **Неизменность ранее зафиксированных Output при `Deprecate` версии Schema.** Что вывод версии
  из обращения не меняет уже произведённые Output — это **неизменяемость Output**
  (`DOMAIN_MODEL.md` §6). ⟶ будущий `OUTPUT_ACCEPTANCE`.
- **Непередача невалидного результата дальше по конвейеру.** Маршрутизация по вердикту —
  ответственность **последующих слоёв** (Run/оркестрация), а не Schema (см. флаг **G-2**). ⟶
  приёмка соответствующего слоя.

> Schema выносит вердикт (раздел 5); как вердикт применяется и к каким последствиям для Output/Run
> приводит — ответственность тех сущностей и слоёв, не предмет данной приёмки.

## 10. Флаги и ограничения проверяемости

Согласно принципу «не допридумывать»: ниже — пункты, требующие решения или принципиально не
выражаемые как сценарий Schema.

**Флаги:**

- **G-1. Полномочие (process ownership) против принуждения (runtime enforcement).**
  `SCHEMA_SPEC.md` §6 (инв. 5) / §7 закрепляют, что статус версии меняет **владелец каталога
  определений** — это *процессное полномочие*: кому **позволено** менять статус. `ADR-0008` §10
  сознательно **не вводит рантайм-актора** каталога — на этом этапе действует только *принуждение
  таблицей переходов*, без проверки личности актёра. Следствие: «единственный распорядитель» —
  процессный инвариант, **не принуждаемый кодом**, и потому **не выражается** как acceptance-
  сценарий (нет актёра, чьё нарушение можно наблюдать). *Рекомендуемое решение (вне этого
  документа):* пометить инв. 5 в `SCHEMA_SPEC.md` §6 как «процессное полномочие, не принуждается
  на этом этапе», чтобы SPEC и `ADR-0008` не выглядели противоречащими (`PROJECT.md` §17). При
  вводе catalogue-maintainer-актора добавляется и сценарий отклонения по актёру.

- **G-2. Граница ответственности: маршрутизация невалидного результата.** Требование «невалидный
  результат/Output не передаётся дальше» (`DOMAIN_MODEL.md` §6; `PROJECT.md` §10) — это
  **маршрутизация по вердикту**, ответственность **последующих слоёв** (Run / оркестрация), а не
  Schema. Schema лишь **выносит вердикт** (`SVAL-01/02`). Поэтому это **не зона Schema** (а не
  «отложенный критерий Schema») и проверяется в приёмке соответствующего слоя.

**Структурные ограничения (гарантии дизайна, не выражаемые позитивным сценарием):**

- **Неизменяемость версии на уровне внутренней сущности.** Снаружи корня нет изменяемого объекта
  Schema, поэтому «попытку переписать поле версии» нельзя выразить через публичный контракт.
  Неизменяемость проверяется **позитивно**: содержание стабильно через все переходы (`SINV-01`),
  новая ревизия порождает новую версию (`SID-03`). Доменная ошибка при прямой переустановке поля —
  **структурная гарантия дизайна** (зеркало Run/Task), без отдельного позитивного теста.
- **«Без внешних зависимостей» (валидация).** Утверждение об **отсутствии** обращений наружу;
  проверяется **косвенно** через детерминированность и повторяемость (`SVAL-03/04`), а не
  отдельным сценарием.
- **Самостоятельность корня (инв. 9).** Утверждение об **отсутствии** связи (исполнение не меняет
  Schema) проверяется **косвенно**: у Schema нет такой операции в контракте (`SISO-01`).

### Подтверждение готовности к pytest
